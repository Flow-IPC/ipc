34/**
35 * Internal, non-movable pImpl-lite implementation of sync_io::Blob_stream_mq_receiver class template.
36 * In and of itself it would have been directly and publicly usable; however Blob_stream_mq_receiver adds move semantics
37 * which are essential to cooperation with the rest of the API, Channel in particular.
38 *
39 * @see All discussion of the public API is in Blob_stream_mq_receiver doc header; that class template forwards to this
40 * one. All discussion of pImpl-lite-related notions is also there. See that doc header first please. Then come
41 * back here.
42 *
43 * ### Impl design ###
44 * See sync_io::Blob_stream_mq_sender_impl. The same notes apply. That isn't to say our algorithm is the same
45 * at all: we are receiving; they are sending. However, obviously, they speak the same protocol. And beyond that
46 * it is again about (1) understanding the `sync_io` pattern (util::sync_io doc header) and (2) jumping into the code.
47 *
48 * Again, for that last part, we recommend starting with the data member doc headers.
49 *
50 * ### Protocol negotiation ###
51 * Do see the same-named section in sync_io::Blob_stream_mq_sender_impl doc header. Naturally again we're mirroring
52 * the same protocol; but we're receiving; they're sending. Due to the latter difference, though, a couple additional
53 * notes:
54 * - We are the guy receiving the preceding-all-messages message containing the opposing side's preferred (highest)
55 * protocol version. As of this writing we only speak v1, but they might speak a range -- including or
56 * excluding it. However our Protocol_negotiator takes care of all that; and we only need to add logic to
57 * emit fatal error, if Protocol_negotiator reports that we're incompatible with opposing side.
58 * - Having gotten past that without error, we simply then speak the only version we know how (v1).
59 * - Since only protocol v1 exists as of this writing, there's no need yet for an API to signal a partner
60 * MQ-sender object of the negotiated protocol version, once we have it. As noted in
61 * sync_io::Blob_stream_mq_sender_impl doc header, that may change in the future.
62 *
63 * @note We suggest that if version 2, etc., is/are added, then the above notes be kept more or less intact; then
64 * add updates to show changes. This will provide a good overview of how the protocol evolved; and our
65 * backwards-compatibility story (even if we decide to have no backwards-compatibility).
66 *
67 * @tparam Persistent_mq_handle
68 * See Persistent_mq_handle concept doc header.
69 */

294 /**
295 * The MQ handle adopted by the ctor, through which nb-receives and blocking-waits are executed.
296 * It is not directly protected by any mutex; however it is accessed exclusively as follows:
297 * - By thread W, when #m_user_request is non-null (would-block state), thus thread U has tasked
298 * it with awaiting readability -- which it is doing and is not yet done doing.
299 * - By thread U, when #m_user_request is null (no would-block at the moment); or
300 * it is not null, but thread W has signaled it is done awaiting non-would-block, so thread U
301 * is trying to nullify it via nb-reads.
302 *
303 * Exception: `m_mq->interrupt_receives()` is done from thread U (dtor), potentially *while* thread W
304 * is inside `m_mq->wait_receivable()`. However that specific action is allowed by Persistent_mq_handle contract
305 * (in fact that is the main utility of Persistent_mq_handle::interrupt_receives()). It makes the
306 * Persistent_mq_handle::wait_receivable() immediately return.
307 *
308 * ### When is it null? ###
309 * It is null if and only if in ctor we were unable to create it (an error occurred in subsuming `mq` arg).
310 * Other than this quite-fatal corner case, #m_mq remains non-null until dtor executes.
311 *
312 * Important subtlety: We might be tempted to also nullify it upon detecting an error on the queue, so as to
313 * early-return a resource to the OS. However, for the same reason detailed in doc header for
314 * Native_socket_stream::Impl::m_peer_socket_hosed, we must not do so until dtor runs.
315 * (Technically this is only the case if `Mq::S_HAS_NATIVE_HANDLE == true`, as only then would the user be
316 * potentially directly performing async-waits on the underlying native-handle... but let's not get overly cute
317 * and entropy-laden.)
318 *
319 * So, reminder: Do not `m_mq.reset()` until dtor, if not null upon exiting ctor!
320 */
321 typename Base::Auto_closing_mq m_mq;

682template<typename Persistent_mq_handle>
683template<typename Task_err_sz>
684bool Blob_stream_mq_receiver_impl<Persistent_mq_handle>::async_receive_blob
685 (const util::Blob_mutable& target_blob, Error_code* sync_err_code_ptr, size_t* sync_sz,
686 Task_err_sz&& on_done_func)
687{
688 using util::Blob_mutable;
689
690 if (!op_started("async_receive_blob()"))
691 {
692 return false;
693 }
694 // else
695
696 /* We comment liberally, but tactically, inline; but please read the strategy in the class doc header's impl section.
697 *
698 * Briefly though: If you understand opposing send_blob() impl, then this will be easy in comparison:
699 * no more than one async_receive_blob() can be outstanding at a time; they are not allowed to
700 * invoke it until we invoke their completion handler. (Rationale is shown in detail elsewhere.)
701 * So there is no queuing of requests (deficit); nor reading more messages beyond what has been requested
702 * (surplus).
703 *
704 * That said, we must inform them of completion via on_done_func(), whereas send_blob() has no
705 * such requirement; easy enough -- we just save it as needed. In that sense it is much like
706 * opposing async_end_sending().
707 *
708 * ...Well, no, there's one more complicating matter -- though it's nowhere near as hairy as what rcv-side of
709 * Native_socket_stream::Impl has to deal with: Sure, there is no queuing of requests, but the outgoing-direction
710 * algorithm makes the low-level payloads and then just sends them out -- what's inside doesn't matter (almost).
711 * Incoming-direction is harder, because we need to read payload 1, interpret it, possibly read payload 2.
712 * So there is a bit of tactical nonsense about async-waiting and then resuming from the same spot and so on.
713 * However, unlike with Native_socket_stream::Impl our message boundaries are always preserved, so it is a ton
714 * simpler. Basically it's just a matter of keeping track of the flag m_control_state and acting differently
715 * depending on whether it's true or not (when analyzing an incoming low-level payload). */
716
717 if (m_user_request)
718 {
719 FLOW_LOG_WARNING("Blob_stream_mq_receiver [" << *this << "]: Async-receive requested, but the preceding such "
720 "request is still in progress; the message has not arrived yet (we are awaiting "
721 "readability, before we can get the rest of it). Likely a user error, but who are "
722 "we to judge? Ignoring.");
723 return false;
724 }
725 // else
726
727 Error_code sync_err_code;
728
729 FLOW_LOG_TRACE("Blob_stream_mq_receiver [" << *this << "]: User async-receive request for "
730 "blob (located @ [" << target_blob.data() << "] of max size [" << target_blob.size() << "]).");
731
732 if (m_pending_err_code)
733 {
734 FLOW_LOG_WARNING("Blob_stream_mq_receiver [" << *this << "]: User async-receive request for "
735 "blob (located @ [" << target_blob.data() << "] of "
736 "max size [" << target_blob.size() << "]): Error already encountered earlier. Emitting "
737 "via sync-args.");
738
739 sync_err_code = m_pending_err_code;
740 *sync_sz = 0;
741 }
742 else // if (!m_pending_err_code)
743 {
744 // This next check: background can be found by following the comment on this concept constant.
745 static_assert(!Blob_stream_mq_receiver<Mq>::S_BLOB_UNDERFLOW_ALLOWED,
746 "MQs disallow even trying to receive into a buffer that could underflow "
747 "with the largest *possible* message -- even if the actual message "
748 "happens to be small enough to fit.");
749 if (target_blob.size() < receive_blob_max_size())
750 {
751 FLOW_LOG_WARNING("Blob_stream_mq_receiver [" << *this << "]: User async-receive request for "
752 "blob (located @ [" << target_blob.data() << "] of "
753 "max size [" << target_blob.size() << "]): "
754 "that size underflows MQ size limit [" << receive_blob_max_size() << "]. "
755 "Emitting error immediately; but pipe continues. Note that if we don't do this, "
756 "then the low-level MQ will behave this way anyway.");
757 // As mandated by concept *do not* hose pipe by setting m_pending_err_code given INVALID_ARGUMENT. Just emit.
758
759 sync_err_code = error::Code::S_INVALID_ARGUMENT;
760 *sync_sz = 0;
761 }
762 else
763 {
764 m_user_request.emplace();
765 m_user_request->m_target_blob = target_blob;
766 m_user_request->m_on_done_func = std::move(on_done_func);
767
768 read_msg(&sync_err_code, sync_sz);
769 } // else if (target_blob.size() is fine)
770 } // else if (!m_pending_err_code)
771
772 if ((!sync_err_code) || (sync_err_code != error::Code::S_SYNC_IO_WOULD_BLOCK))
773 {
774 FLOW_LOG_TRACE("Async-request completed synchronously (result "
775 "[" << sync_err_code << "] [" << sync_err_code.message() << "]); emitting synchronously and "
776 "disregarding handler.");
777 m_user_request.reset(); // No-op if we didn't set it up due to early error being detected above.
778 }
779 // else { Other stuff logged enough. }
780
781 // Standard error-reporting semantics.
782 if ((!sync_err_code_ptr) && sync_err_code)
783 {
784 throw flow::error::Runtime_error(sync_err_code, "Blob_stream_mq_receiver_impl::async_receive_blob()");
785 }
786 // else
787 sync_err_code_ptr && (*sync_err_code_ptr = sync_err_code);
788 // And if (!sync_err_code_ptr) + no error => no throw.
789
790 return true;
791} // Blob_stream_mq_receiver_impl::async_receive_blob()

793template<typename Persistent_mq_handle>
794bool Blob_stream_mq_receiver_impl<Persistent_mq_handle>::idle_timer_run(util::Fine_duration timeout)
795{
796 using util::Fine_duration;
797 using util::Task;
798 using boost::chrono::round;
799 using boost::chrono::milliseconds;
800
801 // We comment liberally, but tactically, inline; but please read the strategy in the class doc header's impl section.
802
803 assert(timeout.count() > 0);
804
805 if (!op_started("idle_timer_run()"))
806 {
807 return false;
808 }
809 // else
810
811 // According to concept requirements we shall no-op/return false if duplicately called.
812 if (m_idle_timeout != Fine_duration::zero())
813 {
814 FLOW_LOG_WARNING("Blob_stream_mq_receiver [" << *this << "]: User wants to start idle timer, but they have "
815 "already started it before. Therefore ignoring.");
816 return false;
817 }
818 // else
819
820 m_idle_timeout = timeout; // Remember this, both as flag (non-zero()) and to know to when to schedule it.
821 // Now we will definitely return true (even if an error is already pending). This matches concept requirements.
822
823 if (m_pending_err_code)
824 {
825 FLOW_LOG_INFO("Blob_stream_mq_receiver [" << *this << "]: User wants to start idle timer, but an error has "
826 "already been found and emitted earlier. It's moot; ignoring.");
827 return true;
828 }
829 // else
830
831 FLOW_LOG_INFO("Blob_stream_mq_receiver [" << *this << "]: User wants to start idle-timer with timeout "
832 "[" << round<milliseconds>(m_idle_timeout) << "]. Scheduling (will be rescheduled as needed).");
833
834 /* Per requirements in concept, start it now; reschedule similarly each time there is activity.
835 *
836 * The mechanics of timer-scheduling are identical to those in opposing auto_ping() and are explained there;
837 * keeping comments light here. */
838
839 m_ev_wait_func(&m_ev_wait_hndl_idle_timer_fired_peer,
840 false, // Wait for read.
841 boost::make_shared<Task>
842 ([this]() { on_ev_idle_timer_fired(); }));
843
844 m_idle_timer.expires_after(m_idle_timeout);
845 m_timer_worker.timer_async_wait(&m_idle_timer, m_idle_timer_fired_peer);
846
847 return true;
848} // Blob_stream_mq_receiver_impl::idle_timer_run()

850template<typename Persistent_mq_handle>
851void Blob_stream_mq_receiver_impl<Persistent_mq_handle>::on_ev_idle_timer_fired()
852{
853 /* This is an event handler! Specifically for the *m_idle_timer_fired_peer pipe reader being
854 * readable. To avoid infinite-loopiness, we'd best pop the thing that was written there. */
855 m_timer_worker.consume_timer_firing_signal(m_idle_timer_fired_peer);
856
857 if (m_pending_err_code)
858 {
859 FLOW_LOG_TRACE("Blob_stream_mq_receiver [" << *this << "]: Idle timer fired: There's been 0 traffic past idle "
860 "timeout. However an error has already been found and emitted earlier. Therefore ignoring.");
861 return;
862 }
863 // else
864
865 m_pending_err_code = error::Code::S_RECEIVER_IDLE_TIMEOUT;
866
867 FLOW_LOG_WARNING("Blob_stream_mq_receiver [" << *this << "]: Idle timer fired: There's been 0 traffic past idle "
868 "timeout. Will not proceed with any further low-level receiving. If a user async-receive request "
869 "is pending (is it? = [" << bool(m_user_request) << "]) will emit to completion handler.");
870
871 if (m_user_request)
872 {
873 // Prevent stepping on our own toes: move/clear it first / invoke handler second.
874 const auto on_done_func = std::move(m_user_request->m_on_done_func);
875 m_user_request.reset();
876 on_done_func(m_pending_err_code, 0);
877 FLOW_LOG_TRACE("Blob_stream_mq_receiver [" << *this << "]: Handler completed.");
878 }
879
880 /* That's it. If m_user_request (we've just nullified it) then async read chain will be finished forever as
881 * soon as the user informs us of readability (if it ever does) -- we will detect there's an error
882 * in m_pending_err_code already (and hence no m_user_request). */
883} // Blob_stream_mq_receiver_impl::on_ev_idle_timer_fired()

888 using util::Fine_duration;
889
890 if (m_idle_timeout == Fine_duration::zero())
891 {
892 return;
893 }

895

921template<typename Persistent_mq_handle>
922void Blob_stream_mq_receiver_impl<Persistent_mq_handle>::read_msg(Error_code* sync_err_code, size_t* sync_sz)
923{
924 using util::Task;
925 using util::Blob_mutable;
926 using flow::util::Lock_guard;
927 using raw_ctl_cmd_enum_t = std::underlying_type_t<Control_cmd>;
928
929 assert(!m_pending_err_code);
930 assert(m_user_request);
931
932 /* We just entered possibly-not-would-block state of m_mq; and are about to nb-read message (which might would-block
933 * and other possibilities). Pre-condition is m_user_request is truthy -- has not been satisfied. Lastly
934 * m_control_state may be true or false depending on the last payload (if any).
935 *
936 * -# MQ error or protocol error (invalid ctl msg) or graceful-close ctl msg => m_pending_err_code becomes true;
937 * emit to m_user_request; GTFO.
938 * -# Would-block => async-wait for readability, when ready read_msg() (us) again (unless
939 * idle-timeout in the meantime, so be sure to not read_msg() in that case). GTFO here
940 * (algorithm continues asynchronously).
941 * -# Valid msg:
942 * -# If m_control_state already: So then it's PING => m_control_state=false; register non-idleness; go again*.
943 * -# Else:
944 * -# If empty msg => m_control_state=true; go again*.
945 * -# Else: regular user msg => emit to m_user_request; GTFO.
946 *
947 * (*) - What is "go again"? It means repeat the above steps. One way to structure this would be for read_msg()
948 * to then call read_msg(). That is algorithmically sound conceptually, but for example suppose there are
949 * 100 PINGs queued up; we do not want 100+ stack frames. So we avoid this recursion; instead it's a
950 * do-while loop, with the above steps being one iteration.
951 *
952 * Let us consider the possible outcomes of an iteration as listed above.
953 * -# m_pending_err_code truthy, emit-to-user, do not reiterate.
954 * -# m_pending_err_code falsy, emit-to-user, do not reiterate.
955 * -# Async-wait started, do not emit-to-user, do not reiterate.
956 * -# Reiterate. (m_control_state would have changes to true or to false.)
957 *
958 * Here is how we structure the flow control then. Firstly, outcome (3) stylistically we normally handle in various
959 * places by `return`ing after starting async-wait. It's nice visually to show that when the algorithm continues
960 * asynchronously, then we don't do anything at the tail end (after the async-wait). So, that's that for outcome (3).
961 *
962 * That leaves 2 possibilities if end of iteration is reached:
963 * - emit-to-user, stop loop;
964 * - do not emit-to-user, continue loop.
965 *
966 * Hence track it with `bool emit`; and end the do-while() once it becomes true.
967 */
968 bool emit = false;
969 Blob_mutable target_blob;
970
971 do
972 {
973 FLOW_LOG_TRACE("Blob_stream_mq_receiver [" << *this << "]: Async-receive: Start of payload: "
974 "Either user message or CONTROL-state escape message; unless already in CONTROL state "
975 "(are we? = [" << m_control_state << "]) -- then the CONTROL command enum value.");
976
977 target_blob = m_control_state ? Blob_mutable(m_target_control_blob.data(),
978 m_target_control_blob.size())
979 : m_user_request->m_target_blob;
980
981 const bool rcvd_else_would_block_or_error
982 = m_mq->try_receive(&target_blob, &m_pending_err_code);
983
984 if (!m_pending_err_code)
985 {
986 if (rcvd_else_would_block_or_error)
987 {
988 // Not error, not would block: got low-level message. Interpret it.
989
990 /* Let's discuss protocol negotiation, as it applies to us. Simply, the first thing we must receive
991 * (though there's no requirement as to how soon it happens -- it just needs to precede anything else)
992 * is a CONTROL message, whose payload (the 2nd MQ-message) contains a negative number (unlike regular
993 * CONTROL messages, wherein it's a non-negative number encoding Control_cmd enum: 0, 1, ...), the
994 * negation of which is the opposing side's preferred (highest supported) protocol version to pass-to
995 * m_protocol_negotiator.compute_negotiated_proto_ver() -- which completes the negotiation.
996 *
997 * Since m_protocol_negotiator offers ability to track "first message" or "not first message," we don't
998 * need to keep our own m_ state about that. Thus: */
999 bool proto_negotiating
1000 = m_protocol_negotiator.negotiated_proto_ver() == Protocol_negotiator::S_VER_UNKNOWN;
1001 /* If false, and we haven't errored-out of the loop, then negotiation is behind us.
1002 * If true, then in a couple places below we'll need to zealously handle the negotiation.
1003 * Generally speaking, the tactics we follow (when `true`) are consistent with Protocol_negotiator doc
1004 * header convention which mandates that we interpret as little as we can, until proto_negotiating
1005 * becomes true, or we error out due to negotiation fail. */
1006
1007 if (m_control_state)
1008 {
1009 // Do not forget!
1010 m_control_state = false;
1011
1012 if (target_blob.size() == sizeof(Control_cmd))
1013 {
1014 const auto& cmd = *(reinterpret_cast<const Control_cmd*>(target_blob.data()));
1015 const auto raw_cmd = raw_ctl_cmd_enum_t(cmd);
1016
1017 if (proto_negotiating)
1018 {
1019 /* Protocol_negotiator handles everything (invalid value, incompatible range...); we just know
1020 * the encoding is to be the negation of the version. */
1021#ifndef NDEBUG
1022 const bool ok =
1023#endif
1024 m_protocol_negotiator.compute_negotiated_proto_ver(-raw_cmd, &m_pending_err_code);
1025 assert(ok && "Protocol_negotiator breaking contract? Bug?");
1026 proto_negotiating = false; // Just in case (maintainability).
1027
1028 /* Cool, so now either all is cool, and we should keep reading; or m_pending_err_code contains how
1029 * negotiation failed, and loop will exit. The general code before the } in `if (m_control) {}` will
1030 * handle it either way. */
1031 } // if (proto_negotiating)
1032 else if (raw_cmd >= raw_ctl_cmd_enum_t(Control_cmd::S_END_SENTINEL)) // && !proto_negotiating
1033 {
1034 FLOW_LOG_WARNING("Blob_stream_mq_receiver [" << *this << "]: In CONTROL state expecting encoding "
1035 "enum value < int[" << raw_ctl_cmd_enum_t(Control_cmd::S_END_SENTINEL) << "] but got "
1036 "[" << raw_cmd << "]; emitting protocol error (bug on sender side?).");

1125 * the user will be performing the conceptual async_wait() for us. We must ask them to do so
1126 * via m_ev_wait_func(), giving them m_mq's FD -- m_ev_wait_hndl_mq -- to wait-on. */

1190 "(readable or error, we do not know which yet). We endeavour to receive->pop->receive->... "

1228 })); // m_ev_wait_func(): on_active_ev_func arg

1233 } // else if (!rcvd_else_would_block_or_error) (and !m_pending_err_code, so in fact would-block)
