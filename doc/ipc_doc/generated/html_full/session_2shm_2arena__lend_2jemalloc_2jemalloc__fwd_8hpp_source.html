67/**
68 * This is to session::Client_session what shm::arena_lend::jemalloc::Server_session is to session::Server_session.
69 * In terms of additional (to that of #Client_session) public behavior, from the public API user's point of view,
70 * this class template alias is almost identical to shm::arena_lend::jemalloc::Server_session (see its doc header).
71 * (That is during PEER state.) There is however one difference:
72 *
73 * #Client_session lacks an `app_shm()` method (as well as the related APIs
74 * `app_shm_builder_config()` and `app_shm_lender_session()`). This is due to the nature of arena-lending
75 * SHM-providers. Namely each side maintains its own `Arena`s in which only it can allocate (`Arena::construct()`);
76 * and by definition of session-client, there is no longer-than-lifetime object to maintain any app-scope
77 * `Arena`s. In a session-server process, by contrast, there is Session_server which does
78 * just that and maintains `app_shm()`; hence `Server_session` can return one of those.
79 *
80 * @internal
81 * ### Implementation ###
82 * First see the Implementation section of shm::arena_lend::jemalloc::Server_session. It gives a detailed
83 * description of the overall design -- both its own side and this (client) side too. So read it; then come back here.
84 *
85 * That said, shm::arena_lend::jemalloc::Client_session is a much simpler situation. It is not inter-operating with
86 * any server object; and (as explained in the referenced doc header) all it needs to do is create 1 each of
87 * `Shm_session` and `Arena` objects and register the latter with the former to enable lending objects
88 *
89 * So all we have to do is provide a modified `async_connect()` which does the above. How? Answer:
90 * - execute vanilla session::Client_session_impl::async_connect(); once that triggers the on-done handler:
91 * - set up `Shm_session` and `Arena` (though for the former there's a prerequisite async step to establish an
92 * interal-use IPC channel);
93 * - invoke the user's original on-done handler.
94 *
95 * In `*this`, mechanically: the true implementation of the needed setup and accessors (explained above) is
96 * split between shm::classic::Session_mv (common with `Server_session_impl`) and
97 * shm::arena_lend::jemalloc::Client_session_impl, with the vanilla core in super-class
98 * session::Client_session_impl.
99 *
100 * That's the key; then session::Client_session_mv adds movability around session::Server_session_impl; and lastly
101 * this type aliases to *that* (by way of `Session_mv`). Session_mv completes the puzzle by pImpl-forwarding to the
102 * added (SHM-focused) API. (session::Client_session_mv only pImpl-forwards to the vanilla API.)
103 *
104 * The last and most boring piece of the puzzle are the pImpl-lite wrappers around the SHM-specific API
105 * that shm::arena_lend::jemalloc::Client_session adds to super-class Client_session_mv: `session_shm()` and so on.
106 * Just see same spot in shm::arena_lend::jemalloc::Server_session doc header; it explains both
107 * client and server sides.
108 * @endinternal
109 *
110 * @tparam S_MQ_TYPE_OR_NONE
111 * Identical to session::Client_session.
112 * @tparam S_TRANSMIT_NATIVE_HANDLES
113 * Identical to session::Client_session.
114 * @tparam Mdt_payload
115 * Identical to session::Client_session.
116 */
117template<session::schema::MqType S_MQ_TYPE_OR_NONE, bool S_TRANSMIT_NATIVE_HANDLES,
118 typename Mdt_payload = ::capnp::Void>
119using Client_session
120 = Session_mv
121 <session::Client_session_mv
122 <Client_session_impl<S_MQ_TYPE_OR_NONE, S_TRANSMIT_NATIVE_HANDLES, Mdt_payload>>>;
