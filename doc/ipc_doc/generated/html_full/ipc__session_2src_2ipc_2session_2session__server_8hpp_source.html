29/**
30 * To be instantiated typically once in a given process, an object of this type asynchronously listens for
31 * Client_app processes each of which wishes to establish a *session* with this server process;
32 * emits resulting Server_session objects locally.
33 *
34 * ### When to use ###
35 * In the ipc::session paradigm: See #Server_session doc header "When to use" section. If and only if,
36 * based on that text, you wish to use this application to instantiate `Server_session`s, then you must
37 * instantiate a single Session_server to emit them. (It is not possible to directly construct
38 * a PEER-state -- i.e., at all useful -- #Server_session.)
39 *
40 * Note that while in a typical (non-test/debug) scenario one would only instantiate a single
41 * Session_server, that does *not* mean it cannot or should not instantiate `Client_session`s as well.
42 * In a *given* split, a given application is either the server or the client; but in a different split it
43 * be the other thing. So a Session_server handles all the splits in which this application is the
44 * server; but for other splits one would instantiate 0+ `Client_session`s separately.
45 *
46 * ### How to use ###
47 * Similarly to transport::Native_socket_stream_acceptor constructing `*this` immediately async-listens to
48 * incoming connections (each an attempt to establish a opposing Client_session to partner with a new local
49 * #Server_session). async_accept() shall accept such connections, one at a time, in FIFO order (including any
50 * that may have queued up in the meantime, including after mere construction). The meaning of FIFO order
51 * is discussed below.
52 *
53 * Once a #Server_session is emitted to a handler passed to async_accept(), refer to #Server_session doc header
54 * for how to operate it. Spoiler alert: you must still call `Server_session::init_handlers()`; at which point
55 * the #Server_session is a Session concept impl in PEER state.
56 *
57 * ### Error handling ###
58 * The model followed is the typical boost.asio-like one: Each async_accept() takes a handler, F, and it is
59 * eventually invoked no matter what: either with a falsy (success) #Error_code, or with a truthy (fail) one.
60 * If the op does not complete, and one invokes the `*this` dtor, then `F()` is invoked at dtor time with
61 * a particular operation-aborted #Error_code.
62 *
63 * That said, unlike with (say) transport::Native_socket_stream_acceptor -- and many other classes in ::ipc
64 * and boost.asio -- the failure of a given async_accept() does *not* formally hose `*this`. Formally,
65 * other async_accept(), including future ones, may succeed. Rationale:
66 * - The high-level nature of this class is such that it seemed important to not disregard failed incoming
67 * connect attempts by simply not emitting them. The user might monitor/report based on the nature
68 * of each problematic one. In particular, log-in failures are of interest w/r/t safety + security.
69 * - Given that point, there is no reason to necessarily hose the entire IPC framework for this process (as a
70 * server) due to one badly behaving opposing client.
71 * - Is is unlikely that a Unix domain socket listener will fail, if it was able to begin listening.
72 * So post-ctor problems are unlikely to occur strictly before the log-in exchange procedure.
73 *
74 * Informally there are a few ways one could deal with each given error.
75 * - Simply treating *any* failed async_accept() as `*this`-hosing may well be reasonable. If something
76 * went wrong, perhaps it is safest to exit and let the daemon restart in hopes things will start working.
77 * - Alternatively one can look for particular #Error_code values. If it has to do with log-in
78 * continue trying `async_accept()`s -- perhaps up to a certain number of failures. If it's something not
79 * specifically explainable, revert to the previous bullet point (die, restart). The possible #Error_code
80 * values emitted by async_accept() are documented in its doc header.
81 *
82 * As of this writing I (ygoldfel) lack sufficient data in the field to make certain pronouncements.
83 *
84 * ### Relationship with Server_app and Client_app; relationship with file system ###
85 * This class operates at quite a high level; there should be (outside of test/debug) only on of them in a process.
86 * There are some important relationships to understand:
87 * - This process must be invoked consistently with the description in Server_app as passed to its ctor.
88 * See Server_app (and therefore App) doc header. In particular its App::m_exec_path, App::m_user_id,
89 * App::m_group_id must be consistent with reality.
90 * - The Client_app::Master_set of all `Client_app`s must be equal between all applications (this one and
91 * the potential client apps included) in the relevant IPC universe -- equal by value and therefore
92 * with the same key set (key = App::m_name).
93 * - Similarly for Server_app::Master_set of all `Server_app`s.
94 * - Server_app::m_allowed_client_apps shall list all `Client_app`s that may indeed open sessions against
95 * this application. Any other attempts will be rejected.
96 * - Furthermore each given client app instance (process) will need to know certain address information about
97 * this server process so as to connect to the right guy (not just application but instance -- e.g., to not
98 * connect to some old/dying/zombified instance). While details are intentionally internal there is an
99 * interaction with the file system that you should know: The datum internally known as the *server app namespace*
100 * is the key thing that varies from instance to instance. It is, in fact, the PID (process ID) which has
101 * good uniqueness properties. This is saved in the so-called Current Namespace Store (CNS) which is
102 * simply a PID file, a fairly typical construct in Unix daemons. Therefore:
103 * - Session_server writes this in its ctor.
104 * - The name is, as of this writing, X.pid, where X is Server_app::m_name.
105 * - The location is either a sensible default (/var/run as of this writing) or an override path
106 * Server_app::m_kernel_persistent_run_dir_override. (Reminder: the same `Server_app`s, by value, must
107 * be registered in all processes, whether client or server or both. Hence this override will be
108 * agreed-upon on both sides.)
109 * - Client_session::async_connect() shall check this file immediately to know to which server process --
110 * Session_server -- to connect.
111 *
112 * As of this writing we expect Session_server to be in charge of maintaining the process's only
113 * PID file, in the sense that a typical daemon is expected to maintain a general-use PID file, with the usual
114 * format (where the file contains a decimal encoding of the PID plus a newline). If this
115 * is not acceptable -- i.e., we don't want the ::ipc system to "step" on some other mechanism for PID files
116 * already in use -- then the location may be tweaked, at least via Server_app::m_kernel_persistent_run_dir_override
117 * or possibly with a different default than the current choice (/var/run).
118 *
119 * ### Thread safety; handler invocation ###
120 * After construction of `*this`, concurrent access to API methods and the dtor is not safe, unless all methods
121 * being invoked concurrently are `const`.
122 *
123 * You *may* invoke async_accept() directly from within another async_accept()-passed handler.
124 * (Informally we suggest you post all real handling onto your own thread(s).)
125 *
126 * However: async_accept()-passed user handlers may be invoked *concurrently to each other* (if 2+ async_accept()s are
127 * outstanding). It is your responsibility, then, to ensure you do not invoke 2+ async_accept()s concurrently.
128 * (One way is to only have only 1 outstanding at any given time. Another is to post all handling on your own thread.
129 * Really informally we'd say it's best to do both.)
130 *
131 * ### Subtlety about "FIFO order" w/r/t async_accept() ###
132 * Above we promise that async_accept() shall emit ready `Server_sessions`s in "FIFO" order. The following
133 * expands on that, for completeness, even though it is unlikely a typical user will care. The FIFO order is subtly
134 * different from that in, say, `Native_socket_stream_acceptor` or boost.asio's
135 * transport::asio_local_stream_socket::Acceptor. To wit:
136 *
137 * Suppose async-accept request R1 was followed by R2. As a black box, it is possible that the handler for
138 * R2 will fire before the handler for R1. I.e., the "first out" refers to something different than
139 * the order of emission. Rather: each given request R consists of 2 steps internally (listed here in the
140 * public section only for exposition):
141 * -# Accept a Unix domain socket connection. Immediately upon it being accepted:
142 * -# Execute a log-in exchange on that connection, including things like Client_app identification and verification
143 * and a session token thing.
144 *
145 * The "FIFO order" refers to the first step. Request R1 shall indeed handle the first incoming socket connection;
146 * R2 the second: first-come, first-served. However, the log-in exchange must still occur. In practice this
147 * should be quite quick: Client_session will immediately ping us, and we'll ping them back, and that's that.
148 * However, it is possible that (for whatever reason) R2 will win the race against R1 despite a head start for
149 * the latter. Therefore, it won't look quite like FIFO, though probably it will -- and if not, then close enough.
150 *
151 * Rationale: Honestly: It really has to do with simplicity of implementation (discussed in the impl section below but
152 * of little interest to the user and not included in the public-API Doxygen-generated docs). Implementing it
153 * that way was simpler and easier to maintain; probably the perf less latent too. The fact that, as a result,
154 * the FIFO order might get sometimes switched around did not seem important in practice. Still it is worth
155 * remarking upon formally.
156 *
157 * Informally: are there any practical concerns about this for the user? It seems unlikely. It is, one supposes,
158 * sort-of conceivable that some system would rely on the ordering of incoming sessions being requested to be
159 * significant in some larger inter-application algorithm. However, 1, that sounds highly unorthodox to put it mildly
160 * (more like an abuse of the ipc::session system's versus its intended usefulness); and 2, even then one could then
161 * simply issue async_accept() R2 upon the success of async_accept() R1, if an ordering guarantee is really required.
162 *
163 * @internal
164 * ### Implementation ###
165 * Internally Session_server is really its `private` base: Session_server_impl parameterized
166 * on the vanilla #Server_session_obj it produces. See impl notes for that internal class template.
167 * 100% of the work done by `*this` is done by that base. Why does the base exist then? Answer:
168 * to implement Session_server variants with added capabilities, including the capability of producing
169 * non-vanilla #Server_session variants. For example: shm::classic::Session_server:
170 * - emits shm::classic::Server_session which is like #Server_session but with its own per-session boost.ipc.shm SHM
171 * pool-based arena;
172 * - and which also sets up a per-Client_app SHM arena in a boost.ipc.shm SHM pool, for each new distinct Client_app
173 * to start a session (it creates it if needed by name; and keeps handle open; it's an atomic operation).
174 *
175 * As for this class template, though, we add no such things, because we:
176 * - pass-through a no-op `per_app_setup_func()` that always yields falsy #Error_code;
177 * - emit vanilla Server_session (#Server_session_obj).
178 * @endinternal
179 *
180 * @tparam S_MQ_TYPE_OR_NONE
181 * Emitted `Server_session`s shall have the concrete typed based on this value for `S_MQ_TYPE_OR_NONE`.
182 * See #Server_session_obj.
183 * @tparam S_TRANSMIT_NATIVE_HANDLES
184 * Emitted `Server_session`s shall have the concrete typed based on this value for `S_TRANSMIT_NATIVE_HANDLES`.
185 * See #Server_session_obj.
186 * @tparam Mdt_payload
187 * Emitted `Server_session`s shall have the concrete typed based on this value for `Mdt_payload`.
188 * See #Server_session_obj. In addition the same type may be used for `mdt_from_cli_or_null` (and srv->cli
189 * counterpart) in async_accept(). (Recall that you can use a capnp-`union` internally
190 * for various purposes.)
191 */

223 /**
224 * Constructor: immediately begins listening for incoming session-open attempts from allowed clients.
225 * This will write to the CNS (PID file) and then listen on Unix domain
226 * server socket whose abstract address is based off its contents (as explained in the class doc header).
227 *
228 * If either step fails (file error, address-bind error being the most likely culprits), an error is emitted
229 * via normal Flow error semantics. If this occurs, via the non-exception-throwing invocation style,
230 * then you must not call async_accept(); or behavior is undefined (assertion may trip).
231 *
232 * @param logger_ptr
233 * Logger to use for logging subsequently.
234 * @param srv_app_ref
235 * Properties of this server application. The address is copied; the object is not copied.
236 * @param cli_app_master_set_ref
237 * The set of all known `Client_app`s. The address is copied; the object is not copied.
238 * Technically, from our POV, it need only list the `Client_app`s whose names are
239 * in `srv_app_ref.m_allowed_client_apps`. Refer to App doc header for best practices on
240 * maintaining this master list in practice.
241 * @param err_code
242 * See `flow::Error_code` docs for error reporting semantics. #Error_code generated:
243 * interprocess-mutex-related errors (probably from boost.interprocess) w/r/t writing the CNS (PID file);
244 * file-related system errors w/r/t writing the CNS (PID file) (see class doc header for background);
245 * errors emitted by transport::Native_socket_stream_acceptor ctor (see that ctor's doc header; but note
246 * that they comprise name-too-long and name-conflict errors which ipc::session specifically exists to
247 * avoid, so you need not worry about even the fact there's an abstract Unix domain socket address involved).
248 */

253 /**
254 * Destroys this acceptor which will stop listening in the background and cancel any pending
255 * completion handlers by invoking them ASAP with
256 * session::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER. Subtlety:
257 * expect session::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER, not
258 * transport::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER, regardless of the fact
259 * that internally a transport::Native_socket_stream_acceptor is used.
260 *
261 * You must not call this from directly within a completion handler; else undefined behavior.
262 *
263 * Each pending completion handler will be called from an unspecified thread that is not the calling thread.
264 * Any associated captured state for that handler will be freed shortly after the handler returns.
265 *
266 * We informally but very strongly recommend that your completion handler immediately return if the `Error_code`
267 * passed to it is error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER. This is similar to what one should
268 * do when using boost.asio and receiving the conceptually identical `operation_aborted` error code to an
269 * `async_...()` completion handler. In both cases, this condition means, "we have decided to shut this thing down,
270 * so the completion handlers are simply being informed of this."
271 */
272 ~Session_server();

276 /**
277 * Asynchronously awaits for an opposing Client_session to request session establishment and calls `on_done_func()`,
278 * once the connection occurs and log-in exchange completes, or an error occurs, in the former case move-assigning an
279 * almost-PEER-state Server_session object to the passed-in Server_session `*target_session`.
280 * `on_done_func(Error_code())` is called on success. `on_done_func(E)`, where `E` is a non-success
281 * error code, is called otherwise. In the latter case `*this` may continue operation, and further `async_accept()`s
282 * may succeed. See class doc header regarding error handling.
283 *
284 * Multiple async_accept() calls can be queued while no session-open is pending;
285 * they will grab incoming connections in FIFO fashion as they arrive, but log-ins may race afterwards, resulting
286 * in not-quite-FIFO emission via handler. See class doc header regarding FIFO ordering.
287 *
288 * The aforementioned Server_session generated and move-assigned to `*target_session` on success shall
289 * inherit `this->get_logger()` as its `->get_logger()`.
290 *
291 * `on_done_func()` shall be called from some unspecified thread, not the calling thread *and possibly concurrently
292 * with other such completion handlers*. Your implementation must be non-blocking. Informally we recommend you
293 * place the true on-event logic onto some task loop of your own; so ideally it would consist of essentially a
294 * single `post(F)` statement of some kind.
295 *
296 * You must not call this from directly within a completion handler; else undefined behavior.
297 *
298 * #Error_code generated and passed to `on_done_func()`:
299 * session::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER (destructor called, canceling all pending ops;
300 * spiritually identical to `boost::asio::error::operation_aborted`),
301 * other system codes most likely from `boost::asio::error` or `boost::system::errc` (but never
302 * would-block), indicating the underlying transport::Native_socket_stream_acceptor is hosed for that specific reason,
303 * those returned by transport::Native_socket_stream::remote_peer_process_credentials(),
304 * those emitted by transport::struc::Channel::send(),
305 * those emitted by transport::struc::Channel via on-error handler (most likely
306 * transport::error::Code::S_RECEIVES_FINISHED_CANNOT_RECEIVE indicating graceful shutdown of opposing process
307 * coincidentally during log-in procedure, prematurely ending session while it was starting),
308 * error::Code::S_SERVER_MASTER_LOG_IN_REQUEST_CLIENT_APP_DISALLOWED_OR_UNKNOWN,
309 * error::Code::S_SERVER_MASTER_LOG_IN_REQUEST_CLIENT_APP_INCONSISTENT_CREDS,
310 * error::Code::S_SERVER_MASTER_LOG_IN_REQUEST_CONFIG_MISMATCH,
311 * error::Code::S_INVALID_ARGUMENT (other side expected other async_accept() overload with
312 * non-null `init_channels_by_cli_req` arg),
313 * error::Code::S_SESSION_OPEN_CHANNEL_SERVER_CANNOT_PROCEED_RESOURCE_UNAVAILABLE (unable to acquire init-channel
314 * resources -- as of this writing MQ-related ones).
315 *
316 * ### Error handling discussion ###
317 * See class doc header regarding error handling, then come back here.
318 * The last 3 specific #Error_code values listed can be considered, informally, individual misbehavior on
319 * the part of the opposing client process. While indicating potentially serious misconfiguration in the system,
320 * surely to be investigated ASAP, it is conceivable to continue attempting further `async_accept()`s.
321 *
322 * Do note, though, that this is not a network client-server situation. That is, probably, one should not expect
323 * lots of -- or any -- such problems in a smoothly functioning IPC universe on a given production server machine.
324 * E.g., there could be a security problem, or perhaps more likely there's a software
325 * mismatch between client and server w/r/t their master App sets. Your software deployment story may or may not
326 * be designed to allow for this as a transient condition during software upgrades and such.
327 *
328 * @tparam Task_err
329 * Handler type matching signature of `flow::async::Task_asio_err`.
330 * @param target_session
331 * Pointer to Server_session which shall be assigned an almost-PEER-state (open, requires
332 * Server_session::init_handlers() to enter PEER state) as `on_done_func()`
333 * is called. Not touched on error.
334 * @param on_done_func
335 * Completion handler. See above. The captured state in this function object shall be freed shortly upon
336 * its completed execution from the unspecified thread.
337 */
338 template<typename Task_err>
339 void async_accept(Server_session_obj* target_session, Task_err&& on_done_func);

341 /**
342 * Identical to the simpler async_accept() overload but offers added advanced capabilities: metadata exchange;
343 * initial-channel opening. The other overload is identical to
344 * `async_accept(target_session, nullptr, nullptr, nullptr, ...N/A..., NO_OP_FUNC, on_done_func)`
345 * (where `NO_OP_FUNC()` no-ops) and requires the opposing `async_connect()` to similarly not use the corresponding
346 * features.
347 *
348 * The advanced capabilities complement/mirror the ones on Client_session_mv::async_connect() (advanced overload);
349 * see its doc header first. Then come back here. The same mechanisms underly the following; but due
350 * to client-server asymmetry the API is somewhat different.
351 *
352 * ### Server->client metadata exchange ###
353 * Server may wish to provide information in some way, but without a structured channel -- or any channel -- yet
354 * available, this provides an opportunity to do so in a structured way with a one-off message available at
355 * session open, together with the opposing Session object itself.
356 *
357 * The metadata payload you may wish to load to emit to the opposing `Client_session::async_connect()` may
358 * depend on the following information available *during* the session-open procedure, namely upon receiving
359 * (internally) the log-in request from the client side:
360 * - the Client_app that wishes to open the session;
361 * - how many init-channels the client is requesting be opened (see below);
362 * - the client->Server metadata (see below).
363 *
364 * Thus supply `Mdt_load_func mdt_load_func` arg which takes all 3 of these data, plus a blank
365 * capnp `Mdt_payload` structure to fill; and loads the latter as desired.
366 *
367 * To omit using this feature do nothing in `mdt_load_func()`. (The other async_accept() overload provides such
368 * a no-op function.)
369 *
370 * ### Client->server metadata exchange ###
371 * This is the reverse of the above. Whatever the opposing client chose to supply as client->server metadata
372 * shall be deserializable at `*mdt_from_cli_or_null` once (and if) `on_done_func(Error_code())` (successful
373 * accept) fires. If `mdt_from_cli_or_null` is null, the cli->srv metadata shall be ignored.
374 *
375 * ### Init-channels by server request ###
376 * Once the session is open, open_channel() and the on-passive-open handler may be used to open channels at will.
377 * In many use cases, however, a certain number of channels is required immediately before work can really begin
378 * (and, frequently, no further channels are even needed). For convenience (to avoid asynchrony/boiler-plate)
379 * the init-channels feature will pre-open a requested # of channels making them available right away, together
380 * with the freshly-open session -- to both sides.
381 *
382 * The server may request 0 or more init-channels. They shall be opened and placed into
383 * `*init_channels_by_srv_req`. The number of channels requested may depend on the 3 piece of info outlined
384 * above in "Server->client metadata exchange." Thus supply `N_init_channels_by_srv_req_func
385 * n_init_channels_by_srv_req_func` arg which takes all 3 of these data and returns the `size_t` channel
386 * count (possibly 0). The resulting channels shall be loaded into `*init_channels_by_srv_req`
387 * before successful `on_done_func()` execution.
388 *
389 * If and only if `n_init_channels_by_srv_req_func()` would always return 0, you may provide a
390 * null `init_channels_by_srv_req`. `n_init_channels_by_srv_req_func` will then be ignored.
391 *
392 * ### Init-channels by client request ###
393 * This is the reverse of the above. The opposing side shall request 0 or more init-channels-by-client-request;
394 * that number of channels shall be opened; and they will be placed into `*init_channels_by_cli_req` which shall
395 * be `->resize()`d accordingly, once (and if) `on_done_func(Error_code())` (successful
396 * connect) fires.
397 *
398 * `init_channels_by_srv_cli` being null is allowed, but only if the opposing server requests 0
399 * init-channels-by-server-request. Otherwise an error shall be emitted (see below).
400 *
401 * @tparam Task_err
402 * See other async_accept() overload.
403 * @tparam N_init_channels_by_srv_req_func
404 * See above: function type matching signature
405 * `size_t F(const Client_app&, size_t n_init_channels_by_cli_req, Mdt_reader_ptr&& mdt_from_cli)`.
406 * @tparam Mdt_load_func
407 * See above: function type matching signature
408 * `void F(const Client_app&, size_t n_init_channels_by_cli_req, Mdt_reader_ptr&& mdt_from_cli,
409 * Mdt_builder* mdt_from_srv)`.
410 * @param target_session
411 * See other async_accept() overload.
412 * @param init_channels_by_srv_req
413 * See above: null or pointer to container of `Channel_obj` which shall be `.clear()`ed and replaced
414 * by a container of PEER-state `Channel_obj` on success the number being determined by
415 * `n_init_channels_by_srv_req_func()`. null is treated as-if `n_init_channels_by_srv_req_func() == 0`
416 * for any input.
417 * @param mdt_from_cli_or_null
418 * See above: null or pointer to `Reader` of metadata which shall be set for access on success.
419 * @param init_channels_by_cli_req
420 * See above: null or pointer to container of `Channel_obj` which shall be `.clear()`ed and replaced
421 * by a container of PEER-state `Channel_obj` on success the number being specified by the opposing
422 * (client) side. The number may be zero. null is allowed if and only if the number is zero;
423 * otherwise error::Code::S_INVALID_ARGUMENT is emitted.
424 * @param n_init_channels_by_srv_req_func
