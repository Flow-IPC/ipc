36/**
37 * Implements the SHM-related API common to shm::arena_lend::jemalloc::Server_session and
38 * shm::arena_lend::jemalloc::Client_session. It is, as of this writing, not to be instantiated by itself.
39 * Rather see shm::arena_lend::jemalloc::Server_session (and
40 * shm::arena_lend::jemalloc::Client_session) regarding what to actually instantiate and
41 * further context. As for the API in Session_mv itself, it is quite critical.
42 * shm::arena_lend::jemalloc::Server_session doc header explains the background. Member doc headers formally document
43 * those members given that background. #Arena doc header is particularly useful.
44 *
45 * @tparam Session_t
46 * Class for us to `public`ly sub-class: `Server_session_mv<...>` or `Client_session_mv<...>`.
47 * Note its `public` API is inherited. See the relevant class template's doc header.

59 /**
60 * The arena object on which one may call `construct<T>(ctor_args...)`, where `ctor_args` are arguments
61 * to the `T::T()` constructor. While a full-length discussion of the allowed properties of `T` is outside
62 * out scope here (belongs in ipc::shm docs), here's a short version for convenience:
63 * PODs work; STL nested container+POD combos work, as long as
64 * a shm::stl allocator is used at all levels; manually-implemented non-STL-compliant data
65 * structures work if care is taken to use `Arena::allocate()` and `Arena::Pointer`.
66 *
67 * Suppose `A->construct()` yields handle `h`, where `A` is any `Arena*` returned by a `*this` accessor --
68 * `this->app_shm()`, `this->session_shm()` (the former available only is `*this` is really an instance of sub-class
69 * Server_session_mv). Then `h` can be passed to `this->lend_object()` which yields `Blob b`.
70 * Further, an `h` returned by `s->borrow_object(b)`, executed on the opposing
71 * `shm::arena_lend::jemalloc::Session_mv s` (after receiving `b` via IPC) shall point to the same location in
72 * `A` and will become part of a cross-process `shared_ptr` group with the original `h` (and any copies/moves
73 * thereof), `*h` garbage-collected based on a (conceptual) cross-process ref-count.
74 *
75 * Full #Arena capabilities are described in the documentation for the alias's target,
76 * shm::arena_lend::jemalloc::Ipc_arena.
77 *
78 * `this->lend_object()` and `this->borrow_object()` really forward to the same methods of
79 * #Shm_session. That #Shm_session can itself be accessed via shm_session(). That said it is likely best
80 * to use the forwarding Session_mv methods, as it is conducive to generic programming in which an
81 * alternate-SHM-provider `Session_mv` could be used instead. In particular SHM-classic-based
82 * session::shm::classic::Session_mv::lend_object() and session::shm::classic::Session_mv::borrow_object() APIs
83 * behave identically; whereas `session::shm::classic::Session_mv::shm_session()` does not exist (because the
84 * mechanics of sharing SHM-stored objects in SHM-classic fundamentally differ from those of SHM-jemalloc). Hence
85 * code that uses `Session_mv::lend_object()` and `Session_mv::borrow_object()` would work with either
86 * `Session_mv`.
87 *
88 * @note Unlike with SHM-classic, Session_mv::borrow_object() can accept a `Blob` generated by opposing process's
89 * `Session_mv::Shm_session::lend_object()`; and conversely `Session_mv::Shm_session::borrow_object()`
90 * can accept a `Blob` from Session_mv::lend_object(). While, again, it is probably best to just use
91 * Session_mv methods on both sides, if the need does arise, one can take advantage of this compatibility.
92 */
93 using Arena = typename Base::Base::Impl::Arena;
94
95 /**
96 * The lend/borrow engine type which allows one to "lend" a `T` (obtained via `Arena::construct<T>()`)
97 * to the opposing process, ahead of transmitting the object over a channel obtained from `*this` session.
98 * The main methods of interest -- `lend_object()` and `borrow_object()` -- are explained in context in
99 * #Arena doc header and are more naturally accessed via Session_mv::lend_object() and Session_mv::borrow_object().
100 *
101 * That said, #Shm_session can be used for further advanced purposes. For example one could separately create
102 * an #Arena (beyond those supplied via session_shm() and Server_session_mv::app_shm()) and register it
103 * via `Shm_session::lend_arena()`; then it would be possible to `Shm_session::lend_object()` and transmit
104 * objects `construct()`ed in that additional #Arena. Why do such a thing? Answer: There are various conceivable
105 * use cases. Perhaps a shorter-lived segregated `Arena` is desirable for security or safety reasons for example.
106 *
107 * The point is: ipc::session, per its mission, pre-creates the SHM infrastructure for you thus eliminating the
108 * various difficulties inherent in this task -- while making `Session`s backed by this SHM-provider largely
109 * interchangeable with ones backed by another, such as SHM-classic. In doing so it hides the SHM-provider-specific
110 * work required. Yet it *also* provides to you the ability to use the SHM-provider-specific API directly as
111 * desired. Should you avoid this? Informally: Yes, as it'll make your life easier... when it is sufficient.
112 * If it is *not* sufficient, you have the tools to go beyond it.
113 */
114 using Shm_session = jemalloc::Shm_session;

191 /**
192 * Adds an owner process to the owner set of the given `Arena::construct()`-ed handle; and returns an opaque blob,
193 * such that if one passes it to opposing Session_mv::borrow_object() in the receiving process, that
194 * `borrow_object()` shall return an equivalent `Handle` in that process. The returned `Blob`
195 * is guaranteed to have non-zero size that is small enough to be considered very cheap to copy and hence
196 * transmit over IPC.
197 *
198 * It is the user's responsibility to transmit the returned blob, such as via a transport::Channel,
199 * to the owning process. Failing to do so may leak the object until arena cleanup. (Arena cleanup time
200 * depends on the source #Arena. If it came from session_shm(), arena cleanup occurs at Session destruction.
201 * If from Server_session_mv::app_shm(), arena cleanup occurs at Server_session destruction. If it is a
202 * custom-created other #Arena -- not managed by ipc::session -- then it would occur whenever you choose to destroy
203 * that #Arena. If a destructor does not run, due to crash/etc., then the leaked ipc::session-managed `Arena`s'
204 * pools are cleaned zealously by ipc::session via a heuristic algorithm.)
205 *
206 * ### What it really does ###
207 * It forwards to `shm_session()->lend_object()`. I mention this for context; it is arguably desirable to not
208 * count on these details in code that can generically work with a different SHM-enabled Session, such as
209 * shm::classic::Session_mv, taken as a template param. E.g., shm::arena_lend::jemalloc::Session_mv::lend_object()
210 * does not, and cannot (as it does not exist), call any `Session_mv::shm_session()` on which to invoke
211 * `lend_object()`.
212 *
213 * However, if your code specifically counts on `*this` being a shm::arena_lend::jemalloc::Server_session, then
214 * it is not wrong to rely on this knowledge.
215 *
216 * @tparam T
217 * See #Arena.
218 * @param handle
219 * Value returned by #Arena method `construct<T>()`. Note this is a mere `shared_ptr<T>` albeit with
220 * unspecified custom deleter logic attached. The #Arena instance can be session_shm(),
221 * Server_session_mv::app_shm(), or a non ipc::session-managed custom #Arena, as long as it has been
222 * registered via `shm_session()->lend_arena()`.
223 * @return See above. Never `.empty()`.
224 */
225 template<typename T>
226 Blob lend_object(const typename Arena::template Handle<T>& handle);
227
228 /**
229 * Completes the cross-process operation begun by oppsing Session_mv::lend_object() that returned `serialization`;
230 * to be invoked in the intended new owner process which is operating `*this`.
231 *
232 * @tparam T
233 * See lend_object().
234 * @param serialization
235 * Value, not `.empty()`, returned by opposing lend_object() and transmitted bit-for-bit to this process.
236 * @return See above. Never null.
237 */
238 template<typename T>
239 typename Arena::template Handle<T> borrow_object(const Blob& serialization);

260 /**
261 * Returns shm_reader_config().
262 *
263 * ### Rationale ###
264 * Why does this exist, given that shm_reader_config() exists? Answer: It is for generic programming,
265 * for code that would like to work independently of which SHM-enabled Session is in use; for example
266 * interchangeably between `*this` class or shm::classic::Session_mv. Hence code that wants to set up
267 * a transport::struc::Msg_in (or transport::struc::Channel that receives them), and requires
268 * a reader-config object capable of decoding messages stored in the opposing process's `session_shm()`
269 * (session-scope #Arena), then this method can be used -- and will work, because shm_reader_config()
270 * will work for *any* `Arena` -- which trivially includes opposing `session_shm()`.
271 *
272 * For example, transport::struc::Channel_base::Serialize_via_session_shm tag ctor of
273 * transport::struc::Channel will internally invoke this when loading up its internal
274 * per-session reader-config for you.
275 *
276 * @return See above.
277 */
278 Structured_msg_reader_config session_shm_reader_config();
279
280 /**
281 * Returns shm_reader_config().
282 *
283 * ### Rationale ###
284 * Why does this exist, given that shm_reader_config() exists? Answer: It is for generic programming,
285 * for code that would like to work independently of which SHM-enabled Session is in use; for example
286 * interchangeably between `*this` class or shm::classic::Session_mv. Hence code that wants to set up
287 * a transport::struc::Msg_in (or transport::struc::Channel that receives them), and requires
288 * a reader-config object capable of decoding messages stored in the opposing process's `app_shm()`
289 * (session-scope #Arena), then this method can be used -- and will work, because shm_reader_config()
290 * will work for *any* `Arena` -- which trivially includes opposing `app_shm()`.
291 *
292 * For example, transport::struc::Channel_base::Serialize_via_session_shm tag ctor of
293 * transport::struc::Channel will internally invoke this when loading up its internal
294 * per-app reader-config for you.
295 *
296 * What if `*this` is really super-classing Server_session_mv? The opposing object is of type
297 * Client_session_mv which lacks `app_shm()`; so isn't the present method meaningless? Not exactly:
298 * if someday a message *were* to arrive from a somehow-app-scope-#Arena-capable client, then this
299 * would decode it just fine. As of this writing that's impossible, so it's a moot question;
300 * yet it allows for code such as the aforementioned transport::struc::Channel tag ctor to be written
301 * simply in a way that'll work for shm::classic (which does have a shm::classic::Session_mv::app_shm()) and
302 * `*this` class as well.
303 *
304 * @return See above.
305 */
306 Structured_msg_reader_config app_shm_reader_config();
