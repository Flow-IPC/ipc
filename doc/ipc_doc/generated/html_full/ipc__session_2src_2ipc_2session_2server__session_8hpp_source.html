30/**
31 * Implements Session concept on the Server_app end: a Session that is emitted in almost-PEER state by
32 * local Session_server accepting a connection by an opposing `Client_session_mv::sync_connect()`.
33 * By "almost-PEER state" we mean that the user, upon obtaining a new `Server_session_mv`, must still call
34 * init_handlers() to enter PEER state. See overview of Session hierarchy
35 * in namespace ipc::session doc header; then come back here if desired.
36 *
37 * It is unusual to use Server_session_mv template directly. If you do wish to set up a server-side session peer,
38 * and you do not require SHM support, then use #Server_session alias template; target it with a
39 * Session_server::async_accept() call. The server-specific API, particularly init_handlers(),
40 * is in Server_session_mv and documented here. The side-agnostic API --
41 * active once PEER state is achieved -- is described by Session (concept) doc header and implemented concretely
42 * by Session_mv which is our `public`, non-`virtual` super-class.
43 *
44 * If you do wish to set up a server-side session peer, but you *do* require SHM support, then use
45 * shm::classic::Server_session or similar (for other SHM-provider(s)); target it with a
46 * shm::classic::Session_server::async_accept() (or similar for other SHM-provider(s)). However #Server_session
47 * (hence Server_session_mv to which it aliases) is its super-class, and until PEER state is reached its
48 * API (init_handlers()) remains the only relevant API to use. Once Server_session_mv::init_handlers()
49 * puts `*this` into PEER state, super-class Session_mv (= Session concept) API continues to be relevant.
50 * Also in PEER state SHM-relevant additional API members (e.g. shm::classic::Session::app_shm()`) become of interest.
51 * The last two sentences describe the situation identically for Client_session_mv as well (see its doc header
52 * for context).
53 *
54 * Summary hierarchy (contrast with similar spot in Client_session_mv doc header):
55 * - Session_mv (Session concept impl)
56 * - ^-- Server_session_mv (adds init_handlers())
57 * - =alias= #Server_session (parameterizes by transport-configuring knobs = template params)
58 * - ^-- shm::classic::Session_mv
59 * (adds `session_shm()`...; suitable for transport::struc::Channel_base::Serialize_via_session_shm)
60 * - ^-- shm::classic::Server_session (parameterizes by aforementioned knobs = template params)
61 *
62 * We may refer to Server_session_mv as #Server_session below; particularly since it is likeliest used in that
63 * form by the user.
64 *
65 * ### How to use ###
66 * Per the Session concept a #Server_session is open/capable of operation when in PEER state only.
67 * A publicly available #Server_session object is always in one of 2 states:
68 * - Almost-PEER. At this stage you must call init_handlers() to enter PEER state. Until then other APIs
69 * will no-op/return sentinel, and no handler will fire.
70 * - PEER. At this stage it exactly implements the Session concept. See Session_mv. Reminder:
71 * It is not possible to exit PEER state.
72 *
73 * Once in PEER state #Server_session simply follows Session concept semantics. At this stage our super-class
74 * Session_mv implements that concept in particular. See either doc header (Session, Session_mv).
75 *
76 * ### Error handling ###
77 * Once in PEER state, error handling follows the Session concept (= Session_mv concrete class) doc header.
78 * However, up to that point it can emit no errors. You must call init_handlers() first to enter PEER state.
79 *
80 * @internal
81 * ### Implementation design/rationale ###
82 * See section in the same spot of Client_session_mv doc header. A symmetrical situation (w/r/t pImpl-lite
83 * design) occurs here. But let's be explicit at the risk of some copy/pasting:
84 *
85 * Session_mv is the highest (`public`) super-class and begins the pImpl-lite-for-movability technique.
86 * It pImpl-wraps the Session-concept impl methods/stuff. Server_session_mv merely repeats the same
87 * technique but on the API additions at its own level; in particular init_handlers().
88 *
89 * The impl object is available via Session_mv::impl() (which is `protected`). What type does this object
90 * have though? Answer: `Server_session_impl_t`, our template param! This shall be the proper `..._impl`
91 * internal-use type that matches the level of API (`public` sub-class) the user chose: E.g.:
92 * - Chose #Server_session => impl = Server_session_impl.
93 * - Chose shm::classic::Server_session => impl = shm::classic::Server_session_impl.
94 *
95 * So basically each of the public-API types visible to the user adds the pImpl-wrapping of the methods
96 * of the `..._impl` type matching their name; and the public/movable-API hierarchy has a parallel
97 * internal/non-movable hierarchy, both using `public`/non-`virtual` inheritance.
98 *
99 * ### Facade design ###
100 * Session_server emits a Server_session_mv only once it's in almost-PEER state. Before then it is dealing
101 * with it privately. Namely -- details are in Server_session_impl -- it must
102 * -# construct it;
103 * -# call async_accept_log_in() and await its success.
104 *
105 * These APIs are `protected` in Server_session_mv. The detail/ sub-class Server_session_dtl exposes them publicly
106 * (but only accessible, by convention, internally; namely by Session_server).
107 *
108 * As of this writing the SHM-enabled Session_server variants (e.g., shm::classic::Session_server) do not require
109 * any additional internally-accessed APIs. Instead they employ 3 "customization points" using which avoids
110 * such a need; see Session_server_impl. Therefore there is no detail/ facade beyond Server_session_dtl.
111 *
112 * @see Server_session_impl doc header.
113 * @endinternal
114 *
115 * @see Session: implemented concept.
116 *
117 * @tparam Server_session_impl_t
118 * An implementation detail. Use one of the aliases prescribed near the top of this doc header to
119 * set this correctly.
120 */

210 /**
211 * For use by internal user Session_server: called no more than once, ideally immediately following ctor,
212 * this attempts to get `*this` asynchronously to almost-PEER state by undergoing the log-in request/response
213 * (plus, if needed, init-channel-opening) procedure (the other side of which is done by
214 * Client_session_impl::sync_connect()). On success, `on_done_func(Error_code())` is invoked from unspecified
215 * thread that is not the user's calling thread. On failure, it does similarly but with a non-success code.
216 * If the op does not complete before dtor, then
217 * `on_done_func(error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER)` is invoked at that point.
218 *
219 * In the success-case firing, `*this` is ready to be given to the public user. However it is in almost-PEER state.
220 * To achieve PEER state they shall first call init_handlers().
221 *
222 * #Error_code generated and passed to `on_done_func()`:
223 * session::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER (see above),
224 * those returned by transport::Native_socket_stream::remote_peer_process_credentials(),
225 * those emitted by transport::struc::Channel::send(),
226 * those emitted by transport::struc::Channel via on-error handler (most likely
227 * transport::error::Code::S_RECEIVES_FINISHED_CANNOT_RECEIVE indicating graceful shutdown of opposing process
228 * coincidentally during log-in procedure, prematurely ending session while it was starting),
229 * error::Code::S_SERVER_MASTER_LOG_IN_REQUEST_CLIENT_APP_DISALLOWED_OR_UNKNOWN,
230 * error::Code::S_SERVER_MASTER_LOG_IN_REQUEST_CLIENT_APP_INCONSISTENT_CREDS,
231 * those emitted by `pre_rsp_setup_func()` (in-arg).
232 *
233 * @note If the above list substantially changes, please update Session_server::async_accept()
234 * doc header accordingly for user convenience.
235 *
236 * @tparam Session_server_impl_t
237 * See `srv`.
238 * @tparam Task_err
239 * Handler type matching signature of `flow::async::Task_asio_err`.
240 * @tparam Cli_app_lookup_func
241 * Function type that takes a supposed Client_app::m_name and returns the `const Client_app*` with that
242 * matching name; or null if it is not a validly registered name according to the data given to the
243 * internal user.
244 * @tparam Cli_namespace_func
245 * Function type that returns a sufficiently-distinct value for Session_base::cli_namespace(),
246 * as Session_base::set_cli_namespace() is required before #Server_session can be in PEER state and available
247 * to the public user.
248 * @tparam Pre_rsp_setup_func
249 * Function type with signature `Error_code F()`.
250 * @tparam N_init_channels_by_srv_req_func
251 * See Session_server::async_accept(). Type and arg value forwarded from there.
252 * @tparam Mdt_load_func
253 * See Session_server::async_accept(). Type and arg value forwarded from there.
254 * @param srv
255 * The Session_server_impl whose Session_server_impl::async_accept() is invoking the present method.
256 * `*srv` must exist at least while `*this` does, or behavior is undefined.
257 * This allows for interaction/cooperation with the "parent" `Session_server` if necessary, such as
258 * for shared cross-session resources.
259 * Note: you may use `srv->this_session_srv()` to obtain a pointer to the `Session_server`-like object
260 * on which the user invoked `async_accept()`. E.g.: Server_session_impl::async_accept_log_in()
261 * would get a `Session_server*`; shm::classic::Server_session_impl::async_accept_log_in() would get
262 * a `shm::classic::Session_server*`.
263 * @param init_channels_by_srv_req
264 * See Session_server::async_accept(). Arg value forwarded from there.
265 * @param mdt_from_cli_or_null
266 * See Session_server::async_accept(). Arg value forwarded from there.
267 * @param init_channels_by_cli_req
268 * See Session_server::async_accept(). Arg value forwarded from there.
269 * @param cli_app_lookup_func
270 * See `Cli_app_lookup_func`.
271 * @param cli_namespace_func
272 * See `Cli_namespace_func`.
273 * @param pre_rsp_setup_func
274 * Invoked just before sending successful log-in response to opposing client, which completes the log-in.
275 * It takes no arguments -- but all `*this` accessors up to/including `cli_app_ptr()` shall return
276 * real values -- and shall return falsy on success; or the reason for failure as #Error_code.
277 * In the latter case this method shall emit that code as the reason for overall failure.
278 * This can be used for setting up resources, such as SHM arena(s), that the client shall count
279 * on being available (perhaps at known #Shared_name based on `*this` accessor values).
280 * @param n_init_channels_by_srv_req_func
281 * See `N_init_channels_by_srv_req_func`.
282 * @param mdt_load_func
283 * See `Mdt_load_func`.
284 * @param on_done_func
285 * See above.
286 */
287 template<typename Session_server_impl_t,
