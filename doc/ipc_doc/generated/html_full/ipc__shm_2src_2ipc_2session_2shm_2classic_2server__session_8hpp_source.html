33/**
34 * Identical to session::Server_session in every way, except that it makes available two SHM arenas, from the
35 * SHM-classic provider (ipc::shm::classic::Pool_arena), symmetrically accessible by the opposing side. These
36 * SHM arenas (see #Arena doc header) have different scopes:
37 * - Per-session-scope, accessible via session_shm(): meaning it shall be accessible only during the lifetime of
38 * this session, by this Session (via this accessor) and the opposing Session (via its counterpart thereof).
39 * - Hence this arena is created at the time both mutually opposing Session objects, including `*this`,
40 * enter PEER state (more precisely just before the shm::classic::Client_session receives word from
41 * `*this` that log-in is successful). When `*this` is destroyed or hosed, the arena is invalidated.
42 * - Per-app-scope, meaning its pool shall be accessible potentially beyond the lifetime of
43 * this session but rather until the generating Session_server (i.e., server process) shuts down, by any
44 * Session under that umbrella, now or in the future, as long as its Client_app equals that of `*this` session.
45 * - Hence this arena is created at the time the shm::classic::Session_server (emitter of `*this` to user) first
46 * establishes a session with the *first* shm::classic::Client_session matching the same Client_app as `*this`.
47 * I.e., it's created at the same time as `this->session_shm()` is, if `*this` is the first such session;
48 * otherwise earlier. When `*this` is destroyed or hosed (gracefully at least), the arena is *not*
49 * invalidated but persists.
50 *
51 * ### Explicitly constructing an object in SHM; lending to opposing side ###
52 * This setup is symmetrical w/r/t a given shm::classic::Server_session<->shm::classic::Client_session object pair.
53 * Both can create new objects (via respective `Arena::construct()` methods). Both can write and, of course, read.
54 * Lastly each can transmit any outer handle, returned by `construct()`, to the opposing side. To do this,
55 * call lend_object(), transmit the returned `Blob` to the opposing side via any desired IPC form (probably a
56 * transport::Channel or transport::struc::Channel), then recover a handle via borrow_object() on that
57 * opposing side.
58 *
59 * By using `shm::stl::Arena_activator<Arena>` (alias shm::classic::Pool_arena_activator) and
60 * #Allocator (alias shm::classic::Pool_arena_allocator) it is possible to construct and transmit not just POD (Plain
61 * Old Datatype) objects but combinations of those with unlimited nested levels of STL-compliant containers.
62 * On the borrowing side use #Borrower_allocator for maximum SHM-provider-independence of your code. (But see
63 * the following note: your algorithm may require features not available in the alternative, SHM-jemalloc,
64 * in which case... well, read the note.)
65 *
66 * @note For SHM-classic, #Borrower_allocator *is* #Allocator. If the borrowing side means to make changes to
67 * the data itself, it (stylistically speaking) can simply use #Allocator (and `Arena_activator<Arena>` again).
68 * However, if the borrowing side maintains read-only access throughout, then by using #Borrower_allocator
69 * the code will remain SHM-provider-agnostic and continue working if the ipc::session sub-namespace
70 * is changed back and forth betweeen `classic` and `arena_lend::jemalloc`.
71 *
72 * Detailed instructions can be found in those shm::stl and ipc::shm::arena_lend doc headers.
73 *
74 * ### Using SHM as backing memory for transport::struc::Channel ###
75 * Another use of session_shm() and app_shm() -- in some ways perhaps superior -- is indirect.
76 * The above technique involves manually constructing a C++ data structure and transmitting a short handle
77 * to it over IPC thus achieving high performance. It may be desirable, instead, to use a structured message,
78 * filled out according to schema. This, of course, is why transport::struc::Channel exists.
79 * As is explained in its docs, there is nothing SHM-centered about it. It does however have the key feature
80 * that allows one to supply, at compile-time, the builder and reader engines that acquire memory
81 * while the user mutates an out-message and later accesses it upon IPC-receipt. Naturally one of the
82 * builder/reader pairs uses SHM, instead of regular heap, as the supplier of RAM. Thus when
83 * one invokes transport::struc::Channel::send() and receives it in the opposing object, the actual bits
84 * copied into/out of the low-level transport are merely the SHM handle (and all of this is hidden from the user
85 * outside of the delightful near-zero-copy perf properties).
86 *
87 * There are two ways to make this happen. The easiest and best way is, when constructing the
88 * `struc::Channel`, to use the tag-form ctor with tag
89 * transport::struc::Channel_base::Serialize_via_shm_classic. Simply provide that tag,
90 * `this` (or, symmetrically, `shm::classic::Client_session::this` on the other side), and specify which of the
91 * 2 scopes you desire (per-session or per-app: a `bool`).
92 *
93 * The harder way, albeit allowing for certain advanced setups, is to manually create a
94 * `transport::struc::shm::classic::Builder::Config` and/or `Reader::Config`,
95 * passing in `this->session_shm()` and/or `this->app_shm()`, to those;
96 * and then pass the `Config` or `Config`s to the non-tag-form of `struc::Channel`
97 * ctor.
98 *
99 * This is all documented on transport::struc::Channel. Do realize, though, that those niceties are
100 * really built on this class template and/or the opposing shm::classic::Client_session. To use them with
101 * ipc::session, you must thus choose shm::classic::Server_session and shm::classic::Client_session as your
102 * Session impls.
103 *
104 * @internal
105 * ### Implementation ###
106 * It is probably not too difficult to understand how it works by just reading the code and doc headers.
107 * The key is that the core is still just what's supplied by a vanilla (non-SHM-aware) #Server_session.
108 * Ultimately all we need to add is a bit of setup near the end of the vanilla session-opening procedure.
109 * This is easiest to discuss holistically, meaning both the server (`*this`) and client (opposing) side,
110 * so the discussion is all here.
111 *
112 * As with the vanilla session setup, remember that while in PEER state the APIs are identical/symmetrical,
113 * the roles of the server and client vary quite sharply internally -- especially before PEER state.
114 * The general outline is: Session_server listens for socket connections; a given
115 * `Client_session_impl::async_connect()` connects; at this stage Session_server constructs a not-yet-user-emitted
116 * #Server_session and invokes its `async_accept_log_in()`. Now the #Client_session and #Server_session
117 * have established a channel (internal-use session master channel) and undergo the log-in exchange.
118 * #Client_session sends log-in request. #Server_session verifies that, then replies with log-in response
119 * (including important bits of generated naming) and enters PEER state. Once #Client_session (shortly) receives
120 * this response, it knows the session is ready to rock and also enters PEER state. Voila.
121 *
122 * Where does SHM-classic setup hook into this? Fairly naturally: The server side is in charge of
123 * setting up shared resources -- any required SHM arena(s) -- and once they exist, and only then,
124 * can it send log-in response. The client side *then* opens them, since by then they're guaranteed to exist.
125 * In the case of these SHM-classic `Pool_arena`s, the way that part works is simple: the server side names
126 * each pool according to a certain convention, so the client (knowing that the timing is right) opens
127 * each pool according to the same convention (and the information received in the log-in response is part of
128 * that).
129 *
130 * Thus:
131 * - The additional (SHM-classic-related steps) take place during log-in over session master channel, no earlier.
132 * - The server, first (before log-in response but *after* it knows everything else about the budding session --
133 * so in fact *right* before sending log-in response), ensures the 2 SHM arenas (per-session and per-app) required
134 * to be shared have been created. Then it sends the log-in response like usual.
135 * - The client, like usual, awaits log-in response; and once that arrives it completes the normal vanilla
136 * entry to PEER state. However the additional step it needs is to open the 2 SHM arenas guaranteed to
137 * exist, since log-in succeeded. Once it has done that, we're done.
138 *
139 * Now to discuss the specifics of each SHM scope. The simpler one is per-session (session_shm()).
140 * It is, naturally, created once for each session; so it has to be just before Server_session_impl
141 * `send()`s the log-in response. The naming is by the standard convention (see util::Shared_name doc header;
142 * see build_conventional_shared_name() which applies the convention). `Pool_arena` constructed in
143 * `CREATE_ONLY` mode (1) creates underlying named pool; and (2) retains an open handle to it.
144 * So we just do that which gives us the thing to return in session_shm() from then on.
145 *
146 * Slightly trickier is per-app (app_shm()). It is actually similar in terms of timing: it's done just then
147 * as well, because why not? However, for a given distinct Client_app, pool has to be *created* only the first time
148 * Session_server encounters an instance of the application described by that Client_app; after that it's already
149 * created, so no creation is necessary. Either way, keep the handle around to return in app_shm().
150 * However, by definition, a given per-app arena is to stay alive until the entire server (shm::classic::Session_server)
151 * is destroyed; resources (including transport::struc::Msg_out) residing in an app_shm()-returned
152 * arena stay around potentially over 2+ sessions and can be shared via any of them (as long as they apply to
153 * the same Client_app). Because of this more-than-session lifetime, the app_shm()-returned arenas are actually tracked
154 * in the parent shm::classic::Session_server; it uses one of the Session_server_impl customization points to
155 * create the per-Client_app arena as needed at just the right moment during the log-in process; then
156 * the shm::classic::Server_session_impl saves that pointer for quick return via `this->app_shm()`.
157 * The same arena can also be accessed from the central store via shm::classic::Session_server::app_shm()
158 * (which takes a `const Client_app&` and looks it up in an internal map).
159 *
160 * Other than that, per-app-scope arenas are created and remembered similarly to per-session-scope ones:
161 * create (centrally across server-sessions, on-demand) in `CREATE_ONLY` mode; retain (in each relevant
162 * server-session) a pointer for return via `this->app_shm()`. See shm::classic::Server_session_impl doc header
163 * and code inside shm::classic::Server_session_impl::async_accept_log_in().
164 *
165 * So then to recap:
166 * - Server: Open handle for session_shm() (create-only mode) and for app_shm() (same), just
167 * before send of log-in response, for each log-in. app_shm() is stored centrally in originating
168 * shm::classic::Session_server. Insertion point: *during* vanilla opening procedure.
169 * - Use Server_session_impl::async_accept_log_in()`'s `pre_rsp_setup_func()` arg for this purpose; in turn
170 * inside the body supplied for that guy shm::classic::Session_server uses the Session_server_impl ctor's
171 * `per_app_setup_func()` arg (customization point) for the centrally-performed app_shm() arena generation
172 * on-demand.
173 * - Client: Open handle for session_shm() and app_shm() (open-only mode for each), just after fully completing
174 * the vanilla entry to PEER state (past log-in response receipt).
175 * - shm::classic::Client_session just wraps `session::Client_session_impl::async_connect()` by tacking on
176 * the pool-openings (which are synchronous) onto its user-supplied on-done handler. Mechanically: Details left
177 * to shm::classic::Client_session docs.
178 *
179 * In `*this`, mechanically: the true implementation of the needed setup and accessors (explained above) is
180 * split between shm::classic::Session_mv (common with `Client_session_impl`) and
181 * shm::classic::Server_session_impl, with the vanilla core in super-class session::Server_session_impl.
182 * There's an app-scope snippet in shm::classic::Session_server.
183 *
184 * That's the key; then session::Server_session_mv adds movability around session::Server_session_impl; and lastly
185 * this class sub-classes *that* (by way of `Session_mv`) and completes the puzzle by pImpl-forwarding to the
186 * added (SHM-focused) API. (session::Server_session_mv only pImpl-forwards to the vanilla API.)
187 *
188 * The last and most boring piece of the puzzle are the pImpl-lite wrappers around the SHM-specific API
189 * that shm::classic::Server_session adds to super-class Server_session_mv: `session_shm()` and so on.
190 * Because this API is exactly identical for both shm::classic::Server_session and shm::classic::Client_session,
191 * to avoid torturous copy/pasting these wrappers are collected in shm::classic::Session_mv which we sub-class.
192 * So the `public` API is distributed as follows from bottom to top:
193 * - shm::classic::Server_session
194 * => shm::classic::Session_mv => session::Server_session_mv => session::Session_mv
195 * => shm::classic::Server_session_impl
196 * - shm::classic::Client_session
197 * == shm::classic::Session_mv => session::Client_session_mv => session::Session_mv
198 * => shm::classic::Client_session_impl
199 * - The bottom two in this case are not even inherited but one aliases to the other. This is because
200 * there's nothing added API-wise on the client side. On the server side, as you see below, an internal-use
201 * ctor is necessary to add, so we can't alias and must sub-class. It's close to an alias though.
202 *
203 * You'll notice, like, `Base::Base::Base::Impl` and `Base::Base::Base::Impl()` in certain code below;
204 * that's navigating the above hierarchy.
205 *
206 * @endinternal
207 *

212 * @tparam Mdt_payload
213 * Identical to session::Server_session.
214 */
