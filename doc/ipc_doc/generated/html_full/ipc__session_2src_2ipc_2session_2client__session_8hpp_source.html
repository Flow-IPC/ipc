29/**
30 * Implements Session concept on the Client_app end: a Session_mv that first achieves PEER state by connecting
31 * to an opposing Session_server_mv via Client_session_mv::async_connect(). See overview of Session hierarchy
32 * in namespace ipc::session doc header; then come back here if desired.
33 *
34 * It is unusual to use Client_session_mv template directly. If you do wish to set up a client-side session peer,
35 * and you do not require SHM support, then use #Client_session alias template. The client-specific API,
36 * particularly ctor and async_connect(), is in Client_session_mv and documented here. The side-agnostic API --
37 * active once PEER state is achieved -- is described by Session (concept) doc header and implemented concretely
38 * by Session_mv which is our `public`, non-`virtual` super-class.
39 *
40 * If you do wish to set up a client-side session peer, but you *do* require SHM support, then use
41 * shm::classic::Client_session or similar (for other SHM-provider(s)). However #Client_session
42 * (hence Client_session_mv to which it aliases) is its super-class, and until PEER state is reached its API
43 * remains the only relevant API to use. Once Client_session_mv::async_connect() puts `*this` into PEER
44 * state, super-class Session_mv (= Session concept) API continues to be relevant. Also in PEER state
45 * SHM-relevant additional API members (e.g., shm::classic::Session_mv::app_shm()`) become of interest.
46 *
47 * Summary hierarchy:
48 * - Session_mv (Session concept impl)
49 * - ^-- Client_session_mv (adds async_connect())
50 * - =alias= #Client_session (parameterizes by transport-configuring knobs = template params)
51 * - ^-- shm::classic::Session_mv
52 * (adds `session_shm()`...; suitable for transport::struc::Channel_base::Serialize_via_session_shm)
53 * - =alias= shm::classic::Client_session (parameterizes by aforementioned knobs = template params)
54 *
55 * We may refer to Client_session_mv as #Client_session below; particularly since it is likeliest used in that
56 * form by the user.
57 *
58 * ### How to use ###
59 * Per the Session concept a #Client_session is open/capable of operation when in PEER state only.
60 * A #Client_session object is always in one of 3 states:
61 * - NULL. In this state, the object is doing nothing; neither connected nor connecting.
62 * - CONNECTING. In this state the object is trying to enter PEER state.
63 * - PEER. At this stage it exactly implements the Session concept. See Session_mv. Reminder:
64 * It is not possible to exit PEER state.
65 *
66 * To get from NULL state to PEER state: Use Session_server on the other side. On this side construct
67 * a #Client_session (always in NULL state) and invoke async_connect() which will move to CONNECTING state immediately
68 * and, eventually on success, to PEER state (mission accomplished). If the connect fails it will go back to NULL
69 * state. Being moved-from makes a `*this` as-if default-cted; therefore also moves to NULL state.
70 *
71 * Once in PEER state #Client_session simply follows Session concept semantics. At this stage our super-class
72 * Session_mv implements that concept in particular. See either doc header (Session, Session_mv).
73 *
74 * A related reminder:
75 * - async_connect() or the opposing `async_accept()` or both may optionally specify that 1+ *init-channels*
76 * be opened before this and the opposing Session is in PEER state and emitted to user.
77 * This may be a sufficient replacement of, or complementary to, active-open/passive-open APIs available
78 * once the Session is in PEER state/emitted to user. If not sufficient:
79 * - Either side (being a Session_mv = Session) has open_channel(), and hence either side can accept passive-opens
80 * when the opposing side invokes open_channel().
81 * - However it is possible to configure either side to not accept passive-opens; meaning to be useful
82 * such a side shall be exclusively doing open_channel() (active-opens). One might conclude, then, that
83 * Server_session should be passive-opening, while #Client_session should be active-opening.
84 * That is *not* the case. In PEER state established on both sides, the 2 sides are identical in this
85 * respect. If both sides are configured to accept passive-opens, then there's no question.
86 * If one side is so configured, it can be either side. If both sides are so configured, the session is useless,
87 * as every open_channel() will fail (unless one has set up init-channels).
88 * - In #Client_session, one specifies the on-passive-open handler in ctor; to disable passive-opens use
89 * the ctor that omits that handler.
90 *
91 * ### Thread safety and handler invocation semantics ###
92 * These follow the Session concept policies. In particular the following prohibition extends to
93 * async_connect(): It is *not* allowed to invoke `*this` APIs (practically speaking async_connect() itself again)
94 * from within async_connect() completion handler.
95
96 * ### Error handling ###
97 * Once in PEER state, error handling follows the Session concept (= Session_mv concrete class) doc header.
98 * However, up to that point it is different and more akin to boost.asio semantics. Namely:
99 * - In NULL state: there are no errors.
100 * - In CONNECTING state -- while an async_connect() is outstanding -- an error may occur which moves `*this`
101 * back to NULL state and emits the error into the user handler given to async_connect().
102 * If dtor is invoked before async_connect() has a chance to complete then the handler is invoked
103 * with session::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER.
104 * (Subtlety: transport::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER is a different value;
105 * so just be careful when checking for it.)
106 *
107 * @internal
108 * ### Implementation design/rationale ###
109 * Session_mv is the highest (`public`) super-class and begins the pImpl-lite-for-movability technique.
110 * It pImpl-wraps the Session-concept impl methods/stuff. Client_session_mv merely repeats the same
111 * technique but on the API additions at its own level; in particular the `explicit` ctors and
112 * especially async_connect().
113 *
114 * The impl object is available via Session_mv::impl() (which is `protected`). What type does this object
115 * have though? Answer: `Client_session_impl_t`, our template param! This shall be the proper `..._impl`
116 * internal-use type that matches the level of API (`public` sub-class) the user chose: E.g.:
117 * - Chose #Client_session => impl = Client_session_impl.
118 * - Chose shm::classic::Client_session => impl = shm::classic::Client_session_impl.
119 *
120 * So basically each of the public-API types visible to the user adds the pImpl-wrapping of the methods
121 * of the `..._impl` type matching their name; and the public/movable-API hierarchy has a parallel
122 * internal/non-movable hierarchy, both using `public`/non-`virtual` inheritance.
123 *
124 * @see Client_session_impl doc header.
125 * @endinternal
126 *
127 * @see Session: implemented concept.
128 *
129 * @tparam Client_session_impl_t
130 * An implementation detail. Use one of the aliases prescribed near the top of this doc header to
131 * set this correctly.
132 */

210 /**
211 * To be invoked in NULL state only, and not as-if default-cted,
212 * it asynchronously attempts to connect to a opposing `Session_server`;
213 * and on success invokes the given completion handler with a falsy value indicating `*this` has entered PEER state.
214 * On failure invokes completion handler with a truthy values indicating `*this` has returned to NULL state.
215 * In the meantime `*this` is in CONNECTING state.
216 *
217 * If invoked outside of NULL state, or if it is as-if default-cted (i.e., default-cted or moved-from),
218 * this returns `false` and otherwise does nothing.
219 *
220 * In particular: Do not invoke async_connect() while one is already outstanding: `*this` must be in NULL state, not
221 * CONNECTING. Also note that `*this` (modulo moves) that has entered PEER state can never change state subsequently
222 * (even on transmission error); once a PEER, always a PEER.
223 *
224 * #Error_code generated and passed to `on_done_func()`:
225 * session::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER (destructor called, canceling all pending ops;
226 * spiritually identical to `boost::asio::error::operation_aborted`),
227 * interprocess-mutex-related errors (probably from boost.interprocess) w/r/t reading the CNS (PID file),
228 * file-related system errors w/r/t reading the CNS (PID file) (see Session_server doc header for background),
229 * error::Code::S_CLIENT_NAMESPACE_STORE_BAD_FORMAT (bad CNS contents),
230 * those emitted by transport::Native_socket_stream::async_connect(),
231 * those emitted by transport::struc::Channel::send(),
232 * those emitted by transport::struc::Channel via on-error handler (most likely
233 * transport::error::Code::S_RECEIVES_FINISHED_CANNOT_RECEIVE indicating graceful shutdown of opposing process
234 * coincidentally during log-in procedure, prematurely ending session while it was starting),
235 * error::Code::S_CLIENT_MASTER_LOG_IN_RESPONSE_BAD,
236 * error::Code::S_INVALID_ARGUMENT (other side expected other async_connect() overload with
237 * non-null `init_channels_by_srv_req` arg).
238 *
239 * ### How does it know "where" the Session_server is listening? ###
240 * While details are internal, abstractly speaking this is determined by the Server_app passed to the ctor.
241 * Internal detail (for general knowledge):
242 * - The server application maintains a *CNS* (Current Namespace Store, a/k/a PID) file.
243 * Only at most 1 server *instance* (a/k/a process) shall be active at a time. The CNS contents --
244 * which are simply the active server app process's PID -- determine "where" this `Client_session`
245 * will connect, and all other internally used cross-process resources
246 * (sockets, MQs, SHM pools...) are similarly segregated among different instances of the Server_app-described
247 * application.
248 * - The location of this file is determined by Server_app contents. Namely, Server_app::m_name, at least,
249 * is unique per Server_app in the universe, and most of the PID file's name comprises it; and
250 * its location comprises Server_app::m_kernel_persistent_run_dir_override and/or /var/run (or equivalent).
251 * - async_connect() reads the CNS (PID) file at that location, unique among `Server_app`s. Now it knows
252 * "where" to connect in order to open the session; so it won't accidentally connect to some old zombie
253 * instance of the Server_app (its PID wouldn't be in the CNS file any longer) or another application entirely
254 * (its PID file would be located elsewhere and never accessed by this async_connect()).
255 *
256 * ### What if the CNS (PID) file does not exist? What if server is inactive? ###
257 * Then async_connect() will quickly fail (file-not-found error). Shouldn't we supply some API to await its
258 * appearance? Answer: Probably not. Why? Answer: Consider the situation where the PID file is there, but
259 * it belongs to a now-exited instance of the Server_app; let's say it is currently suspended or restarting.
260 * async_connect() will fail then too: There is nothing listening (anymore) at the advertised PID. So
261 * it'll quickly fail (connection-refused error). So then we'd also need an API to await an active server's
262 * appearance whether or not the PID exists. That leads to the following to-do:
263 *
264 * @todo Consider adding an optional mode to async_connect() wherein it does not fail if the CNS (PID) file does
265 * not exist, or it does but the initial session-open connect op is refused; instead it detects these relatively
266 * common conditions (server not yet up and/or is restarting and/or is operationally suspended for now, etc.) as
267 * normal and waits until the condition is cleared. Without this mode, a typical user would probably do
268 * something like: oh, async_connect() failed; let's sleep 1 sec and try again (rinse/repeat). It's not
269 * awful, but we might as well make it easier and more responsive out of the box (optionally). Upon resolving this
270 * to-do please update the Manual section @ref session_setup accordingly.
271 *
272 * @todo Add a blocking version of `Client_session::async_connect()`; and/or such a version with a timeout. This
273 * would be most useful only after first executing the to-do just preceding this one in the source code comment/docs.
274 *
275 * @return `false` if and only if invoked outside of NULL state; or if `*this` is as-if default-cted.
276 *
277 * @tparam Task_err
278 * Handler type matching signature of `flow::async::Task_asio_err`.
279 * @param on_done_func
280 * `on_done_func(Error_code err_code)` is invoked from some unspecified thread, not the caller thread,
281 * indicating entrance from CONNECTING state to either NULL or PEER state.
282 * If interrupted by destructor the operation-aborted code is passed instead (see ~Client_session_mv()
283 * doc header).
284 */
285 template<typename Task_err>
286 bool async_connect(Task_err&& on_done_func);

288 /**
289 * Identical to the simpler async_connect() overload but offers added advanced capabilities: metadata exchange;
290 * initial-channel opening. The other overload is identical to
291 * `async_connect(mdt_builder(), nullptr, nullptr, nullptr, on_done_func)` and requires
292 * the opposing `async_accept()` to similarly not use the corresponding features.
293 *
294 * ### Client->server metadata exchange ###
295 * Client may wish to provide information in some way, but without a structured channel -- or any channel -- yet
296 * available, this provides an opportunity to do so in a structured way with a one-off message available at
297 * session open, together with the opposing Session object itself.
298 *
299 * Similarly to how it is optionally done on a per-channel-open basis (open_channel()): call `mdt = mdt_builder()`;
300 * fill out the resulting capnp structure `*mdt`; then pass `mdt` to async_connect(). Opposing `async_accept()` shall
301 * receive deserializable `Mdt_reader` together with the freshly-opened Session, though they may choose to ignore it.
302 *
303 * To omit using this feature, skip the "fill out the resulting capnp structure" step. You must still
304 * call mdt_builder() and pass-in the result. (The other overload does so.)
305 *
306 * ### Server->client metadata exchange ###
307 * This is the reverse of the above. Whatever the opposing server chose to supply as server->client metadata
308 * shall be deserializable at `*mdt_from_srv_or_null` once (and if) `on_done_func(Error_code())` (successful
309 * connect) fires. If `mdt_from_srv_or_null` is null, the srv->cli metadata shall be ignored.
310 *
311 * ### Init-channels by client request ###
312 * Once the session is open, open_channel() and the on-passive-open handler may be used to open channels at will.
313 * In many use cases, however, a certain number of channels is required immediately before work can really begin
314 * (and, frequently, no further channels are even needed). For convenience (to avoid asynchrony/boiler-plate)
315 * the init-channels feature will pre-open a requested # of channels making them available right away, together
316 * with the freshly-open session -- to both sides.
317 *
318 * The client may request 0 or more init-channels. They shall be opened and placed into
319 * `*init_channels_by_cli_req_pre_sized`; its `->size()` at entry to the method indicates the number of channels
320 * requested. If null, it is treated as if `->size() == 0`, meaning no init-channels-by-client-request are opened.
321 *
322 * ### Init-channels by server request ###
323 * This is the reverse of the above. The opposing side shall request 0 or more init-channels-by-server-request;
324 * that number of channels shall be opened; and they will be placed into `*init_channels_by_srv_req` which shall
325 * be `->resize()`d accordingly, once (and if) `on_done_func(Error_code())` (successful
326 * connect) fires.
327 *
328 * `init_channels_by_srv_req` being null is allowed, but only if the opposing server requests 0
329 * init-channels-by-server-request. Otherwise an error shall be emitted (see below).
330 *
331 * @tparam Task_err
332 * See other async_connect() overload.
333 * @param mdt
334 * See above.
335 * @param init_channels_by_cli_req_pre_sized
336 * See above: null or pointer to container of `Channel_obj` with `.size()` specifying how many channels
337 * this side is requesting to be opened on its behalf; each element will be move-assigned
338 * a PEER-state `Channel_obj` on success. Recommend simply ct-ing with `(n)` or `.resize(n)` which
339 * loads it with default-cted (NULL-state) objects to be replaced. null is treated same as
340 * `.empty()`.
341 * @param mdt_from_srv_or_null
342 * See above: null or pointer to `Reader` of metadata which shall be set for access on success.
343 * @param init_channels_by_srv_req
344 * See above: null or pointer to container of `Channel_obj` which shall be `.clear()`ed and replaced
345 * by a container of PEER-state `Channel_obj` on success the number being specified by the opposing
346 * (server) side. The number may be zero. null is allowed if and only if the number is zero;
347 * otherwise error::Code::S_INVALID_ARGUMENT is emitted.
348 * @param on_done_func
349 * See other async_connect() overload. Note the above target (pointer) args are touched
350 * only if falsy #Error_code is passed to this handler.
351 * @return See other async_connect() overload.
352 */
353 template<typename Task_err>
354 bool async_connect(const typename Base::Mdt_builder_ptr& mdt,
355 typename Base::Channels* init_channels_by_cli_req_pre_sized,
356 typename Base::Mdt_reader_ptr* mdt_from_srv_or_null,
357 typename Base::Channels* init_channels_by_srv_req,
358 Task_err&& on_done_func);
359

362 using Base::get_log_component;
