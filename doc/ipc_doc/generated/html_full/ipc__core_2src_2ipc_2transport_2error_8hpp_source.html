23/**
24 * Namespace containing the ipc::transport module's extension of boost.system error conventions, so that that API
25 * can return codes/messages from within its own new set of error codes/messages. Note that many errors
26 * ipc::transport might report are system errors and would not draw from this set of codes/messages but rather
27 * from `boost::asio::error` or `boost::system::errc` (possibly others). (If you're familiar with the boost.system
28 * framework, you'll know such mixing is normal -- in fact arguably one of its strengths.)
29 *
30 * @see ipc::transport::struc::error which covers errors from the *structured* layer (ipc::transport::struc).
31 * The present namespace applies to unstructured/lower-level traffic or to both that and structured/higher-level.
32 *
33 * See flow's `flow::net_flow::error` doc header which was used as the model for this and similar.
34 * As of this writing there is discussion there useful for someone new to boost.system error reporting.
35 *
36 * @internal
37 *
38 * @todo This file and error.cpp are perfectly reasonable and worth-it and standard boiler-plate; and
39 * it's not that lengthy really; but it'd be nice to stop having to copy/paste all the stuff outside of just the
40 * error codes and messages. Should add a feature to Flow that'll reduce the boiler-plate to just that; maybe
41 * some kind of base class or something. boost.system already makes it pretty easy, but even that can probably be
42 * "factored out" into Flow. Update: I (ygoldfel) took a look at it for another project, and so far no obvious
43 * de-boiler-plate ideas come to mind. Ideally the inputs are: (1) an `enum` with the codes, like error::Code;
44 * (2) an `int`-to-`string` message table function, for log messages; and (3) a brief `const char*` identifying the
45 * code set, for log messages. The rest is the boiler-plate, but all of it seems to either already be accepably
46 * brief, and where something isn't quite so, I can't think of any obvious way to factor it out.
47 * Of course a macro-based "meta-language" is always a possibility, as we did in `flow::log`, but in this case it
48 * doesn't seem worth it at all.
49 */

58/**
59 * All possible errors returned (via `Error_code` arguments) by ipc::transport functions/methods *outside of*
60 * system-triggered errors such as `boost::asio::error::connection_reset`.
61 * These values are convertible to #Error_code (a/k/a `boost::system::error_code`) and thus
62 * extend the set of errors that #Error_code can represent.
63 *
64 * @internal
65 *
66 * When you add a value to this `enum`, also add its description to
67 * error.cppâ€™s Category::message(). This description must be identical to the
68 * description in the /// comment below, or at least as close as possible. This mirrors Flow's convention.
69 *
70 * When you add a value to this `enum`, also add its symbolic representation to error.cpp's
71 * error.cpp's Category::code_symbol(). This string must be identical to the symbol, minus the `S_`;
72 * e.g., Code::S_INVALID_ARGUMENT => `"INVALID_ARGUMENT". This enables the consistent and human-friendly
73 * serialization `<<` and deserialization `>>` of a Code w/r/t standard streams.
74 *
75 * If, when adding a new revision of the code, you add a value to this `enum`, add it to the end, but ahead of
76 * Code::S_END_SENTINEL.
77 * If, when adding a new revision of the code, you deprecate a value in this `enum`, do not delete
78 * it from this `enum`. Instead mark it as deprecated here and then remove it from
79 * Ipc_transport_category::message().
80 *
81 * Errors that indicate apparent logic bugs (in other words, assertions that we were too afraid
82 * to write as actual `assert()`s) should be prefixed with `S_INTERNAL_ERROR_`, and their messages
83 * should also indicate "Internal error: ". This mirrors Flow's convention.
84 */

175/**
176 * Given a `Code` `enum` value, creates a lightweight #Error_code (a/k/a boost.system `error_code`)
177 * representing that error. This is needed to make the
178 * `boost::system::error_code::error_code<Code>()` template implementation work. Or, slightly more in English,
179 * it glues the (completely general) #Error_code to the (`ipc::transport`-specific) error code set
180 * ipc::transport::error::Code, so that one can implicitly covert from the latter to the former.
181 *
182 * @param err_code
183 * The `enum` value.
184 * @return A corresponding #Error_code.
185 */
186Error_code make_error_code(Code err_code);

188/**
189 * Deserializes a transport::error::Code from a standard input stream. Reads up to but not including the next
190 * non-alphanumeric-or-underscore character; the resulting string is then mapped to a log::Sev. If none is
191 * recognized, Code::S_END_SENTINEL is the result. The recognized values are:
192 * - "1", "2", ...: Corresponds to the `int` conversion of that Code.
193 * - Case-insensitive encoding of the non-S_-prefix part of the actual Code member; e.g.,
194 * "INVALID_ARGUMENT" (or "invalid_argument" or "Invalid_argument" or...) for Code::S_INVALID_ARGUMENT.
195 * This enables a few I/O things to work, including parsing from config file/command line via and conversion from
196 * `string` via `boost::lexical_cast`.
197 *
198 * @param is
199 * Stream from which to deserialize.
200 * @param val
201 * Value to set.
202 * @return `is`.
203 */
204std::istream& operator>>(std::istream& is, Code& val);

207/**
208 * Serializes a transport::error::Code to a standard output stream. The output string is compatible with the reverse
209 * `istream>>` operator.
210 *
211 * ### Rationale ###
212 * This shall print a representation that looks like the identifier in C++ code; e.g.,
213 * Code::S_INVALID_ARGUMENT => `"INVALID_ARGUMENT"`. Nevertheless, when printing an #Error_code storing
214 * a Code, continue to do the standard thing: output the #Error_code (which will print the category name, in our
215 * case, something like `"ipc/transport"`; and the numeric `enum` value) itself plus its `.message()`
216 * (which will print a natural-language message). Naturally that technique will work with any #Error_code, including
217 * but *not* limited to one storing a Code.
218 *
219 * So why does the present `operator<<` even exist? Answer: Primarily it's in order to enable a consistent, yet
220 * human-friendly, machinery of serializing/deserializing `Code`s. No such machinery exists for general #Error_code,
221 * and specifying an #Error_code in an input stream would probably invovle (at least) providing a numeric value; but
222 * we can still provide a symbolic [de]serialization for specifically transport::error::Code. Why did we want that?
223 * Answer: The original use case was in testing scenarios, to easily specify an expected outcome of an ipc::transport
224 * API call symbolically instead of resorting to numeric `enum` values (it's nicer to say "expect INVALID_ARGUMENT"
225 * rather than "expect 13" -- though incidentally the latter also works).
226 *
227 * @param os
228 * Stream to which to serialize.
229 * @param val
230 * Value to serialize.
231 * @return `os`.
232 */
233std::ostream& operator<<(std::ostream& os, Code val);
