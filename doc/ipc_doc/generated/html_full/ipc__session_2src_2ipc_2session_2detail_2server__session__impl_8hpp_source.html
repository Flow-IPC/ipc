30/**
31 * Internal, non-movable pImpl-lite implementation of Server_session_mv class template.
32 * In and of itself it would have been directly and publicly usable; however Server_session_mv adds move semantics.
33 *
34 * @see All discussion of the public (and `protected`, to be exposed `public`ly by Server_session_dtl) API is in
35 * Server_session_mv doc header; that class template forwards to this
36 * one. All discussion of pImpl-lite-related notions is also there. See that doc header first please. Then come
37 * back here.
38 *
39 * Impl design
40 * -----------
41 * Generally, I (ygoldfel) recommend looking through and understanding Client_session_impl first. It is similar but
42 * somewhat more complex, and once it is understood, you'll be able to understand Server_session_impl easily. Plus
43 * its logic is complementary to a `*this`, as the two directly interoperate via a couple of IPC forms. The below
44 * discussion does assume one has read/understood Client_session_impl doc header.
45 *
46 * Thread U represents all threads other than thread W: since the relevant methods are to be called by the user
47 * sans concurrency, those threads as a collection can be thought of as one thread.
48 *
49 * Thread W is the async worker thread where most work is done; this helps us entirely avoid mutexes.
50 * async_accept_log_in() is called exactly once, internally by Session_server; and subsequently
51 * open_channel() is fairly infrequently called. So it's not necessary to avoid
52 * latency by doing work concurrently in threads U and W. So we keep it simple by posting most stuff onto thread W.
53 *
54 * The state machine of Server_session_impl is simpler than Client_session_impl. The asymmetry is, firstly, due to the
55 * client-server setup: Session_server is the one, on the server end, that awaits the initial
56 * transport::Native_socket_stream connection. Once a PEER-state `Native_socket_stream` is ready, only then
57 * does Session_server create a Server_session_dtl (which is a `public` facade to Server_session_mv, which
58 * is a movable `unique_ptr` wrapper around this Server_session_impl). So `*this`, unlike a Client_session_impl,
59 * does not need to connect a socket. Next, Server_session_impl just needs to complete one async step, which is
60 * in practice immediately (and, by contract, only once) invoked: async_accept_log_in(). At this stage user does
61 * not have the Server_session_mv yet, so we're still in internal-code land. (By contrast Client_session is
62 * created directly by the user and is immediately publicly available as a stand-alone object.) async_accept_log_in()
63 * immediately upgrades the `Native_socket_stream` to a transport::Channel and then immediately upgrades that
64 * to a transport::struc::Channel, saving it to #m_master_channel (all synchronously). Lastly is the actual async work:
65 * awaiting the log-in request from Client_session. Once that is received, we respond via #m_master_channel,
66 * either with log-in success or failure. Assuming success, we invoke the (internal) async_accept_log_in()
67 * handler; on failure same thing but with a failure #Error_code.
68 *
69 * ### Sidebar: Init-channels ###
70 * Update (init-channels feature has been added): The preceding paragraph holds fully if and only if:
71 * - client (in its log-in request message, `LogInReq`) requested 0 init-channels to be opened on its behalf; and
72 * - server (via the user args from Session_server::async_accept() forwarded to async_accept_log_in()) requested
73 * 0 init-channels to be opened on local user's behalf.
74 *
75 * However, if the sum of these 2 numbers (as determined upon successful receipt on `LogInReq`, and assuming
76 * everything else was also successful, up-to-and-including sending successful `LogInRsp`) is N=1+, then before
77 * async_accept_log_in() can reach almost-PEER state (and fire completion handler), there is 1 more async-step:
78 * - (send `LogInRsp` as explained);
79 * - (prep) synchronously prepare the N=1+ init-channels (exactly as would occur
80 * if user invoked open_channel() N times in PEER state);
81 * - (prep) sync-send N `OpenChannelToClientReq` messages with names/handles for each of the N init-channels
82 * (exactly as would occur on N open_channel() calls in PEER state);
83 * - async: await a *single* `OpenChannelToClientRsp` response, indicating the last of the N channels has been
84 * accepted, and client is now in PEER state and ready for passive-opens (i.e., we can enter PEER-state
85 * which enabled open_channel() by the local user).
86 *
87 * Why this last async step? Couldn't we just do the `send()`s and go to almost-PEER state immediately?
88 * Almost: but if we did that, then if the user immediately issued init_handlers() and an open_channel(),
89 * then the opposing Client_session_impl might not, technically, be quick enough to have moved to its own PEER state;
90 * then its `OpenChannelToClientReq` handler from the pre-PEER state (CONNECTING state) might be triggered;
91 * this would cause... nothing good. However the ACK from Client_session_impl is sent only just before
92 * entering PEER state, at which point it is safe for us to issue open_channel()s.
93 *
94 * ### Back to general impl discussion ###
95 * Our async_accept_log_in() is analogous/complementary to Client_session_impl::async_connect(), but the diff is that
96 * the user only gets access to a `*this` upon success of this async op; and never gets access upon failure
97 * at all. So that's another source of asymmetry versus Client_session_impl. Thus:
98 * - No need to guard against async_accept_log_in() call while the log-in is outstanding.
99 * - (As of *this* writing Client_session_impl only uses async_connect() internally and does not expose it,
100 * so it's not an issue there either; but that would change, if it became network-enabled at some level.)
101 * - No need to guard against other API calls, notably open_channel(), before (almost-)PEER state is achieved.
102 * (User only gets access to `*this`, once `*this` is in (almost-)PEER state.)
103 *
104 * Therefore there's no need to track an `m_state` the way Client_session_impl does. If #m_log_in_on_done_func
105 * is non-empty, then async_accept_log_in() is outstanding (analogous to Client_session_impl CONNECTING state).
106 * Otherwise see the "almost-PEER" section just below.
107 *
108 * ### Almost-PEER state impl ###
109 * There is a minor wrinkle in Server_session_mv (and thus Server_session_impl). Once the user obtains
110 * the Server_session_mv, it is in the so-called *almost-PEER* state. That is because it needs 1-2 more pieces of
111 * info to be set before it is formally in PEER state as specified by Session concept. Namely its
112 * on-error handler must be set (mandatory), as must its on-passive-open handler (optional; in that leaving it
113 * unspecified is allowed and means passive-opens are disabled on this side). To do so the *user*
114 * shall call init_handlers(). Until this is done, open_channel() and other public PEER-state APIs shall
115 * no-op and return sentinel/`false`.
116 *
117 * There is a subtlety in how this are-handlers-set check is performed. The crux of it is that #m_master_channel
118 * can report an error at any time starting with async_accept_log_in() (where it is first created), and what to
119 * do is quite different in each case:
120 * - Before public availability to user (async log-in ongoing).
121 * - Contingency: report failure of async_accept_log_in(), via its on-done handler.
122 * - After that but in almost-PEER state (on-error handler not yet set by user).
123 * - Contingency: well, we *want* to report it via on-error handler... but it's not set yet.
124 * So memorize it in special member #m_pre_init_err_code. Then in init_handlers() immediately fire off
125 * the freshly-available on-error handler with that #Error_code.
126 * - In PEER state (the handler has been set).
127 * - Contingency: yay, forward the #m_master_channel-reported #Error_code to the on-error
128 * handler.
129 *
130 * Because the #m_master_channel error handler is relevant across all of the above, and it is executed in thread W
131 * for reasons explained above (avoiding mutexes, etc.), to avoid concurrency difficulties the are-handlers-set
132 * check, and the assignment of the handlers, is also done in thread W. Hence, like most code in Server_session_impl,
133 * init_handlers() posts all its work onto thread W. All public APIs, even simple ones like session_token(),
134 * do the same, and that thread-W work always begins by ensuring handlers_are_set().
135 *
136 * Once PEER state is achieved, to the user `*this` is (just like Client_session_impl) an impl of Session concept.
137 * Internally it's somewhat different.
138 *
139 * ### PEER state impl ###
140 * Here our algorithm is complementary to the PEER state algorithm in Client_session_impl. Namely we expect
141 * passive-opens via an appropriate transport::struc::Channel::expect_msgs(); and we allow active-opens via
142 * open_channel(). The details of how these work is best understood just by reading that code inline. All in
143 * all, publicly it's just like Client_session_impl; but internally it is its complement and in fact has
144 * different (complementary) responsibilities. Most notably, regarding active-opens and passive-opens:
145 * - Server_session_impl is the one always internally responsible for acquiring the resources needed to
146 * establish the channel. Namely:
147 * - If #S_MQS_ENABLED: it *creates* the underlying MQs. This involves choosing a distinct-enough name for
148 * each of the 2 MQs (one in each direction, client->server, server->client). These names shall be
149 * sent-over, via #m_master_channel, to the Client_session_impl.
150 * - If #S_SOCKET_STREAM_ENABLED: it *creates* the pre-connected socket-pair, 1/2 of which (a `Native_handle`)
151 * shall be sent-over, via #m_master_channel, to the Client_session_impl.
152 * - Hence:
153 * - An active-open (open_channel()) sends over the references/handles to the above resource(s) in the
154 * open-channel-to-client request out-message via #m_master_channel. It then ensures the client
155 * replies with an OK (or failure) response in-message. open_channel() thus succeeds or fails.
156 * - A passive-open (on_master_channel_open_channel_req()) involves receiving an
157 * open-channel-to-server request in-message via #m_master_channel. It then acquires the above resource(s)
158 * and sends over the refs/handles to said resource(s) in our OK response to that request and invokes
159 * the on-passive-open handler locally.
160 *
161 * Client_session_impl active-opens and passive-opens act in a complementary fashion, never acquiring these resources
162 * but knowing what to expect.
163 *
164 * Otherwise open_channel() is set up similarly to Client_session_impl::open_channel(), namely the
165 * non-blocking/synchronous facade with async-with-timeout internals. See Client_session_impl doc header for
166 * brief discussion.

716TEMPLATE_SRV_SESSION_IMPL
717void CLASS_SRV_SESSION_IMPL::dtor_async_worker_stop()
718{
719 using flow::async::Synchronicity;
720 using boost::promise;
721
722 // We are in thread U (from either our own dtor or subclass dtor).
723
724 assert((!m_async_worker.task_engine()->stopped()) && "This shall be called at most once.");
725
726 /* The following is quite similar to Client_session_impl::dtor_async_worker_stop(), so keeping comments
727 * light here (@todo code reuse?). */
728
729 FLOW_LOG_INFO("Server session [" << *this << "]: Shutting down. Worker thread will be joined, but first "
730 "we perform session master channel end-sending op (flush outgoing data) until completion. "
731 "This is generally recommended at EOL of any struc::Channel, on error or otherwise. "
732 "This is technically blocking but unlikely to take long. Also a Graceful_finisher phase may "
733 "precede the above steps (it will log if so).");
734
735 if constexpr(S_GRACEFUL_FINISH_REQUIRED)
736 {
737 typename Base::Graceful_finisher* graceful_finisher_or_null = {};
738 m_async_worker.post([&]() // Have to access m_graceful_finisher (the optional<>) in thread W only.
739 {
740 // We are in thread W.
741 if (m_graceful_finisher)
742 {
743 graceful_finisher_or_null = &(m_graceful_finisher.value());
744 }
745 }, Synchronicity::S_ASYNC_AND_AWAIT_CONCURRENT_COMPLETION); // m_async_worker.post()
746
747 /* If graceful_finisher_or_null is not null, then we're definitely in PEER state, and this is irreversible;
748 * so it's cool to ->on_dtor_start() on it. It'll block as needed, etc.
749 *
750 * If it *is* null:
751 * - If between the assignment above to null and the next statement, m_graceful_finisher did not become
752 * non-empty, then there's no controversy: It's still not in PEER state; and there's no reason to
753 * block dtor still. So no need to do anything, and that's correct. Can just continue.
754 * - If however in the last few microseconds (or w/e) indeed m_graceful_finisher
755 * has become non-empty, then now *this is in PEER state. ...NOPE! The only way that can happen
756 * for us -- unlike Client_session_impl as of this writing -- is via init_handlers(), which the user
757 * themselves must execute. So actually graceful_finisher_or_null remains accurate from up above
758 * to the next statement. */
759 if (graceful_finisher_or_null)
760 {
761 graceful_finisher_or_null->on_dtor_start();
762 // OK, now safe (or as safe as we can guarantee) to blow *this's various parts away.
763 }
764 } // if constexpr(S_GRACEFUL_FINISH_REQUIRED)
765
766 m_async_worker.post([&]()
767 {
768 // We are in thread W.
769
770 if (!m_master_channel)
771 {
772 return; // Destroyed at early juncture (as of this writing). Whatever; cool; nothing to flush anyway.
773 }
774 // else
775
776 promise<void> done_promise;
777 m_master_channel->async_end_sending([&](const Error_code&)
778 {
779 // We are in thread Wc (unspecified, really struc::Channel async callback thread).
780 FLOW_LOG_TRACE("Server session [" << *this << "]: Shutdown: master channel outgoing-direction flush finished.");
781 done_promise.set_value();
782 }); // Don't care if returned false and did nothing: cool then; did our best.
783 // Back here in thread W:
784 done_promise.get_future().wait();
785 }, Synchronicity::S_ASYNC_AND_AWAIT_CONCURRENT_COMPLETION); // m_async_worker.post()
786 // Back here in thread U: done; yay.
787
788 m_async_worker.stop();
789 // Thread W has been joined.
790} // Server_session_impl::dtor_async_worker_stop()

792TEMPLATE_SRV_SESSION_IMPL
793CLASS_SRV_SESSION_IMPL::~Server_session_impl()
794{
795 using flow::async::Single_thread_task_loop;
796 using flow::async::reset_thread_pinning;
797 using flow::util::ostream_op_string;
798
799 // We are in thread U. By contract in doc header, they must not call us from a completion handler (thread W).
800
801 /* The following is quite similar to ~Client_session_impl(), so keeping comments light here (@todo code reuse?).
802 * This is simpler after the dtor_async_worker_stop() though; async_accept_log_in() is *not* a public API;
803 * so things like calling it more than 1x, let alone while an async-accept-log-in is in progress, are simply
804 * not allowed and treated as undefined behavior. Hence there's no m_mutex/m_state about which to worry. */
805
806 if (!m_async_worker.task_engine()->stopped())
807 {
808 dtor_async_worker_stop();
809 }
810
811 if (!m_log_in_on_done_func.empty())
812 {
813 FLOW_LOG_INFO("Server session [" << *this << "]: Continuing shutdown. Next we will run user async-accept-log-in "
814 "handler from some other thread. In this user thread we will await its completion and then return.");
815
816 Single_thread_task_loop one_thread(get_logger(), ostream_op_string("SvSDeinit", int(S_SHM_TYPE), '-', this));
817 one_thread.start([&]()
818 {
819 reset_thread_pinning(get_logger()); // Don't inherit any strange core-affinity. Float free.
820
821 FLOW_LOG_INFO("Server session [" << *this << "]: In transient finisher thread: Shall run pending handler.");
822
823 m_log_in_on_done_func(error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER);
824 FLOW_LOG_TRACE("User async-accept-log-in handler finished.");
825 }); // one_thread.start()
826 } // if (!m_log_in_on_done_func.empty())
827 // Here thread exits/joins synchronously. Back in thread U:
828
829 // Last thing! See sub_class_set_deinit_func() doc header which will eventually lead you to a rationale comment.
830 if (!m_deinit_func_or_empty.empty())
831 {
832 FLOW_LOG_INFO("Server session [" << *this << "]: Continuing shutdown. A sub-class desires final de-init "
833 "work, once everything else is stopped (might be persistent resource cleanup); invoking "
834 "synchronously.");
835 m_deinit_func_or_empty();
836 FLOW_LOG_TRACE("De-init work finished.");
837 }

888 (flow::async::Task_asio_err&& on_err_func_arg,
889 On_passive_open_channel_func&& on_passive_open_channel_func_or_empty_arg)
890{
891 // We are in thread W.
892
893 if (handlers_are_set())
894 {
895 FLOW_LOG_WARNING("Server session [" << *this << "]: User trying to proceed to PEER state by setting handlers; "
896 "but we are already in that state (handlers are set). Ignoring.");
897 return false;
898 }
899 // else
900
901 FLOW_LOG_INFO("Server session [" << *this << "]: User trying to proceed to PEER state by setting handlers: "
902 "Success. Will emit any cached master-channel error ([" << m_pre_init_err_code << "] "
903 "[" << m_pre_init_err_code.message() << "]).");

908
909 if (m_pre_init_err_code)
910 {
911 assert((!Base::hosed()) && "We only just entered PEER state.");
912
913 /* Post this asynchronously onto thread W, even though we are in W already, so that they don't delay our,
914 * and therefore thread U caller's, return. */
915 m_async_worker.post([this]()
916 {
917 // We are in thread W.
918
919 if (Base::hosed())
920 {
921 FLOW_LOG_INFO("Server session [" << *this << "]: We were about to fire cached error observed earlier "
922 "during the almost-PEER phase; but somehow *yet another* error condition hosed us during "
923 "the async-gap between that recent decision and this task executing in worker thread. "
924 "It is hard to imagine what it could be, but it is not known to be a bug or problem. "
925 "It is interesting enough for INFO log level though. Recommend investigation.");
926 }
927 else
928 {
929 Base::hose(m_pre_init_err_code);
930 }
931 });
932 return true;
933 }
934 // else if (!m_pre_init_err_code):
935
936 m_master_channel->expect_msgs(Master_structured_channel::Msg_which_in::OPEN_CHANNEL_TO_SERVER_REQ,
937 [this](const typename Master_structured_channel::Msg_in_ptr& open_channel_req_cref)

940 m_async_worker.post([this, open_channel_req = open_channel_req_cref]() mutable

942 // We are in thread W.
943 on_master_channel_open_channel_req(std::move(open_channel_req));
944 });
945 });

948 if constexpr(S_GRACEFUL_FINISH_REQUIRED)
949 {
950 m_graceful_finisher.emplace(get_logger(), this, &m_async_worker, m_master_channel.get());
951 }
952
953 return true;
954} // Server_session_impl::init_handlers_impl()

996TEMPLATE_SRV_SESSION_IMPL
997typename CLASS_SRV_SESSION_IMPL::Mdt_builder_ptr CLASS_SRV_SESSION_IMPL::mdt_builder()
998{
999 using util::Native_handle;
1000 using flow::async::Synchronicity;
1001 namespace asio_local = transport::asio_local_stream_socket;
1002
1003 // We are in thread U.
1004
1005 Channel_obj opened_channel;
1006 Native_handle remote_hndl_or_null;
1007 Shared_name mq_name_c2s_or_none;
1008 Shared_name mq_name_s2c_or_none;
1009 bool resources_acquired_ok = false;
1010
1011 typename Master_structured_channel::Msg_out open_channel_req_msg;
1012 bool open_channel_req_msg_inited = false;
1013 m_async_worker.post([&]()
1014 {
1015 // We are in thread W: need to be to access m_master_channel and to call create_channel_and_resources().
1016
1017 if (!handlers_are_set())
1018 {
1019 FLOW_LOG_WARNING("Server session [" << *this << "]: Channel open active request: Invoked before PEER state. "
1020 "Ignoring.");
1021 assert((!open_channel_req_msg_inited) && "Should have remained false as initialized.");
1022 assert((remote_hndl_or_null.null()) && "Should have remained .null() as initialized.");
1023 return;
1024 }
1025 // else
1026
1027 resources_acquired_ok
1028 = (!Base::hosed())
1029 && create_channel_and_resources(&mq_name_c2s_or_none, &mq_name_s2c_or_none, &remote_hndl_or_null,
1030 &opened_channel, true); // true => active-open, not passive-open.
1031
1032 assert(resources_acquired_ok || remote_hndl_or_null.null());
1033 open_channel_req_msg = m_master_channel->create_msg(std::move(remote_hndl_or_null));
1034 open_channel_req_msg_inited = true;
1035 // (Might be sending them this Native_handle too. ---^)
1036 }, Synchronicity::S_ASYNC_AND_AWAIT_CONCURRENT_COMPLETION);
1037
1038 // The rest we can do back in thread U again.
1039
1040 if (!open_channel_req_msg_inited)
1041 {
1042 // As advertised: Previous error (reported via on-error handler) => return null.
1043 return Mdt_builder_ptr();
1044 }
1045 // else
1046
1047 auto open_channel_root = open_channel_req_msg.body_root()->initOpenChannelToClientReq();
1048 if (resources_acquired_ok)
1049 {
1050 open_channel_root.setClientToServerMqAbsNameOrEmpty(mq_name_c2s_or_none.str());
1051 open_channel_root.setServerToClientMqAbsNameOrEmpty(mq_name_s2c_or_none.str());
1052 }
1053 /* else if (!resources_acquired_ok):
1054 * Resource unavailable. Now, we know this already, even though we haven't even sent any request to the
1055 * client -- because we *are* the server and thus immediately can and must acquire the channel resources.
1056 * Our contract is to report such a problem via the later open_channel() though. So we save this fact
1057 * and will report it quickly from open_channel(), right at its start.
1058 *
1059 * By saving a default-cted Channel_obj in this event, we indicate this fact.
1060 *
1061 * Rationale: Well, we promised this behavior. But why? We could report it now. Answer: It would be a clumsy
1062 * API contract. Either we must return `false`, in which case they cannot easily distinguish between
1063 * "prior incoming-direction error => false, just count on the on-error handler" and
1064 * "resource unavailable error => false, your decision whether to blow everything up or what.";
1065 * or we'd need to add an Error_code* out-arg, which means complicating the Session concept API, which means
1066 * changing Server_session too. Anyway, it is weird, from user's PoV, to even deal with the idea that
1067 * in a Client_session the metadata-builder-returning method somehow can already fail -- they have no reason
1068 * to think resource acquisition occurs at that stage. So at the expense of a short delay we just delegate
1069 * new-error reporting to open_channel() instead of making it an odd 2-step thing.
1070 *
1071 * That said, open_channel() contract still has to specifically note that the particular resource-unavailable
1072 * error -- not fatal to the Session -- is a possibility, unlike other `Error_code`s which are fatal.
1073 * However (1) that's open_channel()'s problem; and (2) it is manageable (and is mandated by Session concept). */
1074
1075 Open_channel_req_ptr open_channel_req_ptr(new Open_channel_req
1076 { open_channel_root.initMetadata(),
1077 std::move(open_channel_req_msg),
1078 // If this is empty: indicates resource-unavail eventuality.
1079 resources_acquired_ok
1080 ? std::move(opened_channel)
1081 : Channel_obj() });
1082 return Mdt_builder_ptr(std::move(open_channel_req_ptr), &open_channel_req_ptr->m_mdt_builder);
1083} // Server_session_impl::mdt_builder()

1102 "null metadata-builder passed-in; either user error passing in null; or more likely "
1103 "that routine failed and returned null, meaning an incoming-direction error was or is being "
1104 "emitted to the on-error handler. Emitting code-less failure in this synchronous path.");

1106 }

1108

1124

1126 "sufficiently quick to be considered non-blocking, request-response exchange initiating; timeout set "

1138

1146 sync_error_but_do_not_emit = true;
1147 return;
1148 }
1149 // else if (!Base::hosed()): We can attempt sync_request() (which could hose us itself).
1150
1151 auto& open_channel_req = *(reinterpret_cast<Open_channel_req*>(mdt.get()));
1152
1153 /* Now simply issue the open-channel request prepared already via `mdt` and synchronously
1154 * await response, or error, or timeout. struc::Channel::sync_request() is tailor-made for this. */

1161 "(due to incoming-direction error which shall hose session around this time). No-op.");
1162 sync_error_but_do_not_emit = true;
1163 return;
1164 }
1165 // else if (*err_code || open_channel_rsp)
1166
1167 if (*err_code) // && !open_channel_rsp)
1168 {
1169 assert((!open_channel_rsp) && "struc::Channel::sync_request() must emit no Error_code if it gets response.");
1170
1171 if (*err_code == transport::error::Code::S_TIMEOUT)
1172 {
1173 FLOW_LOG_WARNING("Server session [" << *this << "]: Channel open active request: It timed out. It may "
1174 "complete in the background still, but when it does any result shall be ignored. "
1175 "Emitting (non-session-hosing) error.");
1176
1177 *err_code = error::Code::S_SESSION_OPEN_CHANNEL_ACTIVE_TIMEOUT;
1178 // Non-session hosing error will be synchronously emitted; *this will continue (as promised).
1179 return;
1180 }
1181 // else if (*err_code truthy but not timeout): Session got hosed.
1182
1183 /* As promised, open_channel() hosing the session shall be reported asynchronously like all errors;
1184 * while we will merely return false. */
1185 FLOW_LOG_WARNING("Server session [" << *this << "]: Channel open request failed at the sync-request stage, "
1186 "meaning either the nb-send failed, or in-error detected on channel while waiting for "
1187 "response (new error: [" << *err_code << "] [" << err_code->message() << "]). Hosing "
1188 "session (emit to session on-error handler); will synchronously return failure to "
1189 "open channel.");
1190 Base::hose(*err_code);
1191 sync_error_but_do_not_emit = true; // err_code is ignored.
1192 return;
1193 }
1194 // else if (!*err_code): sync_request() succeeded all the way.
1195 assert(open_channel_rsp);
1196
1197 FLOW_LOG_TRACE("Server session [" << *this << "]: Channel open request: Request message sent to client "
1198 "sans error, and response received in time.");
1199
1200 const auto root = open_channel_rsp->body_root().getOpenChannelToClientRsp();
1201 switch (root.getOpenChannelResult())
1202 {
1203 case OpenChannelResult::ACCEPTED:
1204 assert(!*err_code);
1205 break;
1206 case OpenChannelResult::REJECTED_PASSIVE_OPEN:
1207 *err_code = error::Code::S_SESSION_OPEN_CHANNEL_REMOTE_PEER_REJECTED_PASSIVE_OPEN;
1208 break;
1209 case OpenChannelResult::REJECTED_RESOURCE_UNAVAILABLE:
1210 assert(false && "Only server grabs channel resources; not possible in server-active-open path.");
1211 break;
1212 case OpenChannelResult::END_SENTINEL:
1213 assert(false);
1214 } // Compiler should catch missing enum value.
1215
1216 if (*err_code)
1217 {
1218 FLOW_LOG_WARNING("Server session [" << *this << "]: Channel open active request: Response received but "
1219 "indicates remote peer (client) refused; emitting "
1220 "[" << *err_code << "] [" << err_code->message() << "] with explanation from remote "
1221 "peer. This will not hose session on our end.");
1222 // Note: all of the above are advertised as non-session-hosing.
1223 return;
1224 }
1225 // else: Cool! We already have our side of the channel too.
1226 assert(open_channel_req.m_opened_channel.initialized());
1227
1228 *target_channel = std::move(open_channel_req.m_opened_channel);
1229
1230 FLOW_LOG_INFO("Server session [" << *this << "]: Channel open active request: Succeeded in time yielding "
1231 "new channel [" << *target_channel << "].");
1232 }, Synchronicity::S_ASYNC_AND_AWAIT_CONCURRENT_COMPLETION); // m_async_worker.post()
1233
1234 // It all logged sufficiently. Just return the results.
1235
1236 if (sync_error_but_do_not_emit)
1237 {
1238 err_code->clear();
1239 return false;
1240 }
1241 // else
1242
1243 return true; // *err_code may be truthy or falsy.
1244} // Server_session_impl::open_channel()

1508 * For now let's assume a typical environment.

1514 *
1515 * To be clear: the transport:: MQ stuff above the lowest (Persistent_mq_handle) level will seamlessly handle
1516 * any would-block situation due to reaching this limit. In theory it could affect perf. */

1519 /* We must also decide the max message of each size. If SHM-backing is disabled, then there's no question:
1520 * we have to use Base::S_MQS_MAX_MSG_SZ which (see its doc header) respects both the need to be as large
1521 * as possible to support the widest variety of user payloads; and the (POSIX-MQ-relevant) system limit(s) as
1522 * to the message size. If SHM-backing is enabled, it is tempting to simply use the same value... why not,
1523 * right? Good enough for the heap-backed case; why not the SHM-backed case? At worst it only wastes some
1524 * RAM, since each message transmitted over MQ contains merely a tiny SHM handle, right? Yes and no:
1525 * For bipc MQs, yes; but POSIX MQs impose yet another limit: A given process trying to maintain a *total*
1526 * of X bytes of MQs at a given time (the "total" roughly meaning the sum of `MAX_N_MSG x max_msg_sz` for
1527 * all then-opened MQs) will yield too-many-files (Linux errno=EMFILES) once X is exceeded. A typical value
1528 * for this is 819,200 -- or up to only *ten* MAX_N_MSxMQS_MAX_MSG_SZ such MQs opened simultaneously.
1529 * This is in practice easily exceeded. On the other hand, suppose we instead use S_SHM_MAX_HNDL_SZ
1530 * instead of MQS_MAX_MSG_SZ -- which is sufficient, since we'll only be sending SHM handles, not the actual
1531 * data. As of this writing it's 64; so that's 8,192/64=128 *times* the number of MQs creatable this way;
1532 * errno=EMFILES occurs not after 10 MQs now but after 1,280 such MQs. Much more reasonable.
1533 * Trust me (ygoldfel): the ~10-MQ limit can be quite annoying. */
1534 size_t max_msg_sz;
1535 if constexpr(S_SHM_ENABLED)
1536 {
1537 max_msg_sz = S_SHM_MAX_HNDL_SZ;
1538 }
1539 else
1540 {
1541 max_msg_sz = Base::S_MQS_MAX_MSG_SZ;
1542 }
1543
1544 ++m_last_channel_mq_id;

1546 const auto mq_name_func = [&](bool c2s) -> Shared_name
1547 {
1548 // Session scope.
1549 auto mq_name = build_conventional_shared_name(Persistent_mq_handle_from_cfg::S_RESOURCE_TYPE_ID,
1550 Shared_name::ct(Base::m_srv_app_ref.m_name),
1551 Base::srv_namespace(),
1552 Shared_name::ct(Base::cli_app_ptr()->m_name),
1553 Base::cli_namespace());
1554 // That was the standard stuff. Now the us-specific stuff:
1555 mq_name /= c2s ? "c2s" : "s2c";
1556 mq_name /= to_string(m_last_channel_mq_id);
1557 return mq_name;
1558 }; // mq_name_func =
1559
1560 // These ctors should log in any case.
1561
1562 const auto perms = util::shared_resource_permissions(Base::m_srv_app_ref.m_permissions_level_for_client_apps);
1563
1564 *mq_c2s = Persistent_mq_handle_from_cfg(get_logger(), mq_name_func(true), util::CREATE_ONLY,
1565 MAX_N_MSG, max_msg_sz, perms, err_code);
1566 if (*err_code)
1567 {
1568 --m_last_channel_mq_id; // Might as well.
1569 return false;
1570 }
1571 // else
1572 *mq_s2c = Persistent_mq_handle_from_cfg(get_logger(), mq_name_func(false), util::CREATE_ONLY,
1573 MAX_N_MSG, max_msg_sz, perms, err_code);
1574 if (*err_code)
1575 {
1576 --m_last_channel_mq_id; // Might as well.
1577 const auto name = mq_c2s->absolute_name();
1578 *mq_c2s = Persistent_mq_handle_from_cfg(); // Close the first guy; won't be used.
1579 Error_code ignored; // Don't forget to remove underlying MQ we'd successfully created.
1580 Persistent_mq_handle_from_cfg::remove_persistent(get_logger(), name, &ignored);
1581 return false;
1582 }
1583 // else
1584 *mq_name_c2s = mq_c2s->absolute_name();
1585 *mq_name_s2c = mq_s2c->absolute_name();
1586 return true;
1587} // Server_session_impl::make_channel_mqs()
