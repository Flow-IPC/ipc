31/**
32 * This is to vanilla Session_server what shm::classic::Server_session is to vanilla #Server_session: it is
33 * the session-server type that starts SHM-enabled sessions with SHM-classic provider
34 * (ipc::shm::classic::Pool_arena). Its API is identical to that of Session_server, except that it
35 * emits #Server_session_obj that are shm::classic::Server_session and not vanilla #Server_session. In addition:
36 *
37 * ### Max pool size configuration API (optional) ###
38 * If using this, as opposed to (at least) SHM-jemalloc provider (session::shm::arena_lend::jemalloc::Session_server),
39 * you could potentially encounter "No space left on device" (`ENOSPC` in at least Linux) in async_accept().
40 * This has nothing to do with drive space, or even physical RAM in fact. It has to do with certain kernel parameters
41 * governing virtual SHM-mapped space. If this becomes a problem then please look into
42 * Session_server::pool_size_limit_mi() API. See the doc header for the accessor for discussion.
43 *
44 * In most cases it should not come up.
45 *
46 * @internal
47 * ### Implementation ###
48 * See similar section of session::Session_server. It explains why we sub-class Session_server_impl and even how
49 * how that's used for this SHM-classic scenario. To reiterate:
50 *
51 * We use 2 of 2 available customization points of `private` super-class Session_server_impl. We:
52 * - pass-up a `per_app_setup_func()` that, given the new session's desired Client_app, creates-if-needed the per-app
53 * SHM-arena and keeps handle open as well as available via `this->app_shm(Client_app::m_name)`; and
54 * - parameterize Session_server_impl on shm::classic::Server_session which, during log-in, creates
55 * the per-session SHM-arena and keeps handle open; and saves `this->app_shm(Client_app::m_name)`.
56 *
57 * shm::classic::Server_session doc header delves deeply into the entire impl strategy for setting up these arenas.
58 * If you read/grok that, then the present class's impl should be straightforward to follow.
59 * @endinternal
60 *
61 * @tparam S_MQ_TYPE_OR_NONE
62 * See vanilla #Session_server.
63 * @tparam S_TRANSMIT_NATIVE_HANDLES
64 * See vanilla #Session_server.
65 * @tparam Mdt_payload
66 * See vanilla #Session_server.
67 */

125 /**
126 * Destructor: contract is identical to session::Session_server dtor.
127 *
128 * @internal
129 * In particular this dtor shall perform
130 * shm::classic::Pool_arena::remove_persistent() on *each* pool returnable by app_shm()
131 * (the per-Client_app pools). Reminder: like all kernel-persistent `remove_persistent()` methods this will not
132 * necessarily free the RAM held on behalf of a given pool; any currently opened `Pool_arena`s -- presumably
133 * ones still (possibly) held by counterpart `shm::classic::Client_session_impl`s -- will continue to function
134 * fine.
135 *
136 * ### Rationale for aforementioned `remove_persistent()` op ###
137 * Why do it? Answer: Naturally this should be freed sometime; otherwise the RAM will leak, even though the
138 * session pertaining to the per-session pool is gone.
139 *
140 * Why not do it earlier? Answer: It would be nice to do it super-early... like, say, just after creating the
141 * pool during the async portion of async_accept(), namely in init_app_shm_as_needed().
142 * The only reason *not* to do it is if a client handle (`Pool_arena`) to the pool has not yet had a chance to
143 * get opened by the name deleted via `remove_persistent()`. But this can occur an indefinite number of times
144 * after init_app_shm_as_needed() creates the pool: the whole idea is the pool is shared between different
145 * sessions with the same (by value equality) Client_app, across different client processes in fact.
146 *
147 * Hence we postpone it until this dtor, at which point all the sessions are definitely done-for.
148 *
149 * Why not do it later? Answer: Session-server is done-for, if we are being invoked; the counterpart
150 * `Client_session`s could not possibly hope to open the pool now, even if they somehow wanted to do so. So why wait?
151 *
152 * ### What if this dtor never runs? ###
153 * In any program that exits without aborting abruptly, it *will* run -- either proactively during user program
154 * de-init code or implicity when it exits.
155 *
156 * It however may not run due to abrupt termination (a crash). In that case the `app_shm()` pools will indeed
157 * leak, in that this dtor won't execute, so the `remove_persistent()`s will not occur in that fashion.
158 * The shm::classic::Session_server has a contingency for this, which we don't describe here, wherein it has
159 * *cleanup points* at which it cleans up anything thus leaked during previous invocations of the same
160 * Server_app.
161 *
162 * ### What about the per-session pools? ###
163 * The above talks only of the app_shm()-returned pools; what about shm::classic::Session_impl::session_shm()?
164 * Shouldn't we clean it up here? Answer: Well, we could, but since it's a per-session thing,
165 * each shm::classic::Session_server_impl takes care of its own `session_shm()`.
166 */
167 ~Session_server();

171 /**
172 * The pool-size value, in mebibytes, which will be used to size the pool in subsequent
173 * `async_accept()`s. One can change this via the mutator overload.
174 *
175 * A large default value is used if one does not invoke that mutator.
176 *
177 * ### How it works ###
178 * Each time async_accept() succeeds, a SHM pool sized according to this value is created for the session;
179 * and potentially another SHM pool sized similarly is created for the Client_app (if and only if a session against
180 * the same Client_app has not yet been opened by `*this`). The *key point*: only a tiny amount of RAM
181 * is actually taken at that time for each pool; the pool size only counts against vaddr space which is
182 * essentially unlimited. Physical RAM is reserved only upon actual allocation for objects subsequently, with
183 * kernel-page-sized quantization. Therefore a huge value can be used here with no RAM-use penalty; on the
184 * other hand if this value is too small, and an allocation makes a pool run out of vaddr space, then
185 * a `bad_alloc` exception will be thrown, and you're pretty much kaput. So you should use a huge value!
186 * And indeed the default is quite large.
187 *
188 * Unfortunately, at least in Linux, there is nevertheless a system-wide limit against the sum of these
189 * SHM-pool virtual sizes. This is a kernel parameter and is usually admin-configurable; it might default to
190 * half your physical RAM for example. Therefore unfortunately if too much virtual space is used by active
191 * SHM-pools across the system, a Linux (at least) `ENOSPC` (No space left on device) error might result
192 * (in our case be passed to async_accept() completion handler). In that case, you can either tweak
193 * the relevant kernel parameter(s); or use pool_size_limit_mi() mutator to reduce your pool sizes -- assuming
194 * of course it'll be sufficient for your allocation needs.
195 *
196 * For most use cases none of this will be a problem. If it becomes a problem, either use a solution above;
197 * or consider ipc::session::shm::arena_lend::jemalloc::Session_server (SHM-jemalloc) which is a multi-pool
198 * system that adjusts dynamically without your having to worry about it at all.
199 *
200 * @return See above.
201 */
202 size_t pool_size_limit_mi() const;

209 * A large default value is used if one does not call this.

281 /**
282 * Returns pointer to the per-`app` SHM-arena, whose lifetime extends until `*this` is destroyed;
283 * or null if the given Client_app has not yet opened at least 1 shm::classic::Server_session via
284 * async_accept(). Alternatively you may use shm::classic::Session_mv::app_shm() off any session object
285 * filled-out by `*this` async_accept(), as long as its Server_session_mv::client_app() equals
286 * `app` (by App::m_name equality).
287 *
288 * If non-null is returned, then the same pointer value shall be returned for all subsequent calls
289 * with the same (by App::m_name equality) `app`. The non-null pointers returned for any 2 calls, where `app`
290 * is different (by App::m_name equality) among them, shall always differ.
291 *
292 * See shm::classic::Session_mv::Arena doc header for useful instructions on working with #Arena,
293 * `lend_object()`, and `borrow_object()`.
294 *
295 * ### Perf ###
296 * Given the choice between Server_session_mv::app_shm() and the present method, the latter is somewhat
297 * slower; internally it involves a mutex-protected map lookup, while the former simply returns a cached
298 * pointer as of this writing.
299 *
300 * Generally it is also quite fast for the user to save any non-null value returned by either `app_shm()`;
301 * the pointer returned shall always be the same after all.
302 *
303 * @internal
304 * ### Thread safety ###
305 * For internal use, namely by shm::classic::Server_session_impl::async_accept_log_in() at least,
306 * it is guaranteed the app_shm() may be called on the same `*this` concurrently to itself
307 * and init_app_shm_as_needed(). Formally speaking this isn't publicly documented, as I (ygoldfel) didn't want
308 * to get users into any bad habit, but internally it does have this property -- as it is required.
309 * @endinternal
310 *
311 * @param app
312 * Client_app whose segregated SHM-arena to return, if a session for a client of the app has been
313 * opened prior to this call.
314 * @return Pointer to `*this`-held per-`app` SHM-arena, if it has been created; null otherwise.
315 * See above.
316 */
317 Arena* app_shm(const Client_app& app);

384 /**
385 * Implementation of our `per_app_setup_func()` customization point used in the ctor impl:
386 * invoked from an unspecified thread, asynchronously, by async_accept() during log-in, it
387 * synchronously makes it so that `this->app_shm(app)` returns non-null from now on.
388 * What it returns is a pointer (always the same one) to the segregated SHM-arena pertaining to
389 * all sessions (current and future) pertaining to `app` in `*this` server. On success returns
390 * falsy #Error_code. Success means either `app_shm(app)` was already going to return non-null,
391 * or it wasn't, but we successfuly made it so it will.
392 *
393 * On failure to set this up:
394 * - Truthy #Error_code reason for failure is returned.
395 * - `app_shm(app)` shall continue to return null.
396 * - `*this` is not hosed.
397 *
398 * This method is thread-safe to invoke on the same `*this` w/r/t concurrent calls to itself
399 * and app_shm().
400 *
401 * @param app
402 * The Client_app for which to set up, if needed, the per-app SHM-arena.
403 * @return See above; falsy if and only if `app_shm(app)` shall return non-null from now on.
404 */
405 Error_code init_app_shm_as_needed(const Client_app& app);

428 * Set of names of every SHM-pool created w/r/t #m_app_shm_by_name. Each time an #Arena is added to

474 /* This is a (as of this writing -- the) *cleanup point* for any pools previously created on behalf of this
475 * Server_app by previous active processes before us. The session_shm() pools are gracefully cleaned up
476 * in shm::classic::Server_session_impl dtor; while app_shm() pools are gracefully cleaned up by
477 * our own dtor, as set up a few lines down via Impl::sub_class_set_deinit_func(). This cleanup point is a
478 * best-effort attempt to clean up anything that was skipped due to one or more such destructors never getting
479 * to run (due to crash, abort, etc.).
480 *
481 * We simply delete everything with the Shared_name prefix used when setting up app_shm() pools
482 * (see init_app_shm_as_needed()) and session_shm() pools
483 * (see shm::classic::Server_session_impl::async_accept_log_in()). The prefix is everything up-to
484 * (not including) the PID (m_srv_namespace). Our own m_srv_namespace was just determined and is unique
485 * across time by definition (internally, it's -- again -- our PID); so any existing pools are by
486 * definition old. Note that as of this writing there is at most *one* active process (instance) of a
487 * given Server_app. */
488 util::remove_each_persistent_with_name_prefix<Arena>
489 (get_logger(), build_conventional_shared_name_prefix(Shared_name::S_RESOURCE_TYPE_ID_SHM,
490 Shared_name::ct(m_srv_app_ref.m_name)));

492 // Set up the final-cleanup step via impl customization point. For explanation see init_app_shm_as_needed().
493 Impl::sub_class_set_deinit_func([this,
494 /* ^-- Careful: We only use this->get_logger() which is explicitly okay.
495 * *this is destroyed w/r/t shm::classic::Session_server_impl by the
496 * time this lambda is executed; so we do not use that.
497 *
498 * Copy-capture the shared_ptr (see rationale in m_shm_pool_names doc header).
499 * Basically this ensures the vector<Shared_name> lives, even after
500 * *this->m_shm_pool_names has been destroyed via our own dtor, before
501 * Session_server_impl dtor runs. */
502 shm_pool_names = m_shm_pool_names]()
503 {
504 // We are in thread U (in dtor chain).
505 Error_code sink; // We do not care about any error. This is our best effort; it'll WARN on (unlikely) error.
506 for (const auto& shm_pool_name : *shm_pool_names)
507 {
508 Arena::remove_persistent(this->get_logger(), shm_pool_name, &sink);
509 }
510 });

529TEMPLATE_CLSC_SESSION_SRV
530Error_code CLASS_CLSC_SESSION_SRV::init_app_shm_as_needed(const Client_app& app)
531{
532 using boost::movelib::make_unique;
533
534 /* We are in some unspecified thread; actually *a* Session_server_impl thread Ws (a Server_session_impl thread W).
535 * Gotta lock at least to protect from concurrent calls to ourselves on behalf of other async_accept()s. */
536
537 Lock_guard app_shm_lock(m_app_shm_mutex);
538
539 auto& app_shm = m_app_shm_by_name[app.m_name]; // Find; insert if needed.
540 if (app_shm)
541 {
542 // Cool; already exists, as app.m_name seen already (and successfully set-up, as seen below) by us.
543 return Error_code();
544 }
545 // else
546
547 // Below should INFO-log already; let's not litter logs explaining why this is being created; context = sufficient.
548
549 auto shm_pool_name = build_conventional_shared_name(Shared_name::S_RESOURCE_TYPE_ID_SHM,
550 Shared_name::ct(m_srv_app_ref.m_name),
551 m_srv_namespace,
552 Shared_name::ct(app.m_name))
553 / SHM_SUBTYPE_PREFIX
554 / Shared_name::S_1ST_OR_ONLY;
555 /* As of this writing we only maintain the one --^-- per-Client_app SHM pool. (Of course the user can do whatever
556 * they want and make their own -- but within the ipc::session-maintained ones, this is it.)
557 * So use 1ST_OR_ONLY per its doc header; and as that suggests, if we want to maintain more in the future
558 * we can switch to a uint or atomic<uint> data member instead. */
559
560 Error_code err_code;
561 app_shm = make_unique<Arena>(get_logger(), shm_pool_name, util::CREATE_ONLY,
562 size_t(1024 * 1024) * pool_size_limit_mi(),
563 util::shared_resource_permissions(m_srv_app_ref.m_permissions_level_for_client_apps),
564 &err_code);
565 /* Either err_code is truthy/app_shm is null; or vice versa. In the former case just leave null in the map; meh.
566 * .erase()ing it from there is just pedantic at best. (The [] lookup above will do the right thing next time.) */
567
568 /* Cool! Let's return (and continue the log-in async op!). ...Not so fast. Almost.
569 * Please now see the doc header for our dtor, where we promise -- on dtor invocation -- to remove all
570 * app_shm() pools. It explains why we do that, and what actual effect it has. The below is *how*
571 * we accomplish it: We have just added a pool to what app_shm() can return; and thus (and this is the only way it
572 * would need cleaning up) we now ensure that *in* the dtor, the pool removal shall execute.
573 *
574 * The technique we use, and the rationale for it, is very similar to that in
575 * shm::classic::Server_session_impl::async_accept_log_in() (in that case for the per-session pool). Nevertheless
576 * here is a full explanation tailored to our specific case.
577 *
578 * It's tempting to Just Do It in the dtor, but there are caveats. Vaguely, all we want to do is something like,
579 * right in the dtor, for all m_shm_pool_names N (each computed into shm_pool_name above):
580 * Arena::remove_persistent(N);
581 * This would work 99.999% of the time just fine... except for the annoying possibility that the user
582 * destroys *this while an async_accept() is outstanding still.
583 *
584 * In my mind this leaves two possible approaches.
585 *
586 * One is to operate in the various threads started by *this (really by super-class Session_server_impl).
587 * However this is entropy-laden and seems to involve a race against the potentially ongoing async_accept()s.
588 * Like, suppose we do that -- and some app_shm() returnee is not yet set-up -- so we don't clean its name;
589 * then while the super-class dtors are running (including the highly significant Session_server_impl dtor)
590 * the async-accept happens to advance a bit and adds an app_shm() returnee, before that dtor has a chance to
591 * stop/join the relevant thread(s). Now we've leaked the RAM. (Granted, the anti-crash contingency
592 * in shm::classic::Session_server should catch it when the next
593 * instance -- if any -- of this Server_app starts up. Relying on that outside a crash scenario is a misuse
594 * of that contingency logic; there is nothing un-graceful about our dtor being invoked before an async-accept
595 * completes.)
596 *
597 * The other approach is more orderly: We know the dtor chain is being invoked, so when ~Session_server_impl()
598 * does execute (very soon), it will indeed stop all relevant threads -- in whatever state they happen to be
599 * at the time. At *that* stage there is no more race involved: there's just what that super-class dtor executes
600 * (in caller thread U) after stopping all internal threads. Now the state is straightforward: Either
601 * the various internal threads worked together to add shm_pool_name-pool into app_shm() returnees --
602 * then shm_pool_name should be removed -- or it did not (then there is nothing to remove).
603 *
604 * So that is what we do. Session_server_impl supplies a `protected` API for any custom de-init code which
605 * we invoke elsewhere and which scans *m_shm_pool_names. We simply use that by adding to the latter;
606 * and we only do that right here, the one place we add a new app_shm() returnee. */
607 if (app_shm)
608 {
609 m_shm_pool_names->emplace_back(std::move(shm_pool_name));
610 }
611
612 return err_code;
613 // Lock_guard app_shm_lock(m_app_shm_mutex): unlocks here.
614} // Session_server::init_app_shm_as_needed()

617typename CLASS_CLSC_SESSION_SRV::Arena* CLASS_CLSC_SESSION_SRV::app_shm(const Client_app& app)
618{
619 // We are in some unspecified thread; we promised thread safety form any concurrency situation.
620
621 Lock_guard app_shm_lock(m_app_shm_mutex);
622
623 /* Subtlety: Due to an intentional quirk of init_app_shm_as_needed(), if it's in the map, the ptr may still be null:

625 const auto map_it = m_app_shm_by_name.find(app.m_name);
626 return (map_it == m_app_shm_by_name.end()) ? nullptr : map_it->second.get();
627
628 // Lock_guard app_shm_lock(m_app_shm_mutex): unlocks here.
629}
