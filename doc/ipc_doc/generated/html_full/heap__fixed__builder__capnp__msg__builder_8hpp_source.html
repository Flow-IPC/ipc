35/**
36 * A `capnp::MessageBuilder` used by Heap_fixed_builder: similar to a `capnp::MallocMessageBuilder`
37 * with the FIXED_SIZE alloc-strategy but with framing space around the allocated segment(s).
38 *
39 * It can also be used as a #Capnp_msg_builder_interface (`capnp::MessageBuilder`) independently of the rest
40 * of ipc::transport or even ::ipc, although that was not the impetus for its development.
41 *
42 * ### Move-ctible and move-assignable ###
43 * Super-class #Capnp_msg_builder_interface, which is an abstract class but not a pure interface (it has data
44 * and concrete methods), is move-ctible and move-assignable (but not copyable). Accordingly, the present
45 * class is as well. (So is our peer `capnp::MallocMessageBuilder`.) However: as of this writing
46 * (newest version capnp 0.10) #Capnp_msg_builder_interface consists of a raw data area that = 1 `bool` + 22
47 * words; or at least 176 bytes in a 64-bit system. Its (defaulted) move ctor/assignment will thus need to
48 * copy the 176+ bytes. That is a little "fat." We add another 3 `size_t`, a pointer, and a `vector` of 1+
49 * pointers, or 30+ bytes. Hence the user may wish to wrap us in a `unique_ptr`, and only then use the move
50 * semantics; as the trade-off against the extra alloc/dealloc is arguably worth it. The same arguably holds
51 * for any #Capnp_msg_builder_interface sub-class (interface implementation).
52 *
53 * Style note: Much of the rest of the code explicitly documents deleted copy-ctor/assignment and defaulted
54 * move-ctor/assignment (as separate items each with a doc header); we do not. This is of course for brevity;
55 * but we justify it in that the abstract class (interface) we implement (#Capnp_msg_builder_interface) omits them
56 * as well.
57 *
58 * ### Rationale ###
59 * Why we'd use a `MallocMessageBuilder`-like approach is explained within Heap_fixed_builder doc header.
60 * So why not just use that actual class? Answer: We send the resulting serialization segments over IPC pipes
61 * (Blob_sender, Native_handle_sender), and in some cases struc::Channel needs to surround the serialization
62 * with a bit of metadata. `MallocMessageBuilder` will, for each segment, allocate X bytes -- and then potentially
63 * fill all of them. The only way for struc::Channel to work with this is to allocate another buffer of
64 * X + (frame size) bytes and then copy the X bytes into it -- losing the adherence to the zero-copy-when-possible
65 * paradigm. Hence this class which will use `flow::util::Blob` machinery to maintain the frame space.
66 *
67 * (Update: The previous paragraph remains correct in that in order to support the framing feature it was necessary
68 * to write this class, as `MallocMessageBuilder` does not have it. However struc::Channel no longer needs
69 * to frame "a bit of metadata" around the serialization; hence it does not require this framing feature; hence
70 * we could now get rid of the present class and just use `MallocMessageBuilder`.
71 * Nevertheless I (ygoldfel) have kept this class around, as it is a nice feature in general. Plus niceties
72 * like emit_segment_blobs() and n_segments() remain nice.)
73 *
74 * Note that in order to make use of the added feature (frame space) one should *not* access the serialization result
75 * via the usual `MessageBuilder::getSegmentsForOutput()`. That still works, and that method is essential in
76 * the impl of this class; but it returns only the actual serialization segments (each one's start and size,
77 * not including for example the allocated-but-unused data probably at the tail of the allocated buffer).
78 * Technically it would indeed be correct/safe to simply write frame prefix data by subtracting `frame_prefix_sz`
79 * from the segment's start pointer and writing there; and similarly write postfix data immediately past the end of the
80 * start ptr + size. However that is error-prone and inconvenient. Call emit_segment_blobs() instead: it will return
81 * the `flow::util::Blob`s which will have set `start()` and `size()` appropriately (see emit_segment_blobs()
82 * doc header).
83 */

92 /**
93 * Constructs the message-builder, with the policy to allocate the segments in a particular way, always
94 * with a fixed size. Namely, capnp will call allocateSegment() with `min_sz * sizeof(word) == M`. This will:
95 * - Create an internal `flow::util::Blob B` with:
96 * - `capacity()` = `max(M, segment_sz) + frame_prefix_sz + frame_postfix_sz`, up to ceiling-nearest word;
97 * - `start()` = `frame_prefix_sz`.
98 * - `size()` = `capacity() - start() - frame_postfix_sz` which equals ~`max(M, segment_sz)`.
99 * - Return `ArrayPtr` pointing at `B.begin()` (which is buffer start + `start()`),
100 * sized `B.size()` (that is, `max(M, segment_sz)` adjust to ceiling-nearest word).
101 *
102 * Furthermore, this will cause the following emit_segment_blobs() behavior:
103 * - It will emit a sequence of `flow::util::Blob` pointers to the above.
104 * - Each `Blob` pointer will correspond to an earlier allocateSegment() call 1-1.
105 * - The pointee's `start()` will be as above.
106 * - The pointee's `size()` will equal the serialization segment's size as determined ultimately by capnp
107 * (per `getSegmentsForOutput()`).
108 * This will be at most `max(M, segment_sz)` (less if it didn't need every single byte to hold the
109 * serialization ultimately -- it is typical some bytes are wasted).
110 * - Its `capacity() - start() - size() >= frame_postfix_sz`.
111 *
112 * In this way the consumer of emit_segment_blobs() (presumably struc::Channel) will be able to
113 * adjust `start()` and `size()` of each serialization `Blob` to add framing info if needed.
114 *
115 * @param logger_for_blobs
116 * Passed to each `Blob` ctor.
117 * @param segment_sz
118 * See above.
119 * @param frame_prefix_sz
120 * Guaranteed space for prefix framing. See above.
121 * This must be a multiple of `sizeof(void*)`; else behavior undefined (assertion may trip).
122 * @param frame_postfix_sz
123 * Guaranteed space for postfix framing. See above.
124 */
125 explicit Heap_fixed_builder_capnp_message_builder
126 (flow::log::Logger* logger_for_blobs, size_t segment_sz, size_t frame_prefix_sz, size_t frame_postfix_sz);
127

130 /**
131 * Returns pointers to `Blob`s which are the serialization segments at this time. They are as follows:
132 * - Each `Blob` holds a serialization segment, in order. There shall be at least 1.
133 * - For each `Blob B` (see ctor):
134 * - `start() == frame_prefix_sz`.
135 * - `capacity() - start() - size() >= frame_postfix_sz`.
136 * - [`begin()`, `end()`) is the serialization segment. Recall that `begin()` starts at position `start()`,
137 * while `end()` is at position `start() + size()`.
138 *
139 * Therefore one can use `B.resize()` and similar and then add frame data without breaking zero-copy.
140 *
141 * This method can be invoked more than once; however the emitted result is meaningless once any mutation occurs
142 * subsequently.
143 *
144 * @param target_blob_ptrs
145 * It is appended-to. Recall these are mere pointers (into `*this`-managed heap memory).
146 */
147 void emit_segment_blobs(Segment_ptrs* target_blob_ptrs);
148
149 /**
150 * Returns what `target_blob_ptrs.size()` would return after calling `emit_segment_blobs(&target_blob_ptrs)` (with
151 * an empty `target_blob_ptrs` going-in), right now. In other words it is the number of times capnp has invoked
152 * allocateSegment().
153 *
154 * @return See above.
155 */
156 size_t n_segments() const;

158 /**
159 * Implements `MessageBuilder` API. Invoked by capnp, as the user mutates via `Builder`s. Do not invoke directly.
160 *
161 * @note The strange capitalization (that goes against standard Flow-IPC style) is because we are implementing
162 * a capnp API.
163 *
164 * For details as to how this will behave in context of `*this` see Heap_fixed_builder_capnp_message_builder()
165 * ctor doc header.
166 *
167 * @param min_sz
168 * See `MessageBuilder` API.
169 * The allocated segment will allow for a serialization of at most
170 * `min(min_sz * sizeof(word), segment_sz)` bytes, where `segment_sz` was passed to ctor
171 * (adjusted to the ceiling-nearest word).
172 * @return See `MessageBuilder` API.
173 * The ptr and size of the area for capnp to serialize-to.
174 * (As noted in ctor doc header: `frame_prefix_sz` bytes will precede this area, and
175 * `frame_postfix_sz` will succeed it.)
176 */
177 kj::ArrayPtr<::capnp::word> allocateSegment(unsigned int min_sz) override;

192 * Before emit_segment_blobs(), stores each segment allocated as requested by capnp via allocateSegment() `virtual`
