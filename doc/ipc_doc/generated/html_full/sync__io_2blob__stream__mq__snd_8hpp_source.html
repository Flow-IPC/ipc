31/**
32 * Implements sync_io::Blob_sender concept by using an adopted Persistent_mq_handle MQ handle to an MQ (message queue)
33 * of that type, such as a POSIX or bipc MQ. This allows for high-performance, potentially zero-copy (except
34 * for copying into the transport MQ) of discrete messages, each containing a binary blob.
35 * This is the `sync_io`-pattern counterpart to transport::Blob_stream_mq_sender -- and in fact the latter use an
36 * instance of the present class as its core.
37 *
38 * @see transport::Blob_stream_mq_sender and util::sync_io doc headers. The latter describes the general pattern which
39 * we implement here; it also contrasts it with the async-I/O pattern, which the former implements.
40 * In general we recommend you use a transport::Blob_stream_mq_sender rather than a `*this` --
41 * but you may have particular needs (summarized in util::sync_io doc header) that would make you decide
42 * otherwise.
43 *
44 * ### Informal comparison to other core transport mechanisms ###
45 * Notes for transport::Blob_stream_mq_sender apply.
46 *
47 * ### Relationship between `Blob_stream_mq_*`, the underlying MQ resource, and its Shared_name ###
48 * Notes for transport::Blob_stream_mq_sender apply.
49 *
50 * ### Cleanup ###
51 * Notes for transport::Blob_stream_mq_sender apply.
52 *
53 * ### Reusing the same absolute_name() for a new MQ and one-direction pipe ###
54 * Notes for transport::Blob_stream_mq_sender apply.
55 *
56 * ### Thread safety ###
57 * Notes for transport::Blob_stream_mq_sender apply.
58 *
59 * @internal
60 * ### Implementation design/rationale ###
61 * Notes for transport::Blob_stream_mq_sender apply: the pImpl stuff; and the fact that:
62 *
63 * The rest of the implementation is inside sync_io::Blob_stream_mq_sender_impl and is discussed in that class's
64 * doc header.
65 *
66 * @see sync_io::Blob_stream_mq_sender_impl doc header.
67 *
68 * @endinternal
69 *
70 * @see sync_io::Blob_sender: implemented concept.
71 */
