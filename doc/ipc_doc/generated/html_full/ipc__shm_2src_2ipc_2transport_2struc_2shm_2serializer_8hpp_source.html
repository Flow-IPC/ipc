49/**
50 * Implements Struct_builder concept with maximal zero-copy perf by (1) storing the actual user-schema-controlled
51 * message using the SHM provider of choice, in SHM, and (2) straightforwardly allocating 1 segment in regular heap
52 * and placing the SHM handle there for transmission over IPC. That is, each mutation via payload_msg_builder()
53 * may, as needed, trigger a SHM allocation. In this algorithm, unlike Heap_fixed_builder, the size of
54 * each (SHM-)allocated segment will be determined intelligently based on an exponential-growth algorithm,
55 * similarly to that mode of capnp's `MallocMessageBuilder`. Hence there is no knob to control segment size, as
56 * it is determined dynamically, internally.
57 *
58 * ### Failure mode; reasonable uses ###
59 * First see "Failure mode" and "Reasonable uses" in Heap_fixed_builder doc header;
60 * then come back here. Back already? Good news:
61 * that failure mode (leaf-too-big) does *not* apply here. As long as the SHM provider is able to dole out RAM,
62 * there is no limit at all on the size/complexity of what one mutates over payload_msg_builder(). In fact `*this`
63 * provides the "2-layer approach" specified in that doc header section you just came back from reading.
64 *
65 * The only realistic failure mode in `*this` is the following:
66 * -# User attempts to mutate via payload_msg_builder().
67 * -# capnp internals recognize the current segment is used up and asks for a new segment of at least N bytes.
68 * -# `*this` asks the SHM provider (see template param docs below) to allocate >=N bytes.
69 * -# The SHM provider determines it has run out of space according to its policies/algorithm/limitations and
70 * throws exception, in spirit similar to `std::bad_alloc`.
71 * -# This propagates to the user via their payload_msg_builder() mutation statement: it throws.
72 *
73 * The key here: This cannot be detected via `*this`. In spirit it is similar to `Heap_fixed_builder`-originated
74 * mutations by the user running out of regular-heap (even virtual disk heap, if enabled) and is outside our scope
75 * to control. The only remedy: choose a SHM provider that does not run out of space beyond simply running out of
76 * RAM; e.g., by mapping more SHM pools, or whatever. Otherwise, all the user can do is catch the `bad_alloc`-like
77 * exception around their mutations on payload_msg_builder() and take whatever contingency steps.
78 *
79 * @see shm::Reader
80 * Counterpart Struct_reader implementation that can deserialize data that this has serialized.
81 * @see Struct_builder: implemented concept.
82 *
83 * @tparam Shm_arena
84 * SHM provider type with the ability to allocate STL-compliant data structures directly in SHM, returning
85 * a `shared_ptr` outer-SHM-handle that is possible to `lend_object()`/`borrow_object()` via
86 * shm::Builder::Session.
87 * `Shm_arena::Handle<T>` must equal `shared_ptr<T>` (meaning type with standard `shared_ptr<T>` semantics;
88 * in practice probably either `std::shared_ptr<T>` or `boost::shared_ptr<T>`).
89 * It must provide a `construct<T>()` method as exemplified by shm::classic::Pool_arena::construct().
90 * (This is an *example*; you do not have to use `Pool_arena` and indeed should choose a SHM provider
91 * suitable to your needs, especially as regards to being able to allocate sufficiently large segments.)
92 * That is: `Handle<T> construct<T>(...ctor_args...)`, such that if it returned `p` then
93 * `session::shm::Arena_to_shm_session_t<Shm_arena>` pointee has method
94 * `flow::util::Blob_sans_log_context lend_object<T>(const Handle<T>& p)`. Next, if the returned blob `b`
95 * is bit-wise copied into an IPC transport and copied out of it and then given to
96 * counterpart shm::Reader, then `shm::Reader::Session` pointee has method `borrow_object<T>(b)` that shall
97 * return (in the receiving process) `Handle<T>` that points to the same SHM-stored data structure originally
98 * returned by `construct()`. In addition `Shm_arena` must be compatible with `Stateless_allocator`
99 * requirements as explained in "Additional formal requirements" below.
100 *
101 * ### Additional formal requirements w/r/t `Shm_arena` ###
102 * Information on `T` that `*this` shall use with `Shm_arena::construct<T>()`: As of this writing it is
103 * `list<Basic_blob>`, representing (internally) the capnp-requested 1+ segments in which the serialization
104 * mutated via payload_msg_builder() is stored in zero-copy fashion. In order for this to work, the `Allocator`
105 * template arg on both `list` and the inner `Blob`s shall be ipc::shm::stl::Stateless_allocator.
106 * Therefore, the requirement is: `Shm_arena` must formally meet the requirements for `Arena` from
107 * ipc::shm::stl::Stateless_allocator doc header. Briefly these are: `allocate(n) -> void*`,
108 * `deallocate(void*)`, `Pointer` fancy-pointer type that is SHM-storable.
109 *
110 * A simple (likely the simplest) example is ipc::shm::classic::Pool_arena, which satisfies both
111 * `Shm_arena` and `session::shm::Arena_to_shm_session_t<Shm_arena>`-pointee requirements. However, it has deficiencies
112 * w/r/t max pool size having to be specified and a non-industry-strength allocation algorithm. The Jemalloc-based SHM
113 * provider in ipc::shm::arena_lend::jemalloc is more complex -- with a separate
114 * `Arena_to_shm_session_t<Shm_arena>` pointee from `Shm_arena` -- but lacks these problems.
115 */

291 * while user mutates via payload_msg_builder(). This is the bottom-serialization builder.
292 *
293 * As explained at the top of the class doc header, this one uses a segment-sizing strategy similar
294 * that of `capnp::MallocMessageBuilder` operating in `GROW_HEURISTICALLY` mode. I.e., it'll start with
295 * a reasonable guess for segment 1 size; then grow exponentially each time a new segment is requested.

321/**
322 * Implements Struct_reader concept by interpreting a serialization by shm::Builder with the
323 * same template params. If one understands how to use shm::Builder (see its doc header in-depth),
324 * it should be straightforward to figure out how to create/use a `*this`. Please read that; then come
325 * back here.
326 *
327 * ### How `Shm_arena` is used by a `*this` ###
328 * It is a good question, as shm::Reader is conceptually read-only w/r/t the SHM-stored serialization
329 * accessed by payload_msg_builder(). So why does it need any #Arena? Answer: well, it does not need *an* #Arena;
330 * unlike with shm::Builder you don't need to pass-in any `Arena*` into the ctor or elsewhere. That makes sense:
331 * it never allocates anything in SHM, only reads it. It does however need an equal #Arena template param.
332 * Why? Answer: There are internal reasons having to do with STL-compliant internal storage by the counterpart
333 * shm::Builder; to decode its data structures properly it needs to have some type information at compile-time;
334 * this "supplies" code -- not data.
335 *
336 * Anyway, just supply an equal #Arena template param. Should be fine.
337 *
338 * ### How shm::Reader::Session (a/k/a `Shm_session`) is used by a `*this` ###
339 * shm::Builder, internally, performs `Shm_arena::construct<T>()` to allocate a STL-compliant data structure
340 * and yield a `Shm_session`-lendable outer SHM handle. It performs `Shm_session::lend_object<T>()` in
341 * shm::Builder::emit_serialization(), registering the recepient-to-be process as the 2nd owner process.
342 * Then the counterpart `*this` performs `Shm_session::borrow_object<T>()` to recover the equivalent
343 * outer SHM handle to the same STL-compliant data structure the user had built up via
344 * shm::Builder::payload_msg_builder().
345 *
346 * @see shm::Builder
347 * The counterpart Struct_builder implementation that can create compatible serializations.
348 * @see Struct_reader: implemented concept.
349 *
350 * @tparam Shm_arena
351 * Must be the same as the serializing Builder counterpart. Note that no object of this type
352 * is required to the ctor unlike with Builder. Only the type is necessary (see discussion above).
353 */
354template<typename Shm_arena>

554void CLASS_SHM_BUILDER::emit_serialization(Segment_ptrs* target_blobs, const Session& session,
555 Error_code* err_code_or_null) const
556{
557 assert(m_btm_engine && "Are you operating on a moved-from `*this`?");
558
559 /* Key subtlety: It may seem correct here to do initRoot<>(), not getRoot<>(), but that would be wrong:
560 * they may be reusing `*this` (to send it out a 2nd, 3rd, ... time). Fortunately getRoot<>() does the right
561 * thing: the 1st time, it's identical to initRoot<>(); subsequently it's a no-op that returns the existing
562 * root ShmTopSerialization::Builder. (If getRoot<>() initially barfed instead, we'd have to keep a
563 * `bool m_initialized` and do initRoot first and set it to true, then getRoot once it's true.)
564 *
565 * Also we must either const_cast<>, or make m_top_engine mutable, for the following to compile.
566 * Of course that is to be questioned; we justify it as follows: getRoot<>() could be done in ctor, and we could
567 * cache the result there; instead we do it lazily on-demand -- a pattern that's a typical justification
568 * for `mutable` (and the following is essentially a smaller-scoped `mutable`). The first time it's done
569 * it create the root (which would otherwise be done in ctor); next time it does nothing; then in either case
570 * `root` becomes the same value.
571 */
572 auto root = const_cast<Heap_fixed_builder&>(m_top_engine)
573 .payload_msg_builder()->getRoot<schema::detail::ShmTopSerialization>();
574
575 /* We have the bottom serialization inside m_btm_engine; the data they'd mutated via this->payload_msg_builder()
576 * is stored in a bunch of SHM segments of whatever sizes was deemed sufficient, all in one STL-compliant
577 * data structure directly in SHM. This will finalize that data structure; *and* emit some scalar(s) that encode
578 * the outer SHM handle to it. These bits are emitted right into the top
579 * serialization which will be copied through to the recipient/reader. (Analogy: They could, instead,
580 * be emitted into a disk file which would be read by the reader. The file, like this top serialization,
581 * is small: just a few scalars. But by following where it points in SHM, we can access an arbitrarily huge
582 * and complex structure.) Also it ups the process-owner ref-count from 1 (us) to 2 (us + them), so that our
583 * dtor won't blow away the SHM-stored data structure unless Reader has already invoked its dtor too.
584 * (If this is emit_serialization() #2, then the ref-count grows potentially to 3, etc.) */
585 m_btm_engine->lend(&root, session); // Target *root; put the outer SHM handle's encoding there.
586
587 /* Secondly obtain that resulting top (local-heap) serialization. This will emit error as needed.
588 * In reality there is only one possible error condition (as documented): S_INTERNAL_ERROR_SERIALIZE_LEAF_TOO_BIG.
589 * It is essentially impossible in our case: m_btm_engine always emits a quite-small, constant-sized serialization
590 * for its SHM handle thingie; it will easily fit into any reasonable first segment and not even require any
591 * further segments, let alone overflow any segment.
592 *
593 * If this is emit_serialization() #2, #3, ...: This value will always be the same, since it's just the SHM handle's
594 * encoding; at least for the 2 SHM providers (SHM-classic, SHM-jemalloc) that's the case, and I (ygoldfel) can't
595 * conceive of anything different. @todo Considering simply no-op-ing here for any emit_serialization()s except the
596 * first one. It's not urgent, probably, as despite looking fancy, in this case the insides of the following call
597 * will barely do any work. */
598#ifndef NDEBUG
599 const size_t n_target_blobs_orig = target_blobs->size();
600#endif
601 m_top_engine.emit_serialization(target_blobs, NULL_SESSION, err_code_or_null);
602
603 assert(((err_code_or_null && *err_code_or_null)
604 || (target_blobs->size() == (n_target_blobs_orig + 1)))
605 && "We guarantee the top serialization consists of exactly 1 segment (storing SHM handle), no more.");
606} // Builder::emit_serialization()

653TEMPLATE_SHM_READER
654template<typename Struct>
655typename Struct::Reader CLASS_SHM_READER::deserialization(Error_code* err_code)
656{
657 using util::Blob_const;
658 using Blob = flow::util::Blob_sans_log_context;
659 using flow::error::Runtime_error;
660 using flow::util::buffers_dump_string;
661 using boost::movelib::make_unique;
662 using std::vector;
663 using ::capnp::word;
664 using Capnp_word_array_array_ptr = kj::ArrayPtr<const Capnp_word_array_ptr>;
665 using Capnp_struct_reader = typename Struct::Reader;
666 using Capnp_heap_engine_opts = ::capnp::ReaderOptions;
667
668 /* When constructing a MessageReader, it takes this ReaderOptions struct which defaults to certain values.
669 * In typical capnp use the user would do this themselves; but in our case we do it for them.
670 * @todo This should be part of our public API throughout (where relevant). Surely a ticket is filed. Until then:
671 * The defaults seem fine, except:
672 * The particular traversalLimitInWords option can cause trouble with large structures. (See capnp source message.h
673 * for its official docs.) It is a security feature for stuff transmitted over the wire; but as of *this* writing
674 * we do local IPC and explicitly assume trust. So until this is made configurable (and it really should be)
675 * it's an OK work-around to just shove a giant value here. Otherwise the mere act of reading a structure with
676 * many sub-structs (in absolute terms, not in terms of depth) can (and has, such as in our test suite's perf_demo)
677 * throw a capnp exception. */
678 constexpr Capnp_heap_engine_opts RDR_OPTIONS = { std::numeric_limits<uint64_t>::max() / sizeof(word),
679 Capnp_heap_engine_opts{}.nestingLimit };
680
681 // Helper to emit error via usual semantics. We return a ref so can't use FLOW_ERROR_EXEC_AND_THROW_ON_ERROR().
682 const auto emit_error = [&](const Error_code& our_err_code) -> Capnp_struct_reader
683 {
684 if (err_code)
685 {
686 *err_code = our_err_code;
687 return Capnp_struct_reader();
688 }
689 // else
690 throw Runtime_error(our_err_code, "shm::Reader::deserialization()");
691 return Capnp_struct_reader(); // Doesn't get here.
692 };
693
694 // The top serialization is simply this (but what it encodes is a handle to the bottom serialization).
695 const auto top_serialization_root
696 = m_top_engine.deserialization<schema::detail::ShmTopSerialization>(err_code);
697 // Either that threw (error), set *err_code to truthy (error), or succeeded. Check for that middle one.
698
699 if (*err_code)
700 {
701 return Capnp_struct_reader();
702 }
703 // else
704
705 assert((!m_btm_serialization_shm_handle) && "Did you call deserialization() more than once?");
706
707 /* Now get the bottom serialization out of SHM. To do so, really we mirror what Heap_reader does --
708 * SegmentArrayReader and all that -- but instead of getting it out of direct-serialized stuff from segments in
709 * regular heap, get it out of SHM based on the handle to list<Basic_blob>, where that handle is
710 * the one little thing stored in top_serialization_root. */
711 {
712 // top_serialization_root is a ShmTopSerialization::Reader.
713 const auto capnp_blob_reader = top_serialization_root.getSegmentListInShm();
714 Blob handle_serialization_blob;
715 capnp_get_shm_handle_to_borrow(capnp_blob_reader, &handle_serialization_blob);
716
717 /* And, as documented in the .capnp file -- and can be seen in Builder -- the handle is
718 * to Segments_in_shm, which is the aforementioned list<Basic_blob>. So interpret it that way
719 * (but read-only: we will never modify it, even if *m_arena is opened for read-write). */
720 m_btm_serialization_shm_handle = m_session->template borrow_object<Segments_in_shm>(handle_serialization_blob);
721 }
722 assert(m_btm_serialization_shm_handle);
723
724 /* Subtlety: We are about to work with serialization_segments, an STL-compliant structure stored directly in
725 * SHM. In so doing we'll have to traverse it with iterators and so on. Needn't we do some kind of
726 * Stateless_allocator/Arena_activator context-setting (e.g., as shm::Capnp_message_builder does when building
727 * up the thing we're now reading)? If so, we have a problem: what "arena" do we even activate? Fortunately
728 * (and it's not a fortunate happenstance thing but rather makes sense in terms of the generic asymmetric design
729 * that separates the write-side Arena and the -- in our case -- read Session) we in fact needn't. This bears
730 * explanation for context:
731 *
732 * Our access to this STL-compliant data structure is read-only. Our ref is formally to a const
733 * list-of-vectors-of-bytes (note: not a const list-of-POINTERS-to-vectors-of-bytes). Anything we do will *not*
734 * invoke any <allocator>.<do stuff -- allocate/deallocate()>() calls. It will certainly access
735 * <allocator>::pointer *type* which will invoke (crucially!) the proper fancy-pointer logic to dereference SHM-stored
736 * (not raw T*) pointers which allows following iterators among other things; but that ::pointer must be able
737 * to generate raw pointers (deref itself) without any context-setting help. So: we are fine.
738 *
739 * While the following question is not formally relevant to the present code, one might still wonder, so I'll
740 * comment on it: What about the destruction of this data structure down the line, like in our dtor when
741 * we let go of m_btm_serialization_shm_handle? If that is the last owner process to hold the cross-process
742 * handle to this in-SHM structure, won't "it" need to call its (list-of-vectors or w/e) dtor? And if so
743 * doesn't "something" need to set the allocator-context before the dtor is called? Well, firstly, when
744 * the Handle logic, in some process, decides it's time to call the dtor, it is in charge of setting the
745 * allocator-context. So somehow it must do the right thing. But for general education, how *does* it do the
746 * right thing? It's easiest to answer on the level of specific impls.
747 * - shm::classic::Pool_arena: This simple symmetric core class is both Arena and Session, and both sides
748 * (in our case the builder and the reader) open the same-named SHM pool in identical ways.
749 * In its case cross-process ref-count-0 is detected by either process; and whichever one it is
750 * simply sets itself (Arena::this) as the allocator-context before invoking dtor. So if it's us, the reader
751 * process, then it'll do that. This was ensured when *we* called Session::borrow_object(), which invisibly
752 * supplied the custom deleter which will in fact do what I just described. This Session::borrow_object(),
753 * really, what is it? It's shm::classic::Pool_arena::borrow_object(), because Session = Pool_arena.
754 * - SHM-jemalloc provider: This asymmetric setup involves, on the builder side, an Arena and a separate
755 * Session; and on the reader side a Session that is able to interpret the opposing Session's
756 * Session::lend_object() in its Session::borrow_object(). The reader-side Session::borrow_object() will set
757 * up a custom deleter, as it must, and that guy will (internally) IPC-inform the builder side Session that our
758 * local Handle ref-count=0. Hence the builder side is in charge of invoking the dtor, when the time comes,
759 * regardless of which side was last to local ref-count=0.
760 *
761 * Specifics aside, generically speaking: The custom deleter on the Handle is in charge of safely disposing
762 * of the structure when no other holder holds it any longer either -- and it can do it however it needs to.
763 * Part of its charge is to -- when it indeed invokes the STL-compliant structure's dtor -- set the proper
764 * allocator-context to ensure the inner data are disposed of properly first. */
765
766 const Segments_in_shm& serialization_segments = *m_btm_serialization_shm_handle; // Attn: const!
767 // OK! So now just do the usual SegmentArrayReader stuff as mentioned above (similarly to Heap_reader).
768
769 if (serialization_segments.empty())
770 {
771 FLOW_LOG_WARNING("shm::Reader [" << *this << "]: The top serialization was valid; and the SHM handle "
772 "therein does point to a list of segments; but that list is empty. Emitting error. "
773 "Other side misbehaved?");
774 return emit_error(error::Code::S_DESERIALIZE_FAILED_INSUFFICIENT_SEGMENTS);
775 }
776 // else
777
778 auto& capnp_segs = m_btm_capnp_segments;
779 assert(capnp_segs.empty());
780 capnp_segs.reserve(serialization_segments.size());
781
782 size_t idx = 0;
783 for (const auto& serialization_segment : serialization_segments) // Reminder: serialization_segment is a Basic_blob.
784 {
785 const uint8_t* data_ptr = &(serialization_segment.front());
786 const size_t seg_size = serialization_segment.size();
787
788 if ((uintptr_t(data_ptr) % sizeof(void*)) != 0)
789 {
790 FLOW_LOG_WARNING("shm::Reader [" << *this << "]: "
791 "Serialization segment [" << idx << "] "
792 "(0-based, of [" << serialization_segments.size() << "], 1-based): "
793 "SHM-heap buffer @[" << static_cast<const void*>(data_ptr) << "] sized [" << seg_size << "]: "
794 "Starting pointer is not this-architecture-word-aligned. Bug? "
795 "Misuse of Builder? Other side misbehaved? "
796 "Misalignment is against the API use requirements; capnp would complain and fail.");
797 return emit_error(error::Code::S_DESERIALIZE_FAILED_SEGMENT_MISALIGNED);
798 }
799 // else
800
801 FLOW_LOG_TRACE("shm::Reader [" << *this << "]: "
802 "Serialization segment [" << idx << "] "
803 "(0-based, of [" << serialization_segments.size() << "], 1-based): "
804 "SHM-heap buffer @[" << static_cast<const void*>(data_ptr) << "] sized [" << seg_size << "]: "
805 "Feeding into capnp deserialization engine.");
806 FLOW_LOG_DATA("Segment contents: "
807 "[\n" << buffers_dump_string(Blob_const(data_ptr, seg_size), " ") << "].");
808
809 capnp_segs.emplace_back(reinterpret_cast<const word*>(data_ptr),
810 seg_size / sizeof(word)); // @todo Maybe also check that seg_size = a multiple? assert()?
811
812 ++idx;
813 } // for (const auto& serialization_segment : serialization_segments)
814
815 /* Initialize the deserialization engine by giving it the pointers to/size of the backing segment blobs.
816 * It doesn't copy this array of pointers/sizes, so that array must stay alive, hence why capnp_segs is
817 * really m_capnp_segments. To be clear: not only must the blobs stay alive, but so must the array referring
818 * to them. */
819 const Capnp_word_array_array_ptr capnp_segs_ptr(&(capnp_segs.front()), capnp_segs.size());
820 m_btm_engine = make_unique<Capnp_heap_engine>(capnp_segs_ptr, RDR_OPTIONS);
821
822 assert(((!err_code) || (!*err_code)) && "*err_code should have been cleared above (unless null).");
823
824 // And lastly set up the structured-accessor API object that'll traverse those blobs via that engine.
825 return m_btm_engine->getRoot<Struct>();
826} // Reader::deserialization()
