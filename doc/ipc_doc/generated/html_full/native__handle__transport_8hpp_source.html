30/**
31 * A documentation-only *concept* defining the behavior of an object capable of reliably/in-order *sending* of
32 * discrete messages, each containing a native handle, a binary blob, or both. This is paired with
33 * the Native_handle_receiver concept which defines reception of such messages.
34 *
35 * ### Concept contents ###
36 * The concept defines the following behaviors/requirements.
37 * - The object has at least 2 states:
38 * - NULL: Object is not a peer: is not connected/capable of transmission. Essentially it is not a useful
39 * state. A default-cted object is in NULL state; and a moved-from object is as-if default-cted, therefore
40 * also in a NULL state. In this state all the transmission methods return `false` and no-op.
41 * - PEER: Object is or has been a peer (connected/capable of transmission). It is not possible to exit PEER
42 * state, except via being moved-from. A pipe-hosing error retains PEER state for instance.
43 * - Other states are allowed but outside the scope of the concept. For example a CONNECTING state may or may
44 * not be relevant (but again not relevant to the concept).
45 * - The (outgoing) transmission-of-messages methods, including transmission of graceful-close message.
46 * See their doc headers.
47 * - Behavior when the destructor is invoked. See ~Native_handle_sender()
48 * doc header.
49 * - Default ctor (which creates a NULL-state object that can be moved-to and to which "as-if"
50 * state any moved-from object is changed).
51 * - `sync_io`-core adopting ctor (which creates PEER-state object from an idle, as-if-just cted PEER-state
52 * sync_io::Native_handle_sender).
53 * - Move ctor, move assigment operator; these do the reasonable thing including setting the moved-from object
54 * to NULL state. Various satellite APIs (e.g., Native_socket_stream_acceptor) shall
55 * need these. That is such APIs do not rely on the factory/shared-ownership pattern.
56 *
57 * The concept (intentionally) does *not* define the following behaviors:
58 * - How to create a Native_handle_sender, except the default, `sync_io`-core-adopting, and move ctors.
59 * That is it does not define how to create a *new* (not moved-from) and *functioning* (PEER state: transmitting,
60 * connected) peer object, except the latter from an existing `sync_io`-pattern core. It only defines behavior
61 * once the user has access to a Native_handle_sender that is already in PEER state: connected to a opposing
62 * peer Native_handle_receiver.
63 *
64 * @see Native_socket_stream: as of this writing one key class that implements this concept (and also
65 * Native_handle_receiver) -- using the Unix domain socket transport.
66 * @see Channel, a pipe-composing class template that potentially implements this concept.
67 * @see Blob_sender: a degenerate version of the present concept: capable of transmitting only blobs, not
68 * `Native_handle`s.
69 *
70 * @todo In C++20, if/when we upgrade to that, Native_handle_sender (and other such doc-only classes) can become an
71 * actual concept formally implemented by class(es) that, today, implement it via the "honor system."
72 * Currently it is a class `#ifdef`-ed out from actual compilation but still participating in doc generation.
73 * Note that Doxygen (current version as of this writing: 1.9.3) claims to support doc generation from formal
74 * C++20 concepts.
75 *
76 * ### Rationale: Why is send_native_handle() not asynchronous? ###
77 * send_native_handle(), as noted in its doc header (and similarly Blob_sender::send_blob()) has 2 key properties:
78 * - It will *never* return would-block despite *always* being non-blocking.
79 * - It takes no completion handler; and it returns pipe-hosing errors synchronously via an out-arg or
80 * exception.
81 * - async_end_sending() does take a completion handler.
82 *
83 * That is, it is "somehow" both non-blocking/synchronous -- yet requires no would-block contingency on the user's
84 * part. Magical! How can this possibly work? After all we know that all known low-level transports will yield
85 * would-block eventually (as of this writing -- Unix domain sockets, POSIX MQs, bipc SHM-based MQs), if the
86 * opposing side is not popping data off the pipe. Yet, as we look at the opposing Native_handle_receiver concept
87 * we see that it makes no requirement on its impls to pop everything off the low-level transport in-pipe as soon
88 * as it arrives. (In actual fact all existing impls in Flow-IPC refuse to do any such thing.)
89 *
90 * Naturally the only answer as to how it can work is: Any given Native_handle_sender (and native Blob_sender) impl
91 * shall be ready to encounter would-block internally; if this occurs it will need to make a copy of the offending
92 * message (the meta-blob part specifically included -- that's the part that involves significant copying),
93 * enqueue it internally -- and keep enqueuing any further send_native_handle()d messages until the would-block
94 * clears, and the queue is eventually emptied.
95 *
96 * Fair enough: But isn't this a performance hazard? Answer: yes but only in a necessary way. We reached this
97 * decision after considering the entire system end-to-end:
98 * -# user prepares message blob B and tells `Native_handle_sender` S to send B;
99 * -# S tells low-level transport (possibly kernel) to accept B;
100 * -# `Native_handle_receiver` R pops B from low-level transport;
101 * -# user tells R it wanst to accept B, and R does give the latter to the user.
102 *
103 * It was a no-go to have Flow-IPC be in charge of allocating either-end buffer for B on the user's behalf:
104 * the user may want to user their own allocator, or the stack, or ??? -- possibly with zero-copy and prefix/postfix
105 * data nearby. The user buffer has to be ready, and S and R need to work with the memory areas provided by the
106 * user.
107 *
108 * Given all that, *one* or more of the following "people" has to be ready to keep B, or a copy of B, around, until
109 * the whole thing (1-4) has gone through: the sending user (1 above), S (2 above), or R (3 above). (If
110 * the receiving user (4 above) is always receiving things ASAP, without that process being processor-pegged or
111 * something, then it won't be a problem: no copying is necessary, except in step 2 when B is copied into
112 * the low-level process. Indeed that should be the case usually; we are discussing what to do *if* something goes
113 * wrong, or the receivier application is mis-coded, or who knows -- the receiver just isn't quick enough.)
114 *
115 * The answer, as already noted: *One* or more of 1, 2, 3 has to make a copy of B long enough until 4 has received it.
116 * At that point, in terms of overall performance, it *which* one of 1, 2, 3 it should be. Upon deliberating
117 * a few options occurred, but it seemed clear enough that 1 (original user) should not be the done, if it can be
118 * helped. IPC is not networking; as an end user I expect sending an out-message to work synchronously. Making
119 * me worry about a completion handler complicates the outgoing-direction API hugely -- and not in a way that
120 * makes the incoming-direction API any simpler, since that one always has to be ready for would-block
121 * (i.e., no in-messages immediately ready -- an async-receive API is required).
122 *
123 * So that left 2 or 3. I (ygoldfel) simply made the decision that one has to be chosen, and of those 2 is earlier
124 * and (internally for the concept implementer, also me at the time) more straightforward. If 3 were always
125 * reading items out of the low-level transport -- even when the user was not invoking `async_receive_*()` --
126 * then it would have to at least sometimes make copies of incoming data before the user chose to pop them out.
127 * By contrast, queueing it -- only on would-block -- in 2 would effectively make it a last-resort activity, as
128 * it should be.
129 *
130 * To summarize: Perfect opposing-receiver behavior cannot be guaranteed, and the low-level transport will eventually
131 * have a limit as to how much data it can buffer => in that case some entity must make a copy of the overflowing data
132 * => we want to make life easier for end user, so that entity should not be them => it must be either
133 * the `*_sender` or `*_receiver` => we chose `*_sender`, because it is safer and easier and faster and a last resort.
134 *
135 * One arguable drawback of this API design is that the Native_handle_sender (and Blob_sender) user won't be
136 * informed of an out-pipe-hosing error quite as soon as that object itself would know of it -- in some (rare)
137 * cases. I.e., if some data are queued up during would-block, and the user has stopped doing `send_*()`
138 * (as far as she is concerned, they're all working fine and have succeeded), and some timer later
139 * `*this` determines (in the background now) that the transport is hosed, the user would only find out about it
140 * upon the next `send_*()` call (it would *then* return the `Error_code` immediately), if any, or via
141 * async_end_sending(). Is this really a problem? Answer: Just... no. Exercise in reasoning this out (granted
142 * it is *somewhat* subjective) left to the reader. (Note we *could* supply some kind of out-of-band
143 * on-any-send-error API, and we considered this. It just was not worth it: inelegant, annoying. async_end_sending()
144 * with its completion handler is well sufficient.)
145 *
146 * It should be noted that this decision to make send_native_handle() synchronous/non-blocking/never-would-blocking
147 * has percolated through much of ipc::transport and even ipc::session: all our impls; Blob_sender and all its
148 * impls; Channel; and quite significantly struc::Channel. You'll note struc::Channel::send() is
149 * also synchronous/non-blocking/never-would-blocking, and even its implementation at the stage of actually
150 * sending binary data via Native_handle_sender/Blob_sender is quite straightforward. In Flow-IPC a
151 * `send*()` "just works."
152 */

163 /**
164 * Default ctor: Creates a peer object in NULL (neither connected nor connecting) state. In this state,
165 * all transmission-related methods (e.g., send_native_handle()) shall return `false` and otherwise no-op
166 * (aside from possible logging).
167 *
168 * This ctor is informally intended for the following uses:
169 * - A moved-from Native_handle_sender (i.e., the `src` arg for move-ctor and move-assignment operator)
170 * becomes as-if defaulted-constructed.
171 * - A target Native_handle_sender for a factory-like method (such as Native_socket_stream_acceptor::async_accept())
172 * shall typically be default-cted by the callin guse. (E.g.: Native_socket_stream_acceptor shall asynchronously
173 * move-assign a logger-apointed, nicely-nicknamed into that target `*this`, typically default-cted.)
174 *
175 * ### Informal corollary -- ctor that begins in PEER state ###
176 * Any functioning Native_handle_sender shall need at least one ctor that starts `*this` directly in PEER state.
177 * In that state the transmission-related methods (e.g., send_native_handle()) shall *not* return `false` and
178 * will in fact attempt transmission. However the form and function of such ctors is entirely dependent on
179 * the nature of the low-level transmission medium being encapsulated and is hence *intentionally* not a part
180 * of the concept. That said there is the `sync_io`-core-adopting ctor as well which *is* part of the concept.
181 *
182 * ### Informal corollary -- NULL-state ctors with 1+ args ###
183 * Other, 1+ arg, ctors that similarly create a NULL-state peer object are allowed/encouraged
184 * where relevant. In particular one taking a `Logger*` and a `nickname` string -- so as to subsequently enter a
185 * CONNECTING state via an `async_connect()` method, on the way to PEER state -- is reasonable.
186 * However this is not a formal part of this concept and is arguably not a general behavior.
187 * Such a ctor is informally intended for the following use at least:
188 * - One creates a Native_handle_sender that is `Logger`-appointed and nicely-`nickname`d; then one calls
189 * `async_connect()` on it in order to move it to CONNECTING and, hopefully, PEER states in that order.
190 * It will retain the logger and nickname (or whatever) throughout.
191 */
192 Native_handle_sender();

197 * That core object becomes as-if default-cted (therefore in NULL state).

215 /**
216 * Destroys this peer endpoint which will end the conceptual outgoing-direction pipe (in PEER state, and if it's
217 * still active) and cancels any pending completion handlers by invoking them ASAP with
218 * error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER.
219 * As of this writing these are the completion handlers that would therefore be called:
220 * - The handler passed to async_end_sending(), if it was not `.empty()` and has not yet been invoked.
221 * Since it is not valid to call async_end_sending() more than once, there is at most 1 of these.
222 *
223 * The pending completion handler will be called from an unspecified thread that is not the calling thread.
224 * Any associated captured state for that handler will be freed shortly after the handler returns.
225 *
226 * We informally but very strongly recommend that your completion handler immediately return if the `Error_code`
227 * passed to it is `error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER`. This is similar to what one should
228 * do when using boost.asio and receiving the conceptually identical `operation_aborted` error code to an
229 * `async_...()` completion handler. In both cases, this condition means, "we have decided to shut this thing down,
230 * so the completion handlers are simply being informed of this."
231 *
232 * In NULL state the dtor shall no-op.
233 *
234 * ### Thread safety ###
235 * Same as for the transmission methods. *Plus* it is not safe to call this from within a completion handler supplied
236 * to one of those methods on the same `*this`. An implication of the latter is as follows:
237 *
238 * Any user source code line that nullifies a `Ptr` (`shared_ptr`) handle to `*this` should be seen as potentially
239 * *synchoronously* invoking this dtor. (By nullification we mean `.reset()` and anything else that makes the
240 * `Ptr` null. For example destroying a `vector<Ptr>` that contains a `Ptr` pointing to `*this` = nullification.)
241 * Therefore, if a nullification statement can possibly make the ref-count reach 0, then the user must zealously
242 * protect that statement from running concurrently with another send/receive API call on `*this`. That is to say,
243 * informally: When ensuring thread safety with respect to concurrent access to a Native_handle_sender, don't forget
244 * that the destructor also is a write-type of access, and that nullifying a `Ptr` handle to it can synchronously
245 * invoke it.
246 *
247 * @see Native_handle_receiver::~Native_handle_receiver(): sister concept with essentially equal requirements.
248 * @see Native_socket_stream::~Native_socket_stream(): implements concept (also implements just-mentioned sister
249 * concept).
250 */
251 ~Native_handle_sender();

283 /**
284 * In PEER state: Synchronously, non-blockingly sends one discrete message, reliably/in-order, to the opposing peer;
285 * the message consists of the provided native handle (if supplied); or the provided binary blob (if supplied);
286 * or both (if both supplied). The opposing peer's paired Native_handle_receiver shall receive it reliably and
287 * in-order via Native_handle_receiver::async_receive_native_handle().
288 *
289 * If `*this` is not in PEER state (in particular if it is default-cted or moved-from), returns `false` immediately
290 * instead and otherwise no-ops (logging aside).
291 *
292 * Providing neither a handle nor a blob results in undefined behavior (assertion may trip).
293 * To *not* supply a handle, provide object with `.null() == true`. To *not* supply a blob, provide
294 * a buffer with `.size() == 0`.
295 *
296 * ### Blob copying behavior; synchronicity/blockingness guarantees ###
297 * If a blob is provided, it need only be valid until this method returns. The implementation shall, informally,
298 * strive to *not* save a copy of the blob (except into the low-level transport mechanism); but it is *allowed* to
299 * save it if required, such as if the low-level transport mechanism encounteres a would-block condition.
300 *
301 * This means, to the user, that this method is *both* non-blocking *and* synchronous *and* it must not
302 * refuse to send and return any conceptual would-block error (unlike, say, with a typical networked TCP socket API).
303 * Informally, the implementation must deal with any (typically quite rare) internal low-level would-block condition
304 * internally.
305 *
306 * However! Suppose send_native_handle() returns without error. This concept does *not* guarantee the message has
307 * been passed to the low-level transport at *this* point in time (informally it shall strive to make that
308 * happen in most cases however). A low-level would-block condition may cause it to be deferred. If one
309 * invokes ~Native_handle_sender() (dtor), the message may never be sent.
310 *
311 * However (part II)! It *is* guaranteed to have been passed to the transport once `*end_sending()`
312 * has been invoked, *and* its `on_done_func()` (if any) callback has been called. Therefore the expected use
313 * pattern is as follows:
314 * - Obtain a connected `Native_handle_sender` via factory.
315 * - Invoke send_native_handle() synchronously, hopefully without error.
316 * - (Repeat as needed for more such messages.)
317 * - Invoke `async_end_sending(F)` or `end_sending()` (to send graceful-close).
318 * - In `F()` destroy the `Native_handle_sender` (its dtor is invoked).
319 * - `F()` is invoked only once all queued messages, and the graceful-close, have been sent.
320 *
321 * ### Error semantics ###
322 * Firstly: See `flow::Error_code` docs for error reporting semantics (including the exception/code dichotomy).
323 *
324 * Secondly: If the generated #Error_code is error::Code::S_INVALID_ARGUMENT:
325 * - This refers to an invalid argument to *this* method invocation. This does *not* hose the pipe:
326 * further sends, etc., may be attempted with reasonable hope they will succeed.
327 * - send_meta_blob_max_size() is the limit for `meta_blob.size()`. Exceeding it leads to `S_INVALID_ARGUMENT`.
328 * - This interacts with the opposing Native_handle_receiver::async_receive_native_handle() in a subtle
329 * way. See "Blob underflow semantics" in that class concept's doc header.
330 * - At the impl's discretion, other conditions *may* lead to `S_INVALID_ARGUMENT`. If so they must
331 * be documented.
332 *
333 * Thirdly: All other non-success `Error_code`s generated should indicate the pipe is now indeed hosed; the user
334 * can count on the pipe being unusable from that point on. (It is recommended they cease to use the pipe
335 * and invoke the destructor to free resources.) If such a non-success code is emitted, the same one shall be emitted
336 * by all further send_native_handle() and async_end_sending() on `*this`.
337 *
338 * In particular error::Code::S_SENDS_FINISHED_CANNOT_SEND shall indicate that *this* method invocation
339 * occurred after invoking `*end_sending()` earlier.
340 *
341 * No would-block-like error condition shall be emitted.
342 *
343 * Lastly: The implementation is allowed to return a non-`INVALID_ARGUMENT` #Error_code in *this* invocation even
344 * though the true cause is a *previous* invocation of send_native_handle().
345 * Informally this may occur (probably in rare circumstances) if internally the handling of a previous
346 * call had to be deferred due to a low-level would-block condition; in that case the resulting problem is to be
347 * reported opportunistically with the next call. This allows for the non-blocking/synchronous semantics without
348 * the need to make send_native_handle() asynchronous.
349 *
350 * ### Thread safety ###
351 * You may call this from any thread. It is *not* required to be safe to call concurrently with
352 * any Native_handle_sender API, including this one and the destructor, invoked on `*this`.
353 *
354 * @param hndl_or_null
355 * A native handle (a/k/a FD) to send; or none if `hndl_or_null.null() == true`.
356 * @param meta_blob
357 * A binary blob to send; or none if `meta_blob.size() == 0`.
358 * @param err_code
359 * See above.
360 * @return `false` if `*this` is not in PEER (connected, transmitting) state;
361 * otherwise `true`.
362 */
363 bool send_native_handle(Native_handle hndl_or_null, const util::Blob_const& meta_blob, Error_code* err_code = 0);

365 /**
366 * Equivalent to send_native_handle() but sends a graceful-close message instead of the usual payload; the opposing
367 * peer's Native_handle_receiver shall receive it reliably and in-order via
368 * Native_handle_receiver::async_receive_native_handle() in the form of
369 * #Error_code = error::Code::S_RECEIVES_FINISHED_CANNOT_RECEIVE. If invoked after already invoking
370 * `*end_sending()`, the method shall no-op and return `false` and neither an exception nor truthy
371 * `*err_code`. Otherwise it shall return `true` -- but potentially emit a truthy #Error_code (if an error is
372 * detected).
373 *
374 * In addition: if `*this` is not in PEER state (in particular if it is default-cted or moved-from), returns
375 * `false` immediately instead and otherwise no-ops (logging aside).
376 *
377 * Informally one should think of async_end_sending() as just another message, which is queued after
378 * preceding ones; but: it is the *last* message to be queued by definition. It's like an EOF or a TCP-FIN.
379 *
380 * ### Synchronicity/blockingness guarantees ###
381 * async_end_sending() is, unlike send_native_handle(), formally asynchronous. The reason for this,
382 * and the expected use pattern of async_end_sending() in the context of preceding send_native_handle()
383 * calls, is found in the doc header for Native_handle_sender::send_native_handle(). We omit further discussion here.
384 *
385 * ### Error semantics ###
386 * This section discusses the case where `true` is returned, meaning `*end_sending()` had not already
387 * been called.
388 *
389 * An #Error_code is generated and passed as the sole arg to `on_done_func()`.
390 * A falsy one indicates success. A truthy one indicates failure; but in particular:
391 * - error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER (destructor called, canceling all pending ops;
392 * spiritually identical to `boost::asio::error::operation_aborted`);
393 * - other arbitrary codes are possible, as with send_native_handle().
394 *
395 * Reminder: By the nature of `on_done_func()`, the pipe is finished regardless of whether it receives a
396 * success or non-success #Error_code.
397 *
398 * ### Thread safety ###
399 * The notes for send_native_handle() apply.
400 *
401 * @internal
402 * The semantic re. calling `*end_sending()` after already having called it and having that exclusively
403 * return `false` and do nothing was a judgment call. As of this writing there's a long-ish comment at the top of
404 * of `"sync_io::Native_socket_stream::Impl::*end_sending()"`" body discussing why I (ygoldfel) went that way.
405 * @endinternal
406 *
407 * @tparam Task_err
408 * A functor type with signature identical to `flow::async::Task_asio_err`.
409 * @param on_done_func
410 * See above. This shall be invoked from an unspecified thread that is not the calling thread.
411 * @return `false` if and only if *either* `*this` is not in PEER (connected, transmitting) state, *or*
412 * `*end_sending()` has already been called before (so this is a no-op, in both cases).
413 * Otherwise `true`.
414 */
415 template<typename Task_err>
416 bool async_end_sending(Task_err&& on_done_func);
417
418 /**
419 * Equivalent to `async_end_sending(F)` wherein `F()` does nothing.
420 *
421 * Informally: If one uses this overload, it is impossible to guarantee everything queued has actually been sent,
422 * as `on_done_func()` is the only mechanism for this. Most likely this is only useful for test or proof-of-concept
423 * code; production-level robustness typically requires one to ensure everything queued has been sent. Alternatively
424 * the user's application-level protocol may already guarantee the message exchange has completed (e.g., by receiving
425 * an acknowledgment message of some sort) -- but in that case one can simply not call `*end_sending()`
426 * at all.
427 *
428 * @return Same as in async_end_sending().
429 */
430 bool end_sending();

432 /**
433 * In PEER state: Irreversibly enables periodic auto-pinging of opposing receiver with low-level messages that
434 * are ignored except that they reset any idle timer as enabled via Native_handle_receiver::idle_timer_run()
435 * (or similar). Auto-pings, at a minimum, shall be sent (subject to internal low-level transport would-block
436 * conditions) at the following times:
437 * - as soon as possible after a successful auto_ping() call; and subsequently:
438 * - at such times as to ensure that the time between 2 adjacent sends of *any* message is no more than
439 * `period`, until `*end_sending()` (if any) or error (if any).
440 * To clarify the 2nd point: If user messages are being sent already, an auto-ping may not be required or may be
441 * delayed until the last-sent message plus `period`). The implementation shall strive not to wastefully
442 * add auto-ping traffic unnecessary to this goal but is not *required* to do so. However, per the 1st point,
443 * an auto-ping shall be sent near auto_ping() time to establish a baseline.
444 *
445 * If `*this` is not in PEER state (in particular if it is default-cted or moved-from), returns `false` immediately
446 * instead and otherwise no-ops (logging aside). If auto_ping() has already been called successfuly,
447 * subsequently it will return `false` and no-op (logging aside). If `*end_sending()` has been called succesfully,
448 * auto_ping() will return `false` and no-op (logging side).
449 *
450 * ### Behavior past `*end_sending()` ###
451 * As noted: auto_ping() returns `false` and no-ops if invoked after successful `*end_sending()`.
452 * Rationale: If `*end_sending()` has been called, then the receiver shall (assuming no other error) receive
453 * graceful-close (and hose the in-pipe as a result) as soon as possible, or it has already received it
454 * (and hosed the pipe). Therefore any potential "further" hosing of the pipe due to idle timer would be redundant
455 * and ignored anyway given the nature of the Native_handle_receiver (and similar) API.
456 *
457 * ### Thread safety ###
458 * The notes for send_native_handle() apply.
459 *
460 * @param period
461 * Pinging occurs so as to cause the opposing receiver to receive *a* message (whether auto-ping or user
462 * message) at least this frequently, subject to limitations of the low-level transport.
463 * The optional default is chosen by the impl to most reasonably work with the opposing `idle_timer_run()`
464 * to detect a zombified/overloaded peer.
465 * @return `false` if `*this` is not in PEER (connected, transmitting) state, or if already called successfully
466 * in `PEER` state, or if `*end_sending()` was called successfully in `PEER` state; otherwise `true`.
467 */
468 bool auto_ping(util::Fine_duration period = default_value);

471/**
472 * A documentation-only *concept* defining the behavior of an object capable of reliably/in-order *receiving* of
473 * discrete messages, each containing a native handle, a binary blob, or both. This is paired with
474 * the Native_handle_sender concept which defines sending of such messages.
475 *
476 * Concept contents
477 * ----------------
478 * The concept defines the following behaviors/requirements.
479 * - The object has at least 2 states, NULL and PEER. See notes in Native_handle_sender w/r/t this;
480 * they apply equally here.
481 * - The (incoming) transmission-of-messages methods, including reception of graceful-close message and
482 * cutting off any further receiving. See their doc headers.
483 * - Behavior when the destructor is invoked. See ~Native_handle_receiver() doc header.
484 * - Default ctor. See notes in Native_handle_sender w/r/t this; they apply equally here.
485 * - `sync_io`-core adopting ctor. Same deal.
486 * - Move ctor, move assigment operator. Same deal.
487 *
488 * The concept (intentionally) does *not* define the following behaviors:
489 * - How to create a Native_handle_receiver, except the default, `sync_io`-core-adopting, and move ctors.
490 * Notes for Native_handle_sender apply equally here.
491 *
492 * @see Native_socket_stream: as of this writing one key class that implements this concept (and also
493 * Native_handle_sender) -- using the Unix domain socket transport.
494 * @see Channel, a pipe-composing class template that potentially implements this concept.
495 * @see Blob_receiver: a degenerate version of the present concept: capable of transmitting only blobs, not
496 * `Native_handle`s.
497 *
498 * Blob underflow semantics
499 * ------------------------
500 * This potentially important subtlety is regarding the interplay between
501 * Native_handle_receiver::S_META_BLOB_UNDERFLOW_ALLOWED, async_receive_native_handle() `meta_blob.size()`,
502 * and Native_handle_sender::send_native_handle() `meta_blob.size()`.
503 *
504 * Consider a `*this` named `R` and an opposing `*_sender` named `S`. Note that `R.receive_meta_blob_max_size()
505 * == S.send_meta_blob_max_size()`; call this limit `L`. `L` shall always be in practice small enough to where
506 * allocating a buffer of this size as the receive target is reasonable; hence kilobytes at most, not megabytes.
507 * (In some cases, such as with `Blob_stream_mq_*`, it is configurable at runtime. In others, as with
508 * Native_socket_stream, it is a constant: sync_io::Native_socket_stream::S_MAX_META_BLOB_LENGTH.)
509 *
510 * As noted in that concept's docs, naturally, `S.send_native_handle()` shall yield immediate, non-pipe-hosing
511 * error::Code::S_INVALID_ARGUMENT, if `meta_blob.size() > L`. This is called overflow and is always enforced.
512 *
513 * Now consider `N = meta_blob.size()` for `R.async_receive_native_handle()`. This is the size of the target
514 * blob: how many bytes it *can* receive. How `L` is enforced by `R` is not uniform among this concept's impls:
515 * rather it depends on the compile-time constant Native_handle_receiver::S_META_BLOB_UNDERFLOW_ALLOWED.
516 *
517 * ### Semantics: `META_BLOB_UNDERFLOW_ALLOWED` is `false` ###
518 * Suppose it is `false` (as is the case for Blob_stream_mq_receiver). In that case `N < L` is *disallowed*:
519 * `R.async_receive_native_handle()` *shall always* fail with non-pipe-hosing error::Code::S_INVALID_ARGUMENT
520 * for that reason. Certainly `N` cannot be zero -- not even if the incoming message is expected to
521 * contain only a non-`.null() Native_handle`.
522 *
523 * @note Formally it's as written. Informally the rationale for allowing this restriction is that some low-level
524 * transports have it; and working around it would probably affect performance and is unnatural.
525 * In particular Persistent_mq_handle::try_receive() (MQ receive) shall immediately fail if a target buffer
526 * is supplied smaller than Persistent_mq_handle::max_msg_size(). For both POSIX (`man mq_receive`) and
527 * bipc (`boost::interprocess::message_queue::try_receive()` docs and source code) it is disallowed at the
528 * low-level API level.
529 *
530 * Thus since `N >= L` is guaranteed before any actual receipt attempt is made, it is *impossible* for
531 * a message to arrive such that its meta-blob's size would overflow the target blob (buffer) (of size `N`).
532 * Therefore error::Code::S_MESSAGE_SIZE_EXCEEDS_USER_STORAGE shall not be emitted.
533 *
534 * ### Semantics: `META_BLOB_UNDERFLOW_ALLOWED` is `true` ###
535 * Suppose it is `true` (as is the case for Native_socket_stream). In that case `N < L` is allowed, in the sense
536 * that `R.async_receive_native_handle()` *shall not* fail with non-pipe-hosing error::Code::S_INVALID_ARGUMENT
537 * for that reason. In particular, in that case, `N` can even be zero: this implies the incoming message *must*
538 * contain a non-`.null() Native_handle` (it is not allowed to send a message with no content at all).
539 * (In the case of degenerate concept Blob_sender, `N` must exceed zero, as it cannot transmit anything but
540 * blobs.)
541 *
542 * However suppose a message has arrived, and its meta-blob's size -- which cannot exceed `L` -- is larger than
543 * `N`. I.e., it would *underflow* the user-supplied target buffer (blob). In that case:
544 * - async_receive_native_handle() shall emit error::Code::S_MESSAGE_SIZE_EXCEEDS_USER_STORAGE.
545 * - This error shall be pipe-hosing (further receive attempts will yield the same error).
546 *
547 * Informal tips: Any blob-size error can be completely avoided by always supplying target blob with `N == L`.
548 * (`N > L` will also work but may be considered wasteful, all else being equal.) In addition this tactic will
549 * ensure the code will generically work with a different Native_handle_receiver impl for which
550 * `S_META_BLOB_UNDERFLOW_ALLOWED == false`. Alternatively, the user protocol may be such that you simply know
551 * that `S` will never send messages beyond some limit (smaller than `L`), perhaps for particular messages.
552 * In that case `N` can be set to this smaller-than-`L` value. However -- such code will fail
553 * if the concept impl is later to switched to one with `S_META_BLOB_UNDERFLOW_ALLOWED == true`. This may or
554 * may not be an issue depending on your future dev plans.
555 *
556 * ### Summary ###
557 * This dichotomy of possible semantics is a conscious choice. Mode `false` is a necessity: some low-level transports
558 * enforce it, period. Mode `true` is to support a possible algorithmic desire to allocate smaller-than-`L`
559 * target buffers. The problem is some low-level transports allow this; but others don't. We did not want to
560 * impose one type's limitations on users of the other type. Therefore, if your code is agnostic to the type
561 * of transport, then code as-if `S_META_BLOB_UNDERFLOW_ALLOWED == false`. Otherwise you may code otherwise.
562 *
563 * Rationale: Why no `end_receiving()`, given opposing concept has `*end_sending()`?
564 * ---------------------------------------------------------------------------------
565 * An early version of Native_handle_receiver and Blob_receiver did have an `end_receiving()`.
566 * The idea was it would have 2 effects (plus 1 bonus):
567 * -# Any ongoing `async_receive_*()`s would be interrupted with in-pipe-hosing error
568 * error::Code::S_RECEIVES_FINISHED_CANNOT_RECEIVE -- the same one as when receing a graceful-close
569 * from opposing `*end_sending()`.
570 * -# There would be no further reading from the low-level transport for any reason.
571 * -# There was also the *thought*, but not the action, that an impl could inform the *opposing sender* of the
572 * pipe being closed by the reader. This would be conceptually similar to SIGPIPE in POSIX. In practice, of
573 * the available low-level transports as of this writing, only Native_socket_stream could actually implement it,
574 * as the otherwise-full-duplex independent pipes do live in this same peer (Unix domain socket), so the out-pipe
575 * could be used to inform the other guy that *his* out-pipe (our in-pipe) is now pointless. `Blob_stream_mq_*`
576 * are fully independent, operating on separate MQ kernel objects, so it was not realistically possible in
577 * that case.
578 *
579 * In order: (1) is fairly dubious: the local user should know when they've stopped receiving and don't need to
580 * use this mechanism to inform themselves. (3), as stated, sounded like a possible future improvement, but time
581 * told us it was not going to work out: if only a 2-pipe transport can do it in any case, organizing a concept
582 * around such a thing is just complicated. If necessary the user can (and we think likely will) just arrange
583 * their own protocol. So that leaves (2).
584 *
585 * To begin with -- in reality user protocols tend to be designed to not need such measures. If the reader is no
586 * longer interested, writer will probably know that. This is not networking after all: it is IPC.
587 * That aside, though, it only makes sense in any case if an impl *chooses* to greedily cache all low-level
588 * in-traffic in user RAM, even while no `async_receive_*()`s are pending to read it. Early on we actually did do
589 * that; but with the move to `sync_io`-pattern cores, and after contemplating
590 * "Rationale: Why is send-native-handle not asynchronous?" (Native_handle_sender concept doc header),
591 * we got rid of this waste of compute, letting unwanted in-traffic accumulate within a sender peer object
592 * (Native_handle_sender, Blob_sender) only. So in practice:
593 *
594 * If `end_receiving()` does not signal anything of value ((1) and (3)), and does not actually prevent any
595 * low-level reading that would otherwise occur, then it is a pointless complication. So that is why we got rid of it.
596 */

701 /**
702 * In PEER state: Asynchronously awaits one discrete message -- as sent by the opposing peer via
703 * Native_handle_sender::send_native_handle() or `"Native_handle_sender::*end_sending()"` -- and
704 * receives it into the given target locations, reliably and in-order. The message is, therefore, one of the
705 * following:
706 * - A binary blob; a native handle; or both. This is indicated by `on_done_func(Error_code(), N)`.
707 * The falsy code indicates success; `N <= target_meta_blob.size()` indicates the number of bytes received into
708 * `target_meta_blob.data()` (zero means no blob was sent in the message). `*target_hndl` is set
709 * (`target_hndl->null() == true` means no handle was sent in the message).
710 * - Graceful-close. This is indicated by `on_done_func(error::code::S_RECEIVES_FINISHED_CANNOT_RECEIVE, 0)`;
711 * neither the target blob nor target native handle are touched.
712 *
713 * If `*this` is not in PEER state (in particular if it is default-cted or moved-from), returns `false` immediately
714 * instead and otherwise no-ops (logging aside).
715 *
716 * ### Blob copying behavior; synchronicity/blockingness guarantees ###
717 * `*target_hndl` and the area described by `target_meta_blob` must both remain valid until `on_done_func()`
718 * executes. The method itself shall be non-blocking.
719 *
720 * The implementation shall, informally, strive to *not* copy the received blob (if any) into
721 * `target_meta_blob.data()...` except from the low-level transport mechanism. That is: it shall strive to not
722 * store the blob data in some internal buffer before ending up in `target_meta_blob.data()...`.
723 *
724 * ### Error semantics ###
725 * An #Error_code is generated and passed as the 1st arg to `on_done_func()`.
726 * A falsy one indicates success. A truthy one indicates failure; but in particular:
727 * - error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER (destructor called, canceling all pending ops;
728 * spiritually identical to `boost::asio::error::operation_aborted`);
729 * - only if #S_META_BLOB_UNDERFLOW_ALLOWED is `true`:
730 * - error::Code::S_MESSAGE_SIZE_EXCEEDS_USER_STORAGE (opposing peer has sent a message with a meta-blob exceeding
731 * `target_meta_blob.size()` in length; in particular one can give an empty buffer if no meta-blob expected);
732 * - error::Code::S_RECEIVES_FINISHED_CANNOT_RECEIVE (peer gracefully closed pipe via `*end_sending()`);
733 * - error::Code::S_RECEIVER_IDLE_TIMEOUT (idle timeout: see idle_timer_run());
734 * - other arbitrary codes are possible.
735 * - No would-block-like error condition shall be emitted.
736 *
737 * A non-success on-done invocation means the incoming pipe is hosed. (It is recommended they cease to use the pipe
738 * and invoke the destructor to free resources.) If such a non-success code is emitted, the same one shall be emitted
739 * by all further async_receive_native_handle() calls on `*this`. Key exception:
740 * - error::Code::S_INVALID_ARGUMENT:
741 * - This refers to an invalid argument to *this* method invocation.
742 * - If and only if #S_META_BLOB_UNDERFLOW_ALLOWED is `false`: A length floor on the target blob is imposed.
743 * Hence if `target_meta_blob.size() < receive_meta_blob_max_size()` then `S_INVALID_ARGUMENT` is emitted.
744 * - This shall *not* hose the pipe. Subsequent async-receives may be attempted with reasonable hope of
745 * success.
746 * - At the impl's discretion, other conditions *may* lead to `S_INVALID_ARGUMENT`. If so they must
747 * be documented.
748 *
749 * ### Thread safety ###
750 * You may call this either from any thread including the unspecified thread running within another call's
751 * `on_done_func()`. It is *not* required to be safe to call concurrently with any Native_handle_receiver API,
752 * including this one and the destructor, invoked on `*this`.
753 *
754 * @tparam Task_err_sz
755 * A functor type with signature identical to `flow::async::Task_asio_err_sz`.
756 * @param target_hndl
757 * `*target_hndl` shall be set by the time `on_done_func()` is executed with a falsy code. See above.
758 * @param target_meta_blob
759 * `target_meta_blob.data()...` shall be written to by the time `on_done_func()` is executed with a falsy
760 * code, bytes numbering `N`, where `N` is passed to that callback. `N` shall not exceed
761 * `target_meta_blob.size()`. See above.
762 * @param on_done_func
763 * See above. This shall be invoked from an unspecified thread that is not the calling thread.
764 * @return `false` if `*this` is not in PEER (connected, transmitting) state;
765 * otherwise `true`.
766 */

768 bool async_receive_native_handle(Native_handle* target_hndl,
769 const util::Blob_mutable& target_meta_blob,
770 Task_err_sz&& on_done_func);

772 /**
773 * In PEER state: Irreversibly enables a conceptual idle timer whose potential side effect is, once at least
774 * the specified time has passed since the last received low-level traffic (or this call, whichever most
775 * recently occurred), to emit the pipe-hosing error error::Code::S_RECEIVER_IDLE_TIMEOUT. The implementation
776 * shall guarantee the observer idle timeout is at least the provided value but may exceed this value for
777 * internal reasons, as long as it's by a less than human-perceptible period (roughly speaking -- milliseconds,
778 * not seconds).
779 *
780 * If `*this` is not in PEER state (in particular if it is default-cted or moved-from), returns `false` immediately
781 * instead and otherwise no-ops (logging aside). If idle_timer_run() has already been called successfuly,
782 * subsequently it will return `false` and no-op (logging aside).
783 *
784 * ### Important: Relationship between idle_timer_run() and async_receive_native_handle() ###
785 * idle_timer_run() is optional: if you have not called it, then the following does not apply. If you *have* called
786 * it:
787 *
788 * It will work usefully if and only if subsequently an async_receive_native_handle() is oustanding at least
789 * once each `timeout`. Informally this means it's best to immediately issue async_receive_native_handle() --
790 * unless one is already oustanding -- and then as soon as that completes (sans error) issue at least one more;
791 * and so on. Why? What are we talking about? Simple: async_receive_native_handle() impl is *not* required
792 * to read any more data from the low-level transport than is sufficient to satisfy the current
793 * async_receive_native_handle() deficit. (If it were required to do so, it would've been required to store copies
794 * of incoming meta-blobs when there's no user-request deficit; we've consciously avoided unnecessary copying.)
795 * So while there's no async_receive_native_handle() outstanding, `*this` will be considered idle; and once
796 * that continues long enough for `timeout` to be exceeded, the idle timer will hose the in-pipe.
797 *
798 * So: If you plan to use idle_timer_run(), then you need to be ~always async-receiving. Otherwise you'll risk
799 * hitting idle-timeout, even as the other side diligently sends stuff (`auto_ping()` or otherwise).
800 *
801 * ### Error semantics ###
802 * If and only if the timeout does occur down the line, the aforementioned error will be emitted via
803 * async_receive_native_handle() (or similar) handler. It shall be treated as the reason to hose the pipe
804 * (assuming it was not hosed by something else earlier).
805 *
806 * ### Thread safety ###
807 * You may call this either from any thread including the unspecified thread running within another call's
808 * `on_done_func()`. It is *not* required to be safe to call concurrently with any API, including this one
809 * and the destructor, invoked on `*this`.
810 *
811 * ### Suggested use ###
812 * Informally: There are roughly two approaches to using idle_timer_run().
813 *
814 * Firstly it can be used to gauge the state of the opposing process; if no auto-pings are arriving regularly,
815 * then the opposing Native_handle_sender (or similar) must be zombified or overloaded. To use it in this
816 * capacity, typically one must use Native_handle_sender::auto_ping() (or similar) on the opposing side.
817 * Typically one would then leave `timeout` at its suggested default value.
818 *
819 * Secondly it can be used to more generally ensure some application algorithm on the opposing side (presumably
820 * cooperating with the algorithm on the local side) is sending messages with expected frequency.
821 * That is, one would not use `auto_ping()` but rather send their own messages in a way that makes sense for
822 * the applications' algorithm.
823 *
824 * @param timeout
825 * The idle timeout to observe.
826 * The optional default is chosen by the impl to most reasonably work with the opposing `auto_ping()`
827 * to detect a zombified/overloaded peer.
828 * @return `false` if `*this` is not in PEER (connected, transmitting) state, or if already called successfully
829 * in `PEER` state; otherwise `true`.
830 */
831 bool idle_timer_run(util::Fine_duration timeout = default_value);
832}; // class Native_handle_receiver
