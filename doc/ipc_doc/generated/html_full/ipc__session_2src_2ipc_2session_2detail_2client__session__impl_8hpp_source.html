34/**
35 * Internal, non-movable pImpl-lite implementation of Client_session_mv class template.
36 * In and of itself it would have been directly and publicly usable; however Client_session_mv adds move semantics.
37 *
38 * @see All discussion of the public API is in Client_session_mv doc header; that class template forwards to this
39 * one. All discussion of pImpl-lite-related notions is also there. See that doc header first please. Then come
40 * back here.
41 *
42 * Impl design
43 * -----------
44 * Thread U represents all threads other than thread W: since the relevant methods are to be called by the user
45 * sans concurrency, those threads as a collection can be thought of as one thread.
46 *
47 * Thread W is the async worker thread where most work is done; this helps us entirely avoid mutexes.
48 * Both async_connect() and open_channel() are fairly infrequently called, so it's not necessary to avoid
49 * latency by doing work concurrently in threads U and W. So we keep it simple by posting most stuff onto thread W.
50 *
51 * There are two distinct states (other than NULL): CONNECTING (async_connect() outstanding) and PEER (it succeeded:
52 * `*this` is now a Session in PEER state per concept).
53 *
54 * ### CONNECTING state impl ###
55 * Once async_connect() is issued we must do the following:
56 * -# Synchronously find out the full `Shared_name` where the opposing Session_server is allegedly listening,
57 * which is computable by a #Base helper. So that it can be computed we need to find out
58 * Session_base::srv_namespace() which we immediately do by checking the expected file that should have been
59 * written by Session_server.
60 * -# Issue the Native_socket_stream::async_connect() to that abstract-address and await success asynchronously.
61 * -# Once that's done, go back to NULL on failure; or continue as follows on success: The Native_socket_stream,
62 * now in PEER state (connected), is upgraded to (wrapped by) a transport::Channel which is in turn upgraded to
63 * a transport::struc::Channel Client_session_impl::m_master_channel.
64 * This is the session master channel, in log-in phase (as client)
65 * (details of that API are in transport::struc::Channel docs). So we immediately issue the next (and final)
66 * async step: sending the log-in requset over `m_master_channel` and async-await log-in response.
67 * -# Once that's done, go back to NULL on failure; or else yay: we're now in PEER state.
68 *
69 * See session_master_channel.capnp for the schema for `m_master_channel`. Reading that gives a nice overview of
70 * the protocol involved, both in CONNECTING (after the socket is connected) and in PEER. Also notable is that
71 * this setup allows us to immediately "eat our own dog food" by using a `struc::Channel` to enable the opening
72 * of any further channels (structured or otherwise).
73 *
74 * ### Protocol negotiation ###
75 * First please see transport::struc::sync_io::Channel doc header section of the same name. (There is involved
76 * discussion there which we definitely don't want to repeat.) Then come back here.
77 *
78 * How we're similar: `Channel` decided to divide the protocols into 2: one in struc::sync_io::Channel code itself,
79 * and the rest in whatever extra stuff might be involved in the (likely SHM-related, optionally used)
80 * non-vanilla transport::struc::Struct_builder and transport::struc::Struct_reader concept impls (namely
81 * most likely transport::struc::shm::Builder and transport::struc::shm::Reader). So essentially there's the
82 * `ipc_transport_structured` part, and the (other) part (likely but not necessarily from `ipc_shm`). We do
83 * something vaguely similar conceptually:
84 * - There is the basic ipc::session protocol, including the `LogInReq/LogInRsp/OpenChannel*` messages. This
85 * is from ipc::session proper (`ipc_session` module/library).
86 * - There is the optionally-used protocol woven into it activated by the fact the user chose to use
87 * ipc::session::shm (or maybe even their own) `Session` hierarchy. (As of this writing, ipc::session::shm
88 * contains two hierarchies, depending on the SHM-provider of choice: SHM-classic `shm::classic` or
89 * SHM-jemalloc `shm::arena_lend::jemalloc`.)
90 *
91 * So we, too, maintain two `Protocol_negotiator`s: the main one and the "aux" one for those two bullet points
92 * respectively. As of this writing there is only version 1 of everything; but if this changes for one of the
93 * "aux" `Session` hierarchies -- e.g., SHM-jemalloc's internally used ipc_session_message.capnp channel/schema --
94 * then that guy's "aux" protocol-version can be bumped up. (In the case of ipc_session_message.capnp specifically,
95 * and I say this only for exposition, that code can -- if it wants -- do its own protocol negotiation along its
96 * own channel; but by leveraging the "aux" `Protocol_negotiator` version infrastructure, they do not have to.
97 * Good for perf and, in this case more importantly, good for keeping code complexity down.)
98 *
99 * How we're different: The mechanics of protocol-version exchange are simpler for us, because we have a built-in
100 * SYN/SYN-ACK like exchange at the start (log-in-request, log-in-response) via the session master channel.
101 * So we piggy-back (piggy-front?) the protocol-negotiation info onto those 2 messages which exist anyway.
102 * You can see that now by looking at session_master_channel.capnp and noting how `ProtocolNegotiation` fields
103 * sit up-top in `LogInReq` and `LogInRsp`. As Protocol_version doc header discussion suggests, it is important
104 * to verify the `ProtocolNegotiation` fields before interpreting any other part of the incoming `LogInReq` and
105 * `LogInRsp` by server and client respectively (and in that chronological order, as guaranteed by the handshake
106 * setup).
107 *
108 * (In contrast to this: struc::sync_io::Channel does not itself have a mandatory built-in handshake phase. Hence
109 * each side has to send a special `ProtocolNegotiation` message ASAP and expect one -- in each pipe of up-to-2 --
110 * to come in as well. No handshake/client/server => the two exchanges are full-duplex, that is mutually
111 * independent, in nature.)
112 *
113 * If you're keeping score: as of this writing, the most complex/full-featured setup for a session is
114 * the SHM-jemalloc session setup. Excluding any structured channels opened within the session, the following
115 * transport::Protocol_negotiator negotiations shall occur:
116 * - Session master channel `Native_socket_stream` low-level protocol (1 `Protocol_negotiator` per side).
117 * Full-duplex exchange.
118 * - Session master channel `struc::Channel` protocol (2 `Protocol_negotiator`s per side, "aux" one unused
119 * because the session master channel is a vanilla, non-SHM-backed channel).
120 * Full-duplex exchange.
121 * - Session master channel vanilla session protocol (1 `Protocol_negotiator` per side).
122 * Handshake exchange.
123 * - Session master channel SHM-jemalloc session protocol (1 `Protocol_negotiator` per side).
124 * Handshake exchange (same one as in the previous bullet point).
125 *
126 * Unrelated to sessions, consider the most complex/full-featured setup for a user `transport::struc::Channel` operating
127 * over a 2-pipe `transport::Channel` (1 pipe for blobs, 1 pipe for `Native_handle`s):
128 * - User channel `Native_socket_stream` low-level protocol (1 `Protocol_negotiator` per side).
129 * Full-duplex exchange (pipe 1).
130 * - User channel `Blob_stream_mq_*er` low-level protocol (1 `Protocol_negotiator` per side).
131 * Full-duplex exchange (pipe 2, same layer as preceding bullet point, different transport type).
132 * - Session master channel `struc::Channel` protocol (2 `Protocol_negotiator`s per side; main one for the vanilla
133 * layer, "aux" one for the SHM layer).
134 * Full-duplex exchange.
135 *
136 * ### PEER state impl ###
137 * Here our algorithm is complementary to the PEER state algorithm in Server_session_impl. Namely we expect
138 * passive-opens via an appropriate transport::struc::Channel::expect_msgs(); and we allow active-opens via
139 * open_channel(). The details of how these work is best understood just by reading that code inline.
140 *
141 * open_channel(), similarly to Server_session_impl::open_channel(), has one interesting quirk which is:
142 * To the user it is presented as synchronous and blocking with the aim of being non-blockingly fast, as long
143 * as the other side is in PEER state. It uses transport::struc::Channel::sync_request() with a generous timeout, while
144 * knowing that if both sides in PEER state in practice the call will complete rather swiftly; the generous
145 * timeout aimed making it clear that open_channel() is failing *not* due to some overly tight internal timeout
146 * but more likely due to some application problem such as not calling Server_session::init_handlers() quickly
147 * enough.
148 *
149 * Once PEER state is reached, the Session-concept error emission semantics come into play. See its doc header.
150 *

153 * @tparam S_TRANSMIT_NATIVE_HANDLES

155 * @tparam Mdt_payload
156 * See #Client_session counterpart.
157 * @tparam S_SHM_TYPE_OR_NONE
158 * Identical to opposing Server_session_impl counterpart.
159 */

352 /**
353 * Utility for sub-classes: executed from async_connect()'s `on_done_func(Error_code())` (i.e., directly
354 * from handler invoked on success) -- hence from thread W -- this instead goes back to CONNECTING state,
355 * essentially indicating "yes, the vanilla async_connect() succeeded, but the true user-called async_connect()

359 * As `on_done_func` was already invoked (and thus forgotten), the caller must supply a new
360 * one to execute once either PEER or NULL state is reached again. Since `*this` has completed its vanilla

363 * complete_async_connect_after_canceling_peer_state(). That said, if a master-session-channel error occurs before

365 * - `*this` will enter NULL state itself.

369 * either complete_async_connect_after_canceling_peer_state() or that channel becoming hosed (either way

371 * See also async_worker().

374 * See async_connect().
375 * @param on_done_func
376 * See async_connect().
377 * @return See above.
378 */

472 /**
473 * An open-channel/log-in request out-message. Together with that is stored a `Builder` into its metadata
474 * sub-field.
475 *
476 * ### Rationale ###
477 * This explanation is written for open_channel() case, but the same applies for the log-in request and
478 * async_connect().
479 *
480 * Why store the `Builder`, if it can always be obtained from #m_req_msg? Answer: On one hand,
481 * mdt_builder() needs to return that `Builder`, so that the user can set the metadata. However,
482 * open_channel() aims to then stuff the underlying bits into the open-channel request out-message which is
483 * is our internal impl detail, unknown to the user. We could return a stand-alone just-the-metadata structure
484 * and `Builder` to it, but (1) then we'd have to transfer that over into the out-message -- slow, possibly
485 * painful (involving capnp orphans), and (2) it has to be backed by a `MessageBuilder`. So we'd have to
486 * set up a `MallocMessageBuilder` or something just for this thing -- only to end up painfully copying over
487 * data from one serialization to another. Instead we maintain zero-copy and avoid these issues:
488 * - Set up the open-channel request out-message directly in mdt_builder(). (Its ref-count shall
489 * always be 1.)
490 * - Point a `Builder` to the metadata sub-field.
491 * - Wrap both things in this `struct` and wrap that in a `shared_ptr`.
492 * - Lastly have mdt_builder() return an *alias* `shared_ptr` that (to the user) looks like it's
493 * to just the `Builder` (all they care about), but it's actually in the `shared_ptr` group of the real
494 * ref-counted datum, this `struct`.
495 * - open_channel() receives the alias, which points to the `Builder`. Since the `Builder` is the first
496 * member of this `struct`, its raw address (by C++ aliasing rules) equals that of the whole `struct`.
497 * Hence open_channel() can get to the `struct` and thus to the out-message member #m_req_msg. It can
498 * then transport::struc::Channel::send() that.
499 */
500 struct Master_channel_req
501 {
502 // Data.
503
504 /**
505 * The `Builder` `m_req_msg->body_root()->getMetadata()`.
506 * @warning As of this writing this *must* be the first member. Otherwise one would need to use something like
507 * `offsetof` to be able to get from `&m_mdt_builder` (what mdt_builder() returns)
508 * to `this` and hence to #m_req_msg to `send()`.
509 */
510 typename Mdt_builder_ptr::element_type m_mdt_builder;
511
512 /// The open-channel-request-to-server out-message for the session master channel (in subsequent open_channel()).
513 typename Master_structured_channel::Msg_out m_req_msg;
514 };

869TEMPLATE_CLI_SESSION_IMPL
870void CLASS_CLI_SESSION_IMPL::dtor_async_worker_stop()
871{
872 using flow::async::Synchronicity;
873 using boost::promise;
874
875 // We are in thread U (from either our own dtor or subclass dtor).
876
877 assert((!m_async_worker.task_engine()->stopped()) && "This shall be called at most once.");
878
879 FLOW_LOG_INFO("Client session [" << *this << "]: Shutting down. Worker thread will be joined, but first "
880 "we perform session master channel end-sending op (flush outgoing data) until completion. "
881 "This is generally recommended at EOL of any struc::Channel, on error or otherwise. "
882 "This is technically blocking but unlikely to take long.");
883 /* @todo Consider adding blocking struc::Channel::sync_end_sending(), possibly with timeout. There might be
884 * a to-do in its or async_end_sending() class doc header. Timeout would help us avoid blocking here, unlikely though
885 * it is in practice probably. A sync_ call(), regardless of that, would just be convenient resulting in less
886 * code here. */
887
888 /* Let's do the async_end_sending() thing we just promised.
889 *
890 * So far everything is still operating; so we need to follow normal rules. m_master_channel access requires
891 * being in thread W. */
892 m_async_worker.post([&]()
893 {
894 // We are in thread W.
895
896 if (!m_master_channel)
897 {
898 return; // Nothing to worry about; async_connect() hasn't even made this yet (or was not invoked).
899 }
900 // else
901
902 promise<void> done_promise;
903 m_master_channel->async_end_sending([&](const Error_code&)
904 {
905 // We are in thread Wc (unspecified, really struc::Channel async callback thread).
906 FLOW_LOG_TRACE("Client session [" << *this << "]: Shutdown: master channel outgoing-direction flush finished.");
907 done_promise.set_value();
908 });
909 // Back here in thread W:
910 done_promise.get_future().wait();
911 }, Synchronicity::S_ASYNC_AND_AWAIT_CONCURRENT_COMPLETION); // m_async_worker.post()
912 // Back here in thread U: done; yay.
913
914 /* This (1) stop()s the Task_engine thus possibly preventing any more handlers from running at all (any handler
915 * possibly running now is the last one to run); (2) at that point Task_engine::run() exits, hence thread W exits;
916 * (3) joins thread W (waits for it to exit); (4) returns. That's a lot, but it's non-blocking. */
917 m_async_worker.stop();
918 // Thread W is (synchronously!) no more. We can now access whatever m_ state we want without concurrency concerns.
919} // Client_session_impl::dtor_async_worker_stop()

970template<typename Task_err>

973 Mdt_reader_ptr* mdt_from_srv_or_null,
974 Channels* init_channels_by_srv_req,
975 Task_err&& on_done_func)
976{
977 using util::process_id_t;
978 using flow::util::ostream_op_string;
979 using flow::async::Synchronicity;
980 using Named_sh_mutex = boost::interprocess::named_mutex;
981 using Named_sh_mutex_ptr = boost::movelib::unique_ptr<Named_sh_mutex>;
982 using Sh_lock_guard = boost::interprocess::scoped_lock<Named_sh_mutex>;
983 using boost::system::system_category;
984 using boost::lexical_cast;
985 using boost::bad_lexical_cast;
986 using boost::movelib::make_unique;
987 using boost::io::ios_all_saver;
988 using fs::ifstream;
989 using std::getline;
990 using std::string;
991 using std::to_string;
992 // using ::errno; // It's a macro apparently.
993
994 // We are in thread U.
995 assert((!m_async_worker.in_thread()) && "Do not call directly from its own completion handler.");
996
997 assert(reinterpret_cast<Master_channel_req*>(mdt.get())->m_req_msg.body_root()->isLogInReq()
998 && "Did you pass PEER-state-mdt_builder() result to async_connect()? Use NULL-state one instead. "
999 "Where did you even get it -- another Client_session? Bad boy.");
1000
1001 const auto do_func = [&]() -> bool // Little helper that lets us concisely use `return <bool>;`.
1002 {
1003 // We are in thread W.
1004
1005 if (m_state != State::S_NULL)
1006 {
1007 FLOW_LOG_WARNING("Client session [" << *this << "]: Session-connect request: Wanted to start "
1008 "but already not in NULL state. Ignoring.");
1009 return false;
1010 }

1018 /* Let's overview our task as the session-opening client. We are a client, so we open sessions. In fact
1019 * *this is expressly for opening 1 (one) session: if they want to open another one, they'd make another
1020 * Client_session (and moreover it would be an abuse of the concept to do so, to the same Server_app, while
1021 * *this exists, but I digress, and we don't care about that). How to open a session? That's isomorphic to
1022 * opening a *session master channel* (see session_master_channel.capnp now). Because it is the channel used
1023 * for opening user channels, it must be able to transmit native handles (see Native_socket_stream doc header,
1024 * regaring "way 1" and "way 2" of establishing an N_s_s connection; we're establishing here the thing that
1025 * will allow the preferred "way 2" to happen subsequently). So this channel will require at least a
1026 * Native_socket_stream (handles) pipe. Would it benefit from MQ pipes? Probably not; we fully control the
1027 * schema (no user schema involved yet), and it's fairly small messages only. So just a single-duplex-handles-pipe
1028 * channel is fine. Therefore, all we need to establish before we can load up our Channel<> peer object is
1029 * a Native_socket_stream peer object in PEER (connected) state. As mentioned before, there's way 1 and way 2
1030 * theoretically available; but (as noted) way 2 requires an existing channel for native-handle transmission
1031 * (chicken/egg), and we lack that. Hence way 1 is the only one available at the moment.
1032 *
1033 * So far, so good; way 1 involves an N_s_s_acceptor listening and doing async_accept()s and an
1034 * N_s_s cted in NULL state performing an async_connect(). We're the client, so we'll be doing the latter.
1035 * Way 1, sadly, means we'll need a name to which to connect. As discussed elsewhere, all shared resources
1036 * that are per-session are first segregated by Server_app::m_name (<=> distinct server application); then by
1037 * *server namespace* within that (<=> distinct server application *instance* a/k/a process <=> PID); and
1038 * then by session. Since we're creating the session now, we need the levels of naming above that.
1039 * We know the Server_app (<=> Server_app::m_name); so we needed the server namespace. This is stored in
1040 * the CNS (Current Namespace Store) which is a formal name for the damned PID file (<=> process instance).
1041 * So read the PID file to get the name. (Access to this must be synchronized against being written; details
1042 * below in that step.)
1043 *
1044 * Then we'll just async-connect to the name built off that PID (and the other stuff including Server_app::m_name).
1045 * As a result we'll have our Native_socket_stream peer object in PEER state. It is then trivial to wrap it
1046 * in a one-pipe Channel<>.
1047 *
1048 * The session master channel is a structured channel (struc::Channel<> peer object, wrapping the Channel<>,
1049 * is therefore what we want next). That's again simple.
1050 *
1051 * However, being the session master channel specifically, it has a log-in phase (otherwise optional: and in fact
1052 * this type of channel is the only one that has one). So we'll need to perform a quick exchange at the
1053 * start of the struc::Channel<>'s lifetime. If that's successful, we are done. */

1055 // So first step is to read the CNS (Current Namespace Store): the PID file.
1056
1057 /* PID file can be written-to by the server (that's the whole point), so no choice but to use locking.
1058 * Various techniques exist, including using the file itself as a lock, but we shall be using a bipc named mutex
1059 * (can be thought of as a tiny SHM pool storing just the single mutex -- semaphore in Linux).
1060 * Because it's so small and unique (per app), we won't worry about corruption and cleanup.
1061 * (@todo Reconsider that.) Why the separate mutex, instead of using file as a lock? Answer: in the
1062 * 1st place, it's arguably easier to use anyway -- the bipc file-lock mechanism is slick but shows the
1063 * slight oddness of the technique. Also, for future development: we may need to store other information, such
1064 * as info on kernel-persistent objects to clean in the even of a crash/abort, in a separate file but
1065 * accessed in the same critical section as the CNS (PID file). Hence a common lock would be required for
1066 * both CNS and these other resources; then it would be odd to use file A as a lock for files A, B, C....
1067 *
1068 * One subtlety *is* added due to the decision to use it: ownership and permissions on the mutex.
1069 * This is discussed in detail in Session_server_impl, where it similarly locks this mutex.
1070 * The bottom line is we may need to be the ones to create it, hence OPEN_OR_CREATE; and therefore
1071 * we need to specify permissions; and the permissions value is based on Server_app [sic] despite our being
1072 * the client. This is explained in the aforementioned spot in Session_server_impl. */

1074 string srv_namespace_mv;

1077 const auto cns_path = Base::cur_ns_store_absolute_path();

1080 FLOW_LOG_INFO("Client session [" << *this << "]: Session-connect request: Starting session-connector thread "
1081 "and connecting. Presumably the CNS (Current Namespace Store), a/k/a PID "
1082 "file [" << cns_path << "] (shared-mutex name [" << mutex_name << "], shared-mutex perms "
1083 "[" << std::setfill('0') << std::setw(4) << std::oct << mutex_perms.get_permissions() << "]), "
1084 "exists.");
1085 }

1087 Named_sh_mutex_ptr sh_mutex;
1088 util::op_with_possible_bipc_exception(get_logger(), &err_code, error::Code::S_MUTEX_BIPC_MISC_LIBRARY_ERROR,
1089 "Client_session_impl::async_connect()", [&]()
1090 {
1091 sh_mutex = make_unique<Named_sh_mutex>(util::OPEN_OR_CREATE, mutex_name.native_str(), mutex_perms);
1092 });

1095 {

1103 "file [" << cns_path << "]; system error details follow. Immediately giving "
1104 "up (non-blocking).");
1105 FLOW_ERROR_SYS_ERROR_LOG_WARNING(); // Log based on sys_err_code.
1106
1107 err_code = sys_err_code;
1108 }
1109 else // if (cns_file.good()) // #1
1110 {
1111 assert(!err_code);
1112
1113 /* Read up to a newline. Be forgiving about whatever is after that; but the newline is required.
1114 * If it's not there, srv_namespace_mv will be fine (and either way won't include the newline), but .good() will
1115 * be false. */
1116 getline(cns_file, srv_namespace_mv);
1117 if (!cns_file.good())
1118 {
1119 FLOW_LOG_WARNING("Client session [" << *this << "]: Session-connect request: Opened CNS (PID) file "
1120 "file [" << cns_path << "]; but couldn't read a full newline-terminated line. "
1121 "Immediately giving up (non-blocking).");
1122 err_code = error::Code::S_CLIENT_NAMESPACE_STORE_BAD_FORMAT;
1123 }
1124 // else if (cns_file.good()): Fall through. // #2
1125 } // else if (cns_file.good()) // #1
1126 // In any case can close file now and unlock.
1127 } // Sh_lock_guard sh_lock(*sh_mutex); // if (!err_code) (might have become truthy inside though)
1128 // else if (err_code) { It logged fine already. }
1129
1130 /* Might as well do this a bit early. Or no-op if null. By the way, it does not destroy the mutex itself,
1131 * just the handle to it. Destroying it (via Named_sh_mutex::remove()) would be counter to the whole point of it.
1132 * This does, sort of, leak a bit of RAM but only a bit. */
1133 sh_mutex.reset();
1134
1135 if (!err_code)
1136 {
1137 // Still OK so far. Parse all of srv_namespace_mv via lexical_cast<> (which uses `istream >> process_id_t`).

1139 try
1140 {
1141 pid = lexical_cast<process_id_t>(srv_namespace_mv);
1142 }

1144 {
1145 FLOW_LOG_WARNING("Client session [" << *this << "]: Session-connect request: Opened CNS (PID) file "
1146 "file [" << cns_path << "]; read a full newline-terminated line; but could not parse "
1147 "it cleanly to an integer PID. Immediately giving up (non-blocking). "
1148 "Exception detail: [" << exc.what() << "].");
1149 err_code = error::Code::S_CLIENT_NAMESPACE_STORE_BAD_FORMAT;
1150 }
1151
1152 if (!err_code)
1153 {
1154 // Might be paranoid but just in case some spaces or smth were in srv_namespace_mv: convert back to eliminate.
1155 srv_namespace_mv = to_string(pid);
1156 Base::set_srv_namespace(Shared_name::ct(std::move(srv_namespace_mv)));
1157 // srv_namespace_mv is now hosed.
1158 const auto acc_name = Base::session_master_socket_stream_acceptor_absolute_name();
1159
1160 // Pretty important value; INFO should be okay verbosity-wise.
1161 FLOW_LOG_INFO("Client session [" << *this << "]: Session-connect request: Opened CNS (PID) file "
1162 "file [" << cns_path << "]; obtained server-namespace [" << Base::srv_namespace() << "]. "
1163 "Can now initiate Native_socket_stream async-connect based on this name which is "
1164 "[" << acc_name << "].");
1165
1166 // Create NULL (unconnected) Native_socket_stream peer object.
1167 auto sock_stm = boost::make_shared<transport::Native_socket_stream>
1168 (get_logger(), ostream_op_string(*this, "->", Base::srv_namespace().str()));
1169 sock_stm->async_connect
1170 (acc_name,
1171 [this, acc_name, sock_stm = std::move(sock_stm),
1172 mdt, init_channels_by_cli_req_pre_sized, mdt_from_srv_or_null, init_channels_by_srv_req]
1173 (const Error_code& async_err_code) mutable
1174 {
1175 // We are in thread Wo (unspecified, really Native_socket_stream worker thread).
1176 if (async_err_code == transport::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER)
1177 {
1178 // Stuff is shutting down. GTFO. (Our dtor will emit session::...::..._ABORTED_COMPLETION_HANDLER.)
1179 return;
1180 }
1181 // else: We post all real work onto our thread W.
1182 m_async_worker.post([this, acc_name, sock_stm = std::move(sock_stm),
1183 mdt = std::move(mdt),
1184 init_channels_by_cli_req_pre_sized, mdt_from_srv_or_null, init_channels_by_srv_req,
1185 async_err_code]
1186 () mutable
1187 {
1188 // We are in thread W.
1189 conn_on_async_connect_or_error(acc_name, std::move(*sock_stm),
1190 std::move(mdt),
1191 init_channels_by_cli_req_pre_sized,
1192 mdt_from_srv_or_null,
1193 init_channels_by_srv_req,
1194 async_err_code);
1195 });
1196 }); // sock_stm->async_connect()
1197 return true; // Do *not* fall through.
1198 } // if (!err_code)
1199 // else if (err_code): Fall through.
1200 } // if (!err_code) (might have become truthy inside though)
1201
1202 assert(err_code && "All code paths except a single success one (above) must have led to a truthy Error_code.");
1203
1204 // Can emit.
1205 assert((m_state == State::S_CONNECTING)
1206 && "Only way to exit CONNECTING state is past NSS async-connect handler which we didn't initiate.");
1207 assert((!m_conn_on_done_func_or_empty.empty())
1208 && "m_conn_on_done_func_or_empty can only be emptied upon executing it.");
1209
1210 m_state = State::S_NULL;
1211
1212 FLOW_LOG_TRACE("Client session [" << *this << "]: Session-connect request: Reporting connect failed "
1213 "(it was right inside async_connect(), before actual socket-stream connect). Executing handler.");
1214 invoke_conn_on_done_func(err_code);
1215
1216 return true;
1217 }; // const auto do_func =
1218
1219 // We post all real work onto our thread W.
1220
1221 bool result;
1222 m_async_worker.post([&]() { result = do_func(); },
1223 Synchronicity::S_ASYNC_AND_AWAIT_CONCURRENT_COMPLETION);
1224
1225 return result;
1226} // Client_session_impl::async_connect()

1637 (typename Master_structured_channel::Msg_in_ptr&& open_channel_msg,
