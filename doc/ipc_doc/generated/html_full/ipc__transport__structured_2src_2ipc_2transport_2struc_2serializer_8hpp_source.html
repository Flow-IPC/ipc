37/**
38 * A documentation-only *concept* defining the behavior of an object capable of zero-copy-serializing,
39 * similar to `capnp::MessageBuilder` but geared to transmission over pipe-like IPC transports.
40 * At its simplest it is conceptually identical to `MessageBuilder` but, crucially, also allows for the (optional)
41 * 2-tier architecture wherein only a small handle is transmitted over an IPC pipe, while the user-controlled
42 * tree is stored in a single central location (most likely in shared memory) and is itself never copied.
43 * The following discussion assumes sophisticated familiarity with capnp (Cap'n Proto).
44 *
45 * @see Struct_reader, the counterpart concept.
46 *
47 * The above description may sound quite a lot like the capnp-supplied `virtual` interface,
48 * `capnp::MessageBuilder`. And some of this concept's most prominent impls have conceptual counterparts
49 * in the implementing classes of `MessageBuilder`; for example Heap_fixed_builder
50 * evokes `capnp::MallocMessageBuilder` (and in fact internally uses a custom `MessageBuilder` quite
51 * similar to `MallocMessageBuilder`).
52 *
53 * Indeed, at its simplest, Struct_builder is conceptually similar to `capnp::MessageBuilder`:
54 * - It must store a concretely-typed `MessageBuilder` that the user would mutate via
55 * payload_msg_builder(), which returns a pointer to said `MessageBuilder`. One would typically
56 * mutate it via `->initRoot<SomeUserSchema>()` and so on.
57 * Through this one can set the structured data within the data structure.
58 * - As one does so, the payload bits are set (by capnp-generated code) in zero-copy fashion inside 1+
59 * segments somewhere in RAM (or other vaddr-mapped storage), without any user intervention outside of the
60 * actual capnp mutator calls.
61 * - One can then obtain the resulting serialization -- consisting of 1+ segments in RAM.
62 * `MessageBuilder::getSegmentsForOutput()` is how it's done in `MessageBuilder`;
63 * while we have emit_serialization().
64 *
65 * Ultimately, in fact, any Struct_builder is always implemented internally as a wrapper around
66 * one `MessageBuilder` or another (quite possibly a custom one also implemented by us).
67 *
68 * So why does Struct_builder exist, outside of cosmetic niceties? Answer: It is designed around (the possiblity of)
69 * two-tier serialization (discussed in "Copying versus zero-copy" below), namely for SHM-based IPC.
70 * `MessageBuilder`, alone, does not supply the necessary facilities to make the SHM/two-tier use-case easy;
71 * we do. A good way of looking at it is: Struct_builder allows one to combine two `capnp::MessageBuilder`s,
72 * using one for user-facing mutation pre-transmission-over-IPC; and the other `MessageBuilder` for the tiny
73 * SHM-handle that's copied over at transmission-over-IPC time. payload_msg_builder() provides access to the
74 * user-mutating; emit_serialization() provides access to the IPC-transmissible bits. When the serialization
75 * strategy is 1-tier, Struct_builder is essentially isomorphic to `MessageBuilder`. When it is 2-tier,
76 * it adds significant conceptual value.
77 *
78 * In any case a Struct_builder formally works as follows.
79 * - Construct it via `Struct_builder(const Struct_builder::Config&)` ctor form, where Struct_builder::Config
80 * is a small (cheaply-copyable) type storing a few knobs controlling Struct_builder behavior.
81 * (It exists to be able to easily store the same set of knob values to keep reusing it, as well to ease generic
82 * programming. The knob set for one Struct_builder impl may hugely differ from another's.
83 * Informally, Config is likely to be an *aggregate* `struct` hence aggregate-initializable
84 * (e.g., `Config c{ knob1_val, knob2_val };`).
85 * - This implies that each Struct_builder impl shall feature this ctor form, such that it is possible/advised
86 * to pass the same (equal by value) Config as args each time with no ill effects within reason.
87 * - Struct_builder must also be efficiently move-constructible and move-assignable but not copyable.
88 * - Call payload_msg_builder() which returns a `MessageBuilder*`; mutate the pointee as usual
89 * (likely `initRoot<SomeUserSchema>()` and so on).
90 * - Call emit_serialization(). This emits a #Segment_ptrs container which is basically a
91 * list of *pointers* to `flow::util::Blob` (which is much like `vector<uint8_t>`), where each `Blob` therein
92 * is stored within `*this` until destruction.
93 * - Transmit each `Blob`, in order, by copying into a transport (e.g., by using a Blob_sender).
94 * Note that the transport must be capable of transmitting these blobs while maintaining ordering and
95 * boundaries and without barfing due to anything, including a single blob, being too large.
96 * (It is up to the Struct_builder impl to document any such limitations it might have.)
97 * - (`*this` can now be destroyed.)
98 * - Use the counterpart Struct_reader on the other side of this transport to obtain access
99 * to this serialization. See Struct_reader doc header.
100 *
101 * The following is an optional continuation of the lifecycle of a Struct_builder. There are two
102 * varieties: read-only and read/write. The read-only lifecycle continuation is as follows:
103 * - Call emit_serialization() again.
104 * - Transmit the emitted `Blob`s again into some transport again (the same one, a different one, whatever).
105 * - Again use the counterpart Struct_reader on the other side of this transport to obtain access
106 * to this serialization.
107 * - Repeat as many times as desired.
108 *
109 * I.e., a message -- once mutated and serialized/sent -- can be re-sent as many times as desired.
110 * The concept implementer shall strive to make emit_serialization() be reasonably performant on the 2nd, 3rd, ...
111 * call. While all capnp serializations are zero-copy (before copying into transport), no particular care
112 * should be required to accomplish this; at worst it's a matter of grabbing the segment pointers and emitting
113 * them into a user-provided target container. There typically will be only a few; with SHM-based builders
114 * just one.
115 *
116 * Lastly, then, the read/write lifecycle continuation is as follows:
117 * - (Only if it is a SHM-based serialization) Receive some guarantee that the recipient is done reading
118 * the deserialization (e.g., an ack message).
119 * - If it is not SHM-based, then this is not necessary: The deserialization is fully a copy of `*this`
120 * serialization, so mutating the original has no effect on the recipient.
121 * - (Wait, what? SHM-based? Yes: See "Copying versus zero-copy" section below.)
122 * - Further mutate the payload `MessageBuilder` (as accessed via payload_msg_builder()).
123 * - Call emit_serialization() again.
124 * - Transmit the emitted `Blob`s into some transport (the same one, a different one, whatever) -- again.
125 * - Again use the counterpart Struct_reader on the other side of this transport to obtain access
126 * to this serialization.
127 * - Repeat as many times as desired.
128 *
129 * I.e., it is much like the read-only variant above, except in a SHM-based builder variety one must
130 * take concurrent read/write access into consideration.
131 *
132 * ### Copying versus zero-copy ###
133 * You may note that the above unapologetically declares that the serialization's blobs must be *copied* into
134 * the transport. Isn't this a bad requirement? Isn't it bad to copy the entirety of the serialization?
135 * Answer: yes, it's bad, but consider what it means to copy the *entirety* of a serialization. Imagine a wacky
136 * scenario where an impl of this concept leverages the file system to store the serialization on disk, since
137 * disk has lots of storage capacity. It would be very bad to copy the contents of the disk file into the transport
138 * (and hence out of it, at the opposing end): it would destroy the whole point of storing on disk to begin with... but:
139 * One could, instead of copying the *contents* of the disk file, copy the file's name into the transport
140 * (and on the other side, read the file based on the received name).
141 *
142 * That is to say: The impl may (and indeed, for at least some important applications, should/must) store
143 * the actual "inner" serialization (the one being mutated via mutators called on `payload_msg_builder()`) in bulk
144 * where it wants; for example in shared memory. emit_serialization() can/should then emit segments (ideally,
145 * just one -- and a small one for that matter) comprised by the "outer" serialization, which conceptually might
146 * be a handle referencing the (more-persistent) location of the "inner" (actual) serialization. This has the
147 * following important constraints:
148 * - This outer (emitted by emit_serialization()) serialization by definition must still be a capnp-`struct`.
149 * - The inner one, too, must still be a capnp-`struct`.
150 * - The inner serialization's lifetime must exceed that of `*this` to be of any use! Otherwise the "handle"
151 * in the outer serialization will be potentially useless by the time Struct_reader counterpart tries
152 * to access it.
153 * - This requirement is no joke: obviously it's necessary; but it means one must worry about cross-process cleanup
154 * of the resources (such as SHM-allocated buffer(s)) used up by the inner serialization.
155 *
156 * To be clear, this inner-vs-outer serialization paradigm is *not* required. It is entirely reasonable, for
157 * some applications, to encode everything into the outer (required) serialization and not use any indirection or
158 * inner serialization at all. (Heap_fixed_builder -- the simplest of all -- does just that and is basically
159 * the same as `MallocMessageBuilder`.) The benefit is clear: there's no cross-process cleanup,
160 * and the internal code is just much simpler. Conversely the negative
161 * is clear: copying large things may be expensive or even impossible (depending on the transport).
162 *
163 * Both have their place. In practice, some special-use channels may use a deliberately simple schema
164 * with small, scalar-composed payloads. Then an outer-serialization-only impl (Heap_fixed_builder, say)
165 * is perfect. When this is not so, use a more complex one (typically, we think, one based on SHM; see shm::Builder).
166 *
167 * ### Allocation and perf ###
168 * For this section, please assume we are dealing with a simple (outer-serialization-only/fully-copied) impl
169 * of Struct_builder. The most obvious impl that comes to mind is one that, each time capnp requests a
170 * new segment of a certain minimum size, simply allocates on the regular heap. That's what `MallocMessageBuilder`
171 * does, and similarly Heap_fixed_builder does so as well.
172 *
173 * That's fine. However, in advanced cases, it is conceivable to want more performance. Regular-heap allocation,
174 * depending on the size and underlying allocator impl (tcmalloc/jemalloc/ptmalloc...), can be somewhat expensive.
175 * In avoiding heap-allocation one could use other systems: memory pools; or even just a single pre-allocated
176 * area to be reused by *every* new `*this`. Let's briefly think about each of those.
177 *
178 * Pools: This is not too different from a regular-heap allocator like Heap_fixed_builder. It would just
179 * allocate from (insert pool library here). Heck, it could even use the shared-buffer functionality of a
180 * single `flow::util::Sharing_blob` to dole out memory chunks from a single heap-allocated buffer.
181 *
182 * Single pre-allocated area to be reused by *every* new `*this`: This would be, of course, blindingly fast:
183 * probably `Config` would allocate a single `Sharing_blob` and each `*this` it created would be told to use that
184 * blob, no questions asked. The problem, obviously, is that 2+ `*this`es might step on each other concurrently.
185 * I.e., any use of this would basically require a pattern wherein a `*this` is created, mutated, emitted,
186 * destroyed; *then* same for the next out-message; and so on. Then there is no problem. Simply put, then:
187 * it is reasonable to develop such an impl of Struct_builder -- if one requires the user to carefully
188 * ensure each out-message is entirely handled before work on the next one begins. For some applications that
189 * may be reasonable.
190 *
191 * Similar reasoning could be extended to 2-tier (zero-copy/SHM-based) impls of Struct_builder. Naturally
192 * however one would need to ensure the given payload is fully consumed by the opposing side (Struct_reader)
193 * before creating the next `*this`; it is not enough for merely the local `*this` to be destroyed, as
194 * the underlying payload may be referred-to by a Struct_reader still.
195 */

344/**
345 * A documentation-only *concept* that is, conceptually, roughly what `capnp::MessageReader` is to
346 * `capnp::MessageBuilder`, to be used on an in-message serialized by a counterpart
347 * Struct_builder, having been transmitted over an IPC transmitter of blobs. The following discussion assumes
348 * understanding of Struct_builder (see its doc header).
349 *
350 * @see Struct_builder, the counterpart concept.
351 *
352 * Here is how a Struct_reader is used (in order):
353 * - Suppose the counterpart Struct_builder was used as documented in its doc header, for a given out-message.
354 * - Construct a matching Struct_reader on the other side.
355 * - See also Struct_reader::Config.
356 * - Call `add_serialization_segment(M)`, where M is the maximum number of bytes the next blob-read out of the
357 * transport might return when reading the next serialization segment produced by Struct_builder.
358 * (M, in practice, is limited by at least the transport and how it is used; and the segment sizes
359 * the Struct_builder impl in practice produces.)
360 * - `*this` shall obtain an area in memory at least M bytes long into which it is guaranteed safe for
361 * for the user to direct-write, until the next add_serialization_segment(), deserialization(), or dtor
362 * call (whichever happens earliest).
363 * - add_serialization_segment() shall return a *pointer* to a `flow::util::Blob` whose
364 * [`begin()`, `end()`) range represents that memory area (with `.size()` equal to M).
365 * - If the impl determines that its (not specified by this concept) memory acquisition algorithm
366 * is incapable of obtaining M bytes, it shall return null. The user must be ready for this
367 * eventuality.
368 * - Copy the next segment's contents out of the transport into the aforementioned `Blob`.
369 * - Use `flow::util::Blob::resize()` (or similar methods that invoke it, notably `start_past_prefix_inc()`)
370 * to modify the [`begin()`, `end()`) range such that it stores the exact contents of the segment.
371 * - In practice: It at least means `resize(K)`, where `K <= M`, to mark the actual size of the received
372 * segment, since M is the *max* possible size. So the garbage area into which no bytes were written must
373 * be excluded.
374 * - It may also mean sliding around `begin()` and/or `end()` to deliberately exclude any framing information
375 * used by the user's protocol when transmitting segments. This is important to maintain zero-copy perf.
376 * - Behavior is undefined if the user calls `.make_zero()` on that blob. Only `.resize()` (and its
377 * convenience wrappers like `start_past_prefix_inc()`) may be invoked.
378 * - If and only if the serialization produced by the Struct_builder consists of 2+ segments:
379 * Repeat the previous 3 bullet points, in order, for each of the additional 1+ segments.
380 * - Lastly execute Struct_reader::deserialization<Struct>() (at most once) to obtain obtain a capnp-generated
381 * `Struct::Reader`.
382 * - Access the structured contents that the other side had mutated-in via the latter-returned `Struct::Reader`.
383 * - Finally, the Struct_reader `*this` can be destroyed.
384 *
385 * ### Copying versus zero-copy ###
386 * See the same-named section of Struct_builder doc header. Similar concerns apply here. Generally speaking,
387 * how a Struct_reader works is dictated by the decisions made for its Struct_builder counterpart.
388 *
389 * ### Allocation and perf ###
390 * See the same-named section of Struct_builder doc header first. Similar concerns apply here, but
391 * actually the decision about how to obtain memory areas as returned by add_serialization_segment() in
392 * Struct_reader can be 100% decoupled from how Struct_builder counterpart acquires memory.
393 * E.g., the builder might allocate from heap, while the reader might allocate from a single
394 * buffer allocated by `Config`, just because the latter has a single-threaded receiver setup, while the former
395 * involves multi-threaded concurrently-sending chaos. One can mix and match: the outer serialization is
396 * always copied-into the transport (then destroyed) on the builder side and copied-from the transport
397 * on the reader side. The manner in which memory is acquired before each write (builder case)
398 * or before each read (reader case) can be determined by the algorithm used by the application on each
399 * side.
400 *
401 * ### Why is add_serialization_segment() designed this way? ###
402 * It might seem unnecessarily complex: user must call add_serialization_segment(), then write to that `Blob`
403 * and adjust its range. They could conceivably just allocate their own `Blob` and then `move()` it into
404 * `*this` via a single, simpler `add_serialization_segment(Blob&&)` call. So why is it done the former way?
405 *
406 * Answer: It's to support the possibilities discussed in the "Allocation and perf" section above.
407 * add_serialization_segment() places the responsibility of obtaining target memory for the blob-read
408 * on the Struct_reader concept impl. Hence different concept impls can be swapped in for different
409 * perf/use-case trade-offs. One would use a pool -- another would use the heap... etc.
410 */

484 /**
485 * Prior to deserialization() obtains a memory area `max_sz` bytes long into which the user may write-to
486 * until the next add_serialization_segment(), deserialization(), or dtor call (whichever happens first);
487 * returns a pointer to that area as described by the pointed-to `Blob`'s [`begin()`, `end()`) range.
488 * If the concept impl decides `max_sz` bytes are not available, returns null. `*this` shall not be used
489 * subsequent to such an eventuality.
490 *
491 * To use deserialization() subsequently, you must:
492 * - call this at least once (successfully);
493 * - call this (N - 1) more times (successfully), where N is the number of segments produced by the counterpart
494 * Struct_builder.
495 * - So N >= 1 by definition.
496 *
497 * The required sequence for each in-segment is:
498 * - Call add_serialization_segment(), which yields pointer to a `Blob`.
499 * - Write segment serialization into (some subrange of) that `Blob`'s [`begin()`, `end()`) range
500 * (presumably directly out of the transport).
501 * - Adjust said `Blob`'s [`begin()`, `end()`) range to contain exactly the serialization segment.
502 * - You may use `.resize()` or its convenience wrappers `.start_past_prefix()` and `.start_past_prefix_inc()`.
503 * Use these to (1) mark the actual segment's size (which may well be < `max_sz`) and (2)
504 * discard any framing data that came in together with the segment serialization.
505 * - You may not `.make_zero()` (or behavior is undefined).
506 * - You may not use `.resize()` and friends to cause `begin()` to go left or `end()` to go right.
507 * They can only remain unchanged or travel right and left respectively (the range's edges cannot "extend"
508 * past the original range provided).
509 *
510 * After these steps, either add_serialization_segment() again (if more segments are forthcoming), or
511 * call deserialization() (to use the structured data), or invoke dtor (if `*this` is no longer of interest).
512 *
513 * ### Alignment ###
514 * The returned `Blob`'s begin-to-end range shall be word-aligned, meaning `.const_data()` shall be on a word
515 * boundary, meaning the numeric value thereof shall be a multiple of `sizeof(void*)`. This is the guarantee
516 * of add_serialization_segment().
517 *
518 * Accordingly, any `.resize()` (etc.) done on the returned `Blob` by the user *must* result in a
519 * `.const_data()` (a/k/a `.begin()`) that is *still* word-aligned by the same definition. Informally there
520 * are ~2 ways to guarantee this on the user's part:
521 * - Just don't change `begin()` (do not use prefix frame data; you may use postfix frame data instead). Or:
522 * - If you require frame data to live in front of the segment, in your protocol ensure that
523 * the size of such frame data is a multiple of `sizeof(void*)`.
524 *
525 * @param max_sz
526 * `Blob` to which the returned pointer points shall have `.size() == max_sz`.
527 * (It may have equal or larger `.capacity()`.) See above.
528 * @return Pointer to the `Blob` inside which to place, subsequently, the next segment's contents. See above.
529 */
530 flow::util::Blob* add_serialization_segment(size_t max_sz);
531
532 /**
533 * After all serialization segments have been acquired in RAM via add_serialization_segment() and finalized
534 * via begin-end-range adjustment, this returns a reference to the reader object whose capnp-generated
535 * accessors can read the structured contents of `*this`. Behavior is undefined if this is called before all
536 * add_serialization_segment() calls have been made (and they must have been successful). Behavior is undefined
537 * if add_serialization_segment() is called after this is called.
538 *
539 * deserialization() shall be called no more than once.
540 *
541 * The returned `Reader` shall remain valid until `*this` is destroyed. (Therefore any copies thereof
542 * shall also be valid in the same time frame. However there is no perf benefit to making such copies, and
543 * there is likely a small penalty to doing so.)
544 *
545 * @tparam Struct
546 * Same meaning as in the vanilla capnp use of `MessageReader::getRoot<Struct>()`.
547 * @param err_code
548 * See `flow::Error_code` docs for error reporting semantics. #Error_code generated:
549 * error::Code::S_DESERIALIZE_FAILED_INSUFFICIENT_SEGMENTS (add_serialization_segment() was called
550 * an insufficient number of times; in particular 0 times always triggers this);
551 * error::Code::S_DESERIALIZE_FAILED_SEGMENT_MISALIGNED (add_serialization_segment() was called with
552 * a segment that starts at a misaligned address);
553 * implementation may also emit other errors.
554 * @return See above. If and only if `err_code` is non-null, and truthy `*err_code` is emitted, a useless
555 * (default-cted) `Reader` is returned.
556 */
557 template<typename Struct>
558 typename Struct::Reader deserialization(Error_code* err_code = 0);
559}; // class Struct_reader
