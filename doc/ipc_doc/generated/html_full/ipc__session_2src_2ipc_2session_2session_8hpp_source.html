33/**
34 * A documentation-only *concept* defining the local side of an IPC conversation (*session*)
35 * with another entity (typically a separate process), also represented by a Session-implementing object,
36 * through which one can easily open IPC channels (ipc::transport::Channel), among other IPC features.
37 * While it is possible to open a `Channel` without the help of this Session concept, for many users writing
38 * large multi-channel applications it would be too laborious to do so. The Session concept is the central
39 * one in ipc::session.
40 *
41 * @see ipc::session doc header.
42 * @see Session_mv which implements the majority of Session concept, namely when it is in PEER state.
43 *
44 * ### Concept contents ###
45 * The concept defines the following behaviors/requirements.
46 * - The object has at least 2 states:
47 * - NULL: Object is not a peer: is not open/capable of operation. Essentially it is not a useful
48 * state. A default-cted object is in NULL state; and a moved-from object is as-if default-cted, therefore
49 * also in a NULL state. In this state all the transmission-related methods return `false`/sentinel and no-op.
50 * - PEER: Object is or has been a peer (open/capable of operation). It is not possible to exit PEER
51 * state, except via being moved-from. The session ending, such as due to the partner process exiting,
52 * retains PEER state for instance.
53 * - Other states are allowed but outside the scope of the concept. For example a CONNECTING state may or may
54 * not be relevant (but again not relevant to this concept).
55 * - Active-opening a channel: open_channel(). See its doc header. It yields a PEER-state #Channel_obj.
56 * - Passive-opening a channel (the counterpart to the partner Session's open_channel()).
57 * - At entry to PEER state (the mechanism of accomplishing which, as noted below, is not specified by the concept)
58 * the user must have provided a passive-open handler (discussed below in detail) via which subsequent
59 * passive-opens are reported to the user. To this handler a PEER-state #Channel_obj is passed in case of
60 * success.
61 * - Alternatively, if at entry to PEER state the user has chosen to *not* supply a passive-open handler,
62 * then the partner Session::open_channel() shall yield failure. In this mode of operation, the user has
63 * chosen this side of the session to not act as a passive-opener. (If both `Session`s do so, the session
64 * is essentially useless. Formally this is supported, simply because there is no need to enforce otherwise.)
65 * - Async error reporting: At entry to PEER state (the mechanism of accomplishing which, as noted below, is not
66 * specified by the concept) the user must have provided an on-error handler via which asynchronously-occurring
67 * session-hosing #Error_code are reported. See details below.
68 * - The (outgoing) transmission-of-messages methods, including transmission of graceful-close message.
69 * See their doc headers.
70 * - Behavior when the destructor is invoked. See ~Session() doc header.
71 * - Default ctor (which creates a NULL-state object that can be moved-to and to which "as-if"
72 * state any moved-from object is changed).
73 * - Move ctor, move assigment operator; these do the reasonable thing including setting the moved-from object
74 * to NULL state. Various satellite APIs (e.g., Session_server) may
75 * need these. That is such APIs do not rely on the factory/shared-ownership pattern.
76 *
77 * The concept (intentionally) does *not* define the following behaviors:
78 * - How to create a Session, except the default and move ctors.
79 * That is it does not define how to create a *new* (not moved-from) and *functioning* (PEER state:
80 * open/capable of communication) peer object. It only defines behavior once the user has access to
81 * a Session that is already in PEER state: connected to an opposing peer Session.
82 *
83 * ### Async-I/O channels versus `sync_io` ones ###
84 * There are 3, overall, ways to obtain a transport::Channel via ipc::session.
85 * - Session::open_channel(). It's a synchronous operation; it sets a `Channel` as a pointer out-arg.
86 * - Passive-opening. It's an async operation via passive-open handler (see below for details).
87 * - Init-channels. Client_session_mv::sync_connect() (et al) and Session_server::async_accept() (et al)
88 * provide ways to pre-open channels to be ready the moment the session enters PEER state.
89 *
90 * In all cases these channels shall bear `sync_io`-pattern peer objects: not async-I/O ones.
91 * - They are lighter-weight to create and pass around.
92 * - They can be immediately fed to transport::struc::Channel or transport::struc::sync_io::Channel ctor
93 * via `move()` semantics.
94 * - They can be trivially upgraded to async-I/O form via transport::Channel::async_io_obj().
95 *
96 * ### Thread safety and handler invocation semantics ###
97 * As is typical: 2 different Session objects are safe to invoke methods on concurrently; it is not safe to
98 * invoke a non-`const` method on a given `*this` concurrently to any other method on the same `*this`.
99 *
100 * The handlers -- namely the passive-open handler (optional) and the on-error handler (mandatory) -- shall be
101 * invoked from some unspecified thread that is not any one of the user's calling threads. It is *not* allowed
102 * to invoke `*this` APIs from within any such handler. For example, you may not open_channel() from
103 * the passive-open handler. Informally, it is recommended that any user handler post its actual handling of the
104 * event onto their own thread(s), such as via `flow::async::Single_thread_task_loop::post()`.
105 *
106 * ### Passive-open handler semantics ###
107 * As summarized above, a Session -- at entry to PEER state -- is optionally capable of accepting the partner
108 * Session's active-open (open_channel()) attempt. The user must make the decision whether to be capable or
109 * incapable of performing such a passive-open. If they decide the latter, they must not set the *on-passive-open*
110 * handler; else they must set it by the time PEER state begins. (The mechanism of doing so is unspecified by
111 * the concept. The current impls, #Server_session and #Client_session, use different APIs for this.)
112 *
113 * If the choice is made to indeed perform passive-opens (accept other side's active-opens), then suppose the
114 * handler H is indeed saved inside `*this` (which is in PEER state). Then the following shall be invoked from some
115 * unspecified thread that is not one of the user's public API-invoking threads: `H(C, M)`, where `C` is a
116 * #Channel_obj onto which the new PEER-state transport::Channel shall be `move()`d; while `M` is the metadata reader
117 * (explained below). Therefore the H signature must be compatible with:
118 *
119 * ~~~
120 * void H(Channel_obj&& new_peer_channel, Mdt_reader_ptr&& metadata_reader)
121 * ~~~
122 *
123 * Note that #Mdt_reader_ptr is a `shared_ptr` to a capnp-generated `Reader` to a capnp-`struct`
124 * parameterized on #Mdt_payload_obj. The memory resources taken by the serialization of the metadata
125 * (including certain internal zero-copy "baggage") shall be freed once that `shared_ptr` ref-count group is empty.
126 * As for what the heck this even *is*: see open_channel()'s metadata arg docs. This is the `Reader` counterpart
127 * to the `Builder` mutated by the opposing open_channel() caller.
128 *
129 * ### Open channel metadata ###
130 * open_channel() is used to initiate a channel opening on one side, and the on-passive-open handler is used
131 * to accept the same on the other side (for a given channel). The result, on each side, is a #Channel_obj,
132 * which is a `Channel` template instantiation. Note that it is not a `struc::Channel`; the user may choose
133 * to immediate convert `Channel` to `struc::Channel` via std::move(), but whether they want to or not is up
134 * to them and their desired use case. Now, there are millions of patterns of what the acceptor might want
135 * to accept in a session; there could be one channel for some app-wide updates and one per processor core;
136 * and possibly more differentiation than that regarding which channel is for what purpose. Meanwhile the only
137 * way to know which channel is being opened, other than a set ordering (certainly inconvenient at best in many
138 * cases), is to perhaps exchange some information on the channels themselves. That too is inconvenient:
139 * even having discovered that a certain channel is for a certain purpose, distributing them around the app is
140 * no fun probably. Furthermore, it's only a `Channel`: is the user supposed to encode something in a binary
141 * blob? Inconvenient. But if one turns it into a `struc::Channel`, then that is irreversible and may not
142 * be what the user wants. The solution we use: the open_channel() call is issued with a separate (to any
143 * structured in-message the user would see) chunk of capnp-structured *metadata*. This is transmitted
144 * and supplied to the on-passive-open handler per the preceding doc section. Details:
145 *
146 * It is encoded in a capnp-`struct` `Metadata`, defined in common.capnp. However that `struct`
147 * is generically parameterized on another capnp-type of the user's choice. That is the Session template arg
148 * #Mdt_payload_obj. Hence the user shall decide how to encode the channel-differentiating info on a
149 * per-Session basis. capnp shall generate the usual `Reader` and `Builder` nested classes; the latter to
150 * build it to pass into open_channel(); the former to read it, the on-passive-open handler having fired.
151 * Call capnp-generated `Builder` setters and `Reader` getters accordingly on either side.
152 *
153 * This feature is optional to use. You may set #Mdt_payload_obj to `capnp::Void`; omit
154 * the metadata argument to open_channel(); and ignore the `Reader` counterpart in the on-passive-open handler.
155 *
156 * Informal suggestion: While Session concept does not specify anything about clients or servers, as of this
157 * writing the client-server model is the only one available, hence #Client_session and #Server_session are the impls
158 * used. `Server_session`s are produced, in this model, via Session_server. The latter class template
159 * requires that `Mdt_payload` be specified as a template arg to be applied to all Server_session
160 * objects it produces. Therefore it must be decided not just for each session but realistically for the entire
161 * client/server split (in other words the entire meta-application that was split into two): all the
162 * `Server_session`s; and therefore to all the `Client_session`s as well (since the 2 are peers to each other).
163 * So the method of signaling channel type (if needed) via metadata is quite a high-level decision.
164 * One obvious idea is to maintain a split-wide capnp schema that specifies the `struct` to use as
165 * `Mdt_payload`; in that `struct` define perhaps a capnp-`enum` type for channel type.
166 * Alternatively one can define a top-level anon capnp-`union`; its `which()` selector can function as an
167 * enumeration (in fact it is one in the generated C++); and any further data (e.g., the processor core index,
168 * if there's some channel-per-core setup) inside that particular `which()`'s `struct`. Do be careful to
169 * (1) maintain backward and forward compatibility as channel types are added over time; and (2)
170 * do not abuse the metadata store with large structures. Anything that is not directly used in differentiating
171 * between channels should simply be communicated over the channel itself, once it has been routed to the
172 * proper area of code; that's what transport::struc::Channel is for in particular.
173 *
174 * ### Error handling semantics ###
175 * There are 2 sources of errors: background (async) errors and synchronously from invoking active-open: open_channel().
176 * A given error is either *session-hosing* (fatal) or not. To make handling error simpler these are the semantics:
177 * - open_channel() synchronously emitted errors: *not* session-hosing.
178 * - Background (async) errors: *always* session-hosing.
179 *
180 * Even if, internally, open_channel() in some way triggers a session-hosing error, it shall be emitted
181 * as-if it were an async error in a way indistinguishable in practice from any other async error.
182 *
183 * When a session-hosing error occurs, it is emitted exactly once (through the on-error handler). After such an
184 * event, open_channel() and other APIs shall no-op/return sentinel subsequently if invoked.
185 *
186 * This is simpler than how transport::struc::Channel emits errors; there the (always synchronous) `send()`
187 * can emit an error synchronously that is session-hosing; in addition to incoming-direction session-hosing errors;
188 * but still only one of these can occur and only once. So in that sense open_channel() is *not* like `send()`:
189 * if it emits an error, `*this` is still formally in non-session-hosed state.
190 *
191 * However, of course, the user is free to treat an open_channel()-emitted error as fatal and destroy `*this`.
192 *
193 * @see Channel and buddies: the main object of opening a Session is to be able to easily active-open and/or
194 * passive-open `Channel`s in PEER state through it.
195 * @see #Client_session, #Server_session: as of this writing, any non-SHM-enabled pair of PEER-state `Session`s is
196 * a #Client_session and a #Server_session. `Server_session`s in a process, designated as the *server* in an
197 * IPC *split*, are obtained (following the acceptor pattern) via a single Session_server in that
198 * process. On the other side, a Client_session is directly constructed (in NULL state) and
199 * enters PEER state via Client_session::sync_connect(). (Conceptually this is similar to the relationship
200 * between, e.g., Native_socket_stream (server side), Native_socket_stream_acceptor, and Native_socket_stream
201 * (client side). However, it so happens that in the case of `Session`s, a different class is used depending
202 * on which side the Session sits. Since they both implement the same concept, however, code can be written
203 * that behaves identically -- given any Session in PEER state -- regardless of whether it's really a
204 * Client_session or Server_session. That said, such code does have to be templated on the particular
205 * Session as a template param.)
206 * @see E.g., shm::classic::Client_session and shm::classic::Server_session which are an example of a SHM-enabled
207 * variant of Session impls.
208 *
209 * @tparam Mdt_payload
210 * capnp-generated class for the user's capnp-`struct` of choice for open_channel() transmitting metadata
211 * information to be received on the other side's on-passive-open handler. See discussion above.
212 * Use `capnp::Void` if not planning to use this metadata feature.
213 */

355 /**
356 * In NULL state, no-op; in PEER state: Ends the session while informing (if possible) the opposing peer Session that
357 * the present session has ended. This may incur a short delay before the dtor exits (in practice likely to
358 * flush out any pending sends over an internal session master channel, informally speaking).
359 *
360 * Any #Channel_obj previously yielded by open_channel() and/or passive-opens shall continue operating, if still
361 * operating, and it is up to the user to perform any cleanup/flushing/etc. on these. This is consistent with
362 * the idea that open_channel() and passive-open passes ownership of such channels to the user.
363 *
364 * There are 2 mutually exclusive expected triggers for why this dtor would be invoked:
365 * - The present process is exiting gracefully (hence the session must end). User invokes dtor. Dtor informs
366 * the other side with maximum expediency; then returns.
367 * - The opposing process is exiting gracefully or otherwise; the opposing Session thus informed ours gracefully
368 * or otherwise; therefore `*this` informs user of this via emitted error (see concept doc header regarding
369 * session-hosing error emission semantics); therefore `*this` is no longer usable; hence user invokes
370 * `*this` dtor. It tries to inform the other side, but there's nothing to inform; it quickly returns.
371 */
372 ~Session();

402 /**
403 * Synchronously active-opens a new channel which, on success, is moved-to `*target_channel`.
404 * No-op and return `false` if `*this` is not in PEER state, the session is hosed by a previous error,
405 * or `mdt` is null. Else returns `true`.
406 *
407 * `mdt` shall be from an mdt_builder() call, and `mdt` must not have been passed to
408 * open_channel() previously; otherwise behavior undefined. See also the open_channel() overload that takes no `mdt`.
409 *
410 * @note This is a compile-time fact, but we point it out just the same: `*target_channel` is of
411 * a `sync_io`-core-bearing type. However, on successful open_channel() you can trivially obtain
412 * an async-I/O version via `target_channel->async_io_obj()`.
413 *
414 * ### Is it blocking or non-blocking? ###
415 * That's a matter of perspective; but formally it must be either non-blocking or, if blocking, then limited to
416 * a timeout no greater than seconds (not minutes). That said, informally, the implementation shall strive to make
417 * this quite fast, enough so to be considered non-blocking, *as long as both Session objects are in PEER state*.
418 *
419 * ### Error emission ###
420 * If `false` is returned, even if open_channel() internally triggered a session-hosing condition, it shall be
421 * fired through the on-error handler and not emitted by open_channel().
422 *
423 * If `true` is returned an error may be detected and is then emitted via standard Flow semantics
424 * (if `err_code` then set `*err_code` to error; otherwise throw). Such an error is *not* session-hosing.
425 *
426 * In particular if passive-opens are disabled on the opposing side it must emit
427 * error::Code::S_SESSION_OPEN_CHANNEL_REMOTE_PEER_REJECTED_PASSIVE_OPEN. Informally: an impl is likely to
428 * be capable of emitting the non-fatal error::Code::S_SESSION_OPEN_CHANNEL_ACTIVE_TIMEOUT, though the user
429 * is reasonably likely to want to treat it as fatal in practice and end the session.
430 *
431 * @param target_channel
432 * On success `*target_channel` is moved-to from a (new) #Channel_obj in PEER state.
433 * @param mdt
434 * See mdt_builder(). If `mdt` is null, open_channel() returns `false`.
435 * @param err_code
436 * See `flow::Error_code` docs for error reporting semantics. #Error_code generated:
437 * error::Code::S_SESSION_OPEN_CHANNEL_REMOTE_PEER_REJECTED_PASSIVE_OPEN,
438 * other possible unspecified errors according to impl's discretion. See above discussion.
439 * @return See above.
440 */
441 bool open_channel(Channel_obj* target_channel, const Mdt_builder_ptr& mdt, Error_code* err_code = 0);

443 /**
444 * Identical to `open_channel(target_channel, mdt_builder(), err_code)`; in other words
445 * attempts to open channel with an uninitialized channel-open metadata payload. Informally: this overload
446 * can be used if differentiation among channels, for the passive-open side's benefit, is not necessary.
447 * In that case #Mdt_payload_obj is typically set to `capnp::Void`.
448 *
449 * @note This is a compile-time fact, but we point it out just the same: `*target_channel` is of
450 * a `sync_io`-core-bearing type. However, on successful open_channel() you can trivially obtain
