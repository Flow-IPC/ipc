40/**
41 * Identical to session::Server_session in every way, except that it makes available two SHM arenas, from the
42 * SHM-jemalloc provider (ipc::shm::arena_lend::jemalloc). These SHM arenas (see #Arena doc header) have different
43 * scopes:
44 * - Per-session-scope, accessible via session_shm(): meaning it shall be accessible only during the lifetime of
45 * this session, by this Session (via this accessor).
46 * - Hence this arena is created at the time `*this` enters PEER state (near Session_server::async_accept()
47 * successfully yielding `*this`). When `*this` is destroyed or hosed, the arena is invalidated.
48 * - Per-app-scope, meaning its pool shall be accessible potentially beyond the lifetime of
49 * this session but rather until the generating Session_server (i.e., server process) shuts down, by any
50 * Session under that umbrella, now or in the future, as long as its Client_app equals that of `*this` session.
51 * - Hence this arena is created at the time the shm::arena_lend::jemalloc::Session_server (emitter of `*this`
52 * to user) first establishes a session with the *first* shm::arena_lend::jemalloc::Client_session matching
53 * the same Client_app as `*this`. I.e., it's created at the same time as `this->session_shm()` is, if
54 * `*this` is the first such session; otherwise earlier. When `*this` is destroyed or hosed (gracefully at
55 * least), the arena is *not* invalidated but persists.
56 *
57 * @note The opposing arena_lend::jemalloc::Client_session is similar but lacks the app_shm() #Arena.
58 *
59 * ### Explicitly constructing an object in SHM; lending to opposing side ###
60 * This setup is symmetrical w/r/t a given
61 * shm::arena_lend::jemalloc::Server_session<->shm::arena_lend::jemalloc::Client_session object pair. Both
62 * can create new objects (via respective `Arena::construct()` methods). Both can write and, of course, read
63 * within locally `construct()`ed objects. Lastly each can transmit any outer
64 * handle, returned by `construct()`, to the opposing side. To do this
65 * call lend_object(), transmit the returned `Blob` to the opposing side via any desired IPC form (probably a
66 * transport::Channel or transport::struc::Channel), then recover a handle via borrow_object() on that opposing
67 * side. As of this writing access to objects on the borrowing side is read-only, for safety reasons,
68 * but that could change (most likely becoming an optional mode) at the prerogative of ipc::shm::arena_lend
69 * developers. (Do note that ipc::shm::classic allows full write access on the borrower side, as in an
70 * arena-sharing SHM-provider by definition both sides allocate and thus write to the same SHM pool(s).
71 * However this comes at the expense of safety. Further discussion is up to ipc::shm docs.)
72 *
73 * @note Be ready for Session_mv::lend_object() to return an empty blob and Session_mv::borrow_object() to return
74 * a null pointer, both indicating an error: the session is hosed (opposing process likely down). See their
75 * doc headers for a bit more info on this.
76 *
77 * By using `shm::stl::Arena_activator<Arena>` (alias shm::arena_lend::jemalloc::Ipc_arena_activator) and
78 * #Allocator (alias shm::arena_lend::jemalloc::Ipc_arena_allocator)
79 * it is possible to construct and transmit not just POD (Plain Old Datatype) objects but combinations of those
80 * with unlimited nested levels of STL-compliant containers. On the borrowing side use
81 * #Borrower_allocator (no activator necessary).
82 *
83 * Detailed instructions can be found in those shm::stl and ipc::shm::arena_lend doc headers.
84 *
85 * ### Using SHM as backing memory for transport::struc::Channel ###
86 * Another use of session_shm() and app_shm() (though only on this session-server side in case of the latter) --
87 * in some ways perhaps superior -- is indirect. The above technique involves manually constructing a
88 * C++ data structure and transmitting a short handle to it over IPC thus achieving high performance.
89 * It may be desirable, instead, to use a structured message, filled out according to schema. This, of course,
90 * is why transport::struc::Channel exists. As is explained in its docs, there is nothing SHM-centered about it.
91 * It does however have the key feature that allows one to supply, at compile-time, the builder and reader engines
92 * that acquire memory while the user mutates an out-message and later accesses it upon IPC-receipt. Naturally one of
93 * the builder/reader pairs uses SHM, instead of regular heap, as the supplier of RAM. Thus when
94 * one invokes transport::struc::Channel::send() and receives it in the opposing object, the actual bits
95 * copied into/out of the low-level transport are merely the SHM handle (and all of this is hidden from the user
96 * outside of the delightful near-zero-copy perf properties).
97 *
98 * There are two ways to make this happen. The easiest and best way is, when constructing the
99 * `struc::Channel`, to use the alias session::shm::arena_lend::jemalloc::Session_mv::Structured_channel and
100 * create one via tag-form ctor with tag transport::struc::Channel_base::Serialize_via_session_shm
101 * (or `Serialize_via_app_shm`, server-side only). Simply provide that tag and `this` (or, symmetrically,
102 * `shm::arena_lend::jemalloc::Client_session::this` on the other side).
103 *
104 * The harder way, albeit allowing for certain advanced setups, is to manually create a
105 * `transport::struc::shm::arena_lend::jemalloc::Builder::Config` and/or `Reader::Config`,
106 * passing in `this->session_shm()` and/or `this->app_shm()` and/or `this->session_shm()`, to those;
107 * and then pass the `Config` or `Config`s, plus `this->shm_session()`, to the non-tag-form of `struc::Channel`
108 * ctor.
109 *
110 * This is all documented on transport::struc::Channel. Do realize, though, that those niceties are
111 * really built on this class template and/or the opposing shm::arena_lend::jemalloc::Client_session.
112 * To use them with ipc::session, you must thus choose shm::arena_lend::jemalloc::Server_session and
113 * shm::arena_lend::jemalloc::Client_session as your Session impls.
114 *
115 * @internal
116 * ### Implementation ###
117 * It is probably not too difficult to understand how it works by just reading the code and doc headers.
118 * The key is that the core is still just what's supplied by a vanilla (non-SHM-aware) #Server_session.
119 * Ultimately all we need to add is some setup near the end of the vanilla session-opening procedure.
120 * This is easiest to discuss holistically, meaning both the server (`*this`) and client (opposing) side,
121 * so the discussion is all here.
122 *
123 * As with the vanilla session setup, remember that while in PEER state the APIs are identical/symmetrical,
124 * the roles of the server and client vary quite sharply internally -- especially before PEER state.
125 * The general outline is: Session_server listens for socket connections; a given
126 * `Client_session::sync_connect()` connects; at this stage Session_server constructs a not-yet-user-emitted
127 * #Server_session and invokes its `async_accept_log_in()`. Now the #Client_session and #Server_session
128 * have established a channel (internal-use session master channel) and undergo the log-in exchange.
129 * #Client_session sends log-in request. #Server_session verifies that, then replies with log-in response
130 * (including important bits of generated naming) and enters PEER state. Once #Client_session (shortly) receives
131 * this response, it knows the session is ready to rock and also enters PEER state. Voila.
132 *
133 * Where does SHM-jemalloc setup hook into this? The simpler case is the session-client which manages only
134 * one #Arena (per-session scope: session_shm()), so let's describe it first. In arena-lending SHM-providers,
135 * what it takes to construct and lend/borrow object in SHM is:
136 * - An #Arena (managed in this process). Things are constructed (allocated) by this.
137 * - A #Shm_session (not to be confused with ipc::Session and company). `Arena`-constructed objects are
138 * *lent* (`lend_object()`) via this object. Opposing process's `Arena`-constructed objects are
139 * locally *borrowed* (`borrow_object()`) via this object.
140 * - The #Arena must be registered with the #Shm_session (`lend_arena()`) during setup.
141 *
142 * In some ways this is simpler than in SHM-classic, as in that one the arena is shared by the two processes, so
143 * one (we chose server) must create it and keep handle; while the other (client) must open it by a known name.
144 * In our case each side maintains its own arena, so the order does not matter per se. There is however one
145 * added complication:
146 * - For the #Shm_session to be created, on either side, it must be passed a pre-opened `struc::Channel`
147 * of a specific type which it shall use for internally-needed IPC. Specifically it just wants a
148 * transport::Socket_stream_channel_of_blobs which internally consists of a pre-connected local stream socket
149 * (Unix domain socket). Much as with user-requested channels in vanilla sessions, we assign the session-server
150 * side the task of creating the socket-pair, sending one socket (transport::Native_handle) to client,
151 * and expecting a quick ack message before proceeding to create #Shm_session.
152 * - Accordingly the client side awaits the message (along session master channel) containing the socket,
153 * acks it, and creates the #Shm_session.
154 *
155 * To summarize, then, this is the flow for Client_session_impl::async_connect().
156 * -# Vanilla session::Client_session_impl::async_connect() gets it to PEER state. Cancel PEER state immediately
157 * in our substituted on-connect-done handler.
158 * -# Await the `Native_handle` to arrive on the session master channel (which was just used for vanilla
159 * session opening procedure and is now not being used). Upon receipt send ack message and using the
160 * received handle:
161 * -# Create #Shm_session (Client_session_impl::shm_session() shall make it available).
162 * - Session_impl::init_shm() does this.
163 * -# Create per-session arena (Client_session_impl::session_shm() shall make it available).
164 * - Session_impl::init_shm() does this.
165 * -# Register the latter with the former.
166 * - Session_impl::init_shm() does this.
167 * -# Re-enter PEER state; and invoke the true (user-supplied) on-connect-done handler.
168 *
169 * If the server had symmetrical needs only, it'd be easy to describe its algorithm -- basically mirror
170 * the above. However it also has the per-app arena (app_shm()). It must be created, but only when an incoming
171 * session-open request is the first for its Client_app. (This logic is above the layer of sessions and must
172 * reside in Session_server, not in Server_session_impl; but the two interact.) Later, regardless of whether it
173 * was just created, app_shm() must be registered with the #Shm_session, once that is created. Thus on the
174 * server side the async_accept_log_in() steps are:
175 * -# Vanilla async_accept_log_in() gets it to almost-PEER state.
176 * -# During the session::Session_server_impl "per-Client_app setup" customization-point hook:
177 * Session_server: Create the per-app arena for that Client_app and save it in its app-to-arena map, unless
178 * it is already there (then do nothing).
179 * -# During the session::Server_session_impl `pre_rsp_setup_func` hook that executes shortly after that:
180 * Cache the appropraite Session_server::app_shm() inside `*this` Server_session_impl, for quick
181 * subsequent access via Server_session::app_shm().
182 * -# Before invoking the user's on-accept-done handler -- giving them their new `Server_session` --
183 * perform the following *synchronous* steps:
184 * -# Create the pre-connected native-handle pair (socket-pair).
185 * -# Synchronously send it via session master channel (which was just used for vanilla session opening
186 * procedure and is now not being used) and get ack from client (see above).
187 * -# Create #Shm_session (Client_session_impl::shm_session() shall make it available).
188 * - Session_impl::init_shm() does this.
189 * -# Create per-session arena (Client_session_impl::session_shm() shall make it available).
190 * - Session_impl::init_shm() does this.
191 * -# Register the latter with the former.
192 * - Session_impl::init_shm() does this.
193 * -# Register app_shm() (cached a few steps earlier) with the former (#Shm_session) also.
194 * - Session_impl::init_shm() does this.
195 * -# Invoke the true (user-supplied) on-accept-done handler.
196 *
197 * The last and most boring piece of the puzzle are the pImpl-lite wrappers around the SHM-specific API
198 * that shm::arena_lend::jemalloc::Server_session adds to super-class Server_session_mv: `session_shm()` and so on.
199 * Because this API is (with the exception of app_shm() (and buddies) being server-only) exactly identical for
200 * both shm::arena_lend::jemalloc::Server_session and shm::arena_lend::jemalloc::Client_session,
201 * to avoid torturous copy/pasting these common wrappers are collected in shm::arena_lend::jemalloc::Session_mv
202 * which we sub-class. So the `public` API is distributed as follows from bottom to top:
203 * - shm::arena_lend::jemalloc::Server_session
204 * => shm::arena_lend::jemalloc::Session_mv => session::Server_session_mv => session::Session_mv
205 * => shm::arena_lend::jemalloc::Server_session_impl
206 * - shm::arena_lend::jemalloc::Client_session
207 * == shm::arena_lend::jemalloc::Session_mv => session::Client_session_mv => session::Session_mv
208 * => shm::arena_lend::jemalloc::Client_session_impl
209 * - The bottom two in this case are not even inherited but one aliases to the other. This is because
210 * there's nothing added API-wise on the client side. On the server side, as you see below, at least
211 * app_shm() is necessary to add, so we can't alias and must sub-class.
212 *
213 * You'll notice, like, `Base::Base::Base::Impl` and `Base::Base::Base::Impl()` in certain code below;
214 * that's navigating the above hierarchy.
215 *
216 * @endinternal
217 *

222 * @tparam Mdt_payload
223 * Identical to session::Server_session.
224 */
