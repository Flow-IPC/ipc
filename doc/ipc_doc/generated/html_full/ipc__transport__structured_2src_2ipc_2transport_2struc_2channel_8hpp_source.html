36/**
37 * Owning and wrapping a pre-connected transport::Channel peer (an endpoint of an established channel over which
38 * *unstructured* messages and optionally native handles can be transported), this template is the central pillar of the
39 * ipc::transport::struc (*structured layer*), capable of communicating structured capnp-schema-based messages (and
40 * native handles).
41 *
42 * @see sync_io::Channel and util::sync_io doc headers. The latter describes a general pattern which
43 * the former implements. In general we recommend you use a `*this` rather than a sync_io::Channel --
44 * but you may have particular needs (summarized in util::sync_io doc header) that would make you decide
45 * otherwise.
46 *
47 * ### Context (short version) ###
48 * struc::Channel (and its `sync_io` counterpart -- and, internally, its core) along with Msg_out and Msg_in
49 * essentially *are* the ipc::transport structured layer. We also
50 * suspect that (and designed ipc::transport accordingly) most users of ipc::transport, or arguably even Flow-IPC
51 * at large, will mostly care about the work they can do with struc::Channel and nothing else. Users just
52 * want to send structured messages (and sometimes native handles) in some forward-compatible protocol. Everything
53 * else -- sessions, unstructured-layer channels, the queues and/or socket streams comprised by those channels --
54 * are the necessary but ideally-ignored-most-of-the-time *means* leading to the *ultimate end* of having a working
55 * struc::Channel over which to exchange that structured stuff with the peer's struc::Channel counterpart.
56 *
57 * ### Context (more in-depth) ###
58 * This has certain broad implications. Suppose one first familiarizes themselves with the unstructured layer
59 * (concepts Blob_sender, Blob_receiver, Native_handle_sender, Native_handle_receiver; bundling class template
60 * transport::Channel; concept impls -- classes/class templates Native_socket_stream, Blob_stream_mq_sender,
61 * Blob_stream_mq_receiver; low-level details like the MQ-related API and server class Native_socket_stream_acceptor).
62 * If one indeed is familiar with that whole unstructured API, they'll note: struc::Channel API is quite
63 * different. Particularly it's very different in terms of how one handles *in-messages* (receiving). To wit:
64 * - The unstructured APIs are roughly boost.asio-like (though in fact *not* requiring any integration with actual
65 * boost.asio). That is:
66 * - Each async operation is invoked via an `X.async_F(H)` method, to which one passes the handler `H` to invoke;
67 * and `H()` is called *exactly* once: when the async op completes (with or without error); or `X` is
68 * destroyed first (in which case an `operation_aborted`-like error code is emitted to `H()`).
69 * - If more than one `X.async_F()` is invoked, for a given `F` (e.g., 3 Blob_receiver::async_receive_blob()
70 * outstanding calls on one Blob_receiver), then they are queued inside X as needed, with each arriving result
71 * firing 1 handler, dequeuing it in FIFO order.
72 * - If a pipe-hosing error occurs, it is fed to all queued handlers (i.e., the internal queue is cleared)
73 * and any future-invoked-async-API-passed handlers also.
74 * - By contrast: The structured (struc::Channel) API is not at all boost.asio-like.
75 * - There are no `async_F()` async ops as such. There is no FIFO-queueing of handlers. Instead:
76 * - One registers events handlers for each type of event in which one is interested. While it's possible
77 * to auto-unregister an event handler after it fires once, the general case is that an event handler
78 * registration is permanent until explicitly undone.
79 * - As a sub-case of this paradigm, the act of sending an out-message can be optionally paired with
80 * registering an event handler for when (and if) the other peer responds to that specific out-message
81 * with a response in-message. Thus request/response semantics (while optional) are built-in.
82 * - As a sub-case of this paradigm, a single error handler is registered; if a pipe-hosing error occurs,
83 * then that handler is invoked just once -- as opposed to invoking all the other stored handlers with
84 * that same #Error_code.
85 *
86 * As for the *out-message* (sending) API: It is actually extremely similar to the unstructured layer's
87 * already admirably simple API. To recap:
88 * - If you want to send a message, you invoke send() or async_request(). Each is non-blocking *and* synchronous
89 * *yet* it cannot fail due to a would-block condition. There is no asynchronous send op!
90 * - Alternatively use sync_request() which is a blocking, optionally subject to timeout, version of
91 * async_request(). It will block until a one-off response is received to the sent message
92 * (or timeout, or error) and return that response synchronously (if indeed successful).
93 * - Note that, while sync_request() is blocking, it can be used for very quick request-response exchanges;
94 * indeed that is its main use case. For example, a memory-allocation op might contact the memory-allocating
95 * process and return once that process has acknowledged completing the alloc request.
96 * - Graceful closing is forwarded to the unstructured layer. The Blob_sender, Native_handle_sender,
97 * transport::Channel async_end_sending() API is mimicked by simply passing-through to the owned #Owned_channel.
98 * - This features an asynchronous component. Discussion omitted here; just see async_end_sending().
99 *
100 * However the out-message (sending) API adds an important (optional but highly encouraged) capability:
101 * - Every out-message can be allocated in SHM (shared memory) instead of heap; send() (et al) then transparently
102 * sends (copies into transport, on other side copies out of transport) a tiny *handle* to the message.
103 * - This is enabled, or disabled, for the entire `*this` via compile-time parameters and ctor args.
104 * After that the API is completely identical regardless of whether this full *zero-copy* mode is in use
105 * or not.
106 * - An out-message can be reused and sent again. It can be modified in-between. It would be practical to use
107 * this technique to store complex data structures integral to your application as opposed to a mere messaging
108 * focus. This does not *require* the SHM-backed (zero-copy) mode, but it greatly improves performance
109 * when sharing with another process.
110 * - This is an alternative to storing a C++ data structure, such as a `vector` of `map`s of ..., in SHM
111 * manually and transmitting a handle manually. Discussion of details of this is outside the scope
112 * here, but you should be aware that it's the alternative approach for structured-data storage in SHM
113 * and its IPC. They each have their pros/cons.
114 *
115 * Long story short:
116 * - Sending is simple as in the unstructured layer.
117 * - However zero-copy backing of the structured message's serialization in SHM allows for far improved
118 * perf.
119 * - Receiving follows an event-handler-registration model, instead of the unstructured layer's preferred
120 * async-op-and-handler model (as inspired by boost.asio).
121 *
122 * ### Overview: How to use it, why to use it ###
123 * How to use this thing? Broadly speaking:
124 * -# One first establishes a channel connection (obtains a transport::Channel in PEER state). See
125 * transport::Channel. Attention: You will need a `sync_io`-pattern-peer-bearing transport::Channel
126 * (which is enforced at compile-time); formally `Owned_channel::S_IS_SYNC_IO_OBJ == true` is required.
127 * ipc::session emits such `Channel`s. If you are manually assembling a `Channel` (or alias or
128 * data-less sub-class), then attaining fresh PEER-state peer objects of the `sync_io`-pattern variety
129 * should be equally easy. In the case of Native_socket_stream specifically, you can perform
130 * async-I/O-pattern `.async_connect()`, then `.release()` to get the `sync_io` core.
131 * -# One selects the capnp message schema they wish to use. In other words, what kinds of messages does
132 * one want to send? This determines the choice of the #Msg_body template parameter.
133 * -# One selects a technique as to where/how out-messages (and consequently in-messages) will be stored in RAM:
134 * SHM (for full zero-copy performance) or heap (for zero-copy on either side of the low-level transport)..
135 * (Details omitted here, but for the sake of this brief recap let's consider this a formality.)
136 * -# One then constructs the struc::Channel, passing the transport::Channel (and the config from previous bullet
137 * point) into ctor -- which is moved-into the resulting `*this`.
138 * -# One then exchanges messages (and, if desired, native handles) with the matching struc::Channel on the
139 * other side via the `*this` transmission API.
140 * -# When `*this` is destroyed, so is the owned transport::Channel (and no earlier).
141 *
142 * So struc::Channel "eats" (via `std::move()`) a presumably fresh (no traffic exchanged) transport::Channel and all
143 * further work on the transport::Channel is through struc::Channel.
144 *
145 * What does struc::Channel add, broadly speaking, that transport::Channel does not already have? Why even bother?
146 * Answer:
147 * - Firstly, each message no longer consists of an (optional unstructured binary blob, optional #Native_handle) pair.
148 * Instead if consists of a (structured capnp-schema-based message, #Native_handle) pair, where the latter is
149 * optional. In other words this class adds the (mandatory) ability to send *structured* data instead of mere
150 * binary blobs. The schema language of choice is Cap'n Proto (capnp), which allows for zero-copy perf
151 * *at least* until blobs enter the transport and after they exit the transport.
152 * - By choosing to use SHM-based serialization storage for out-messages, zero-copy perf can be extended
153 * end-to-end, meaning only a short handle is ever copied into the transport and out of it on the receiving
154 * side.
155 * - Secondly it establishes some convenient protocol notions on top of the basic message-boundary-respecting/reliable
156 * functionality of transport::Channel.
157 * - Message type (a/k/a #Msg_which): While #Msg_body can specify an essentially arbitrarily complex schema for
158 * what messages can be sent, with `*this` API being generally agnostic as to its details, it does establish
159 * one simple standard: The message schema #Msg_body *must* feature a top-level anonymous (capnp) *union*, with
160 * the "which" enumerator specifying the type of message that it is. E.g., something approximating HTTP might
161 * have the top-level union specifying either a GET, POST, or CONNECT message type, with each one then featuring
162 * its own arbitrary capnp `struct` sub-schema, specifying what data come along with that type of message.
163 * - *Message type demuxing*: Based on this, one sets separate handlers for each individual value for
164 * an in-message's #Msg_which.
165 * - One-off demuxing: One can register the expectation for at most 1 message of a particular #Msg_which type.
166 * Then, when it is received, the expectation registration for that #Msg_which is auto-removed, as the
167 * user-supplied handler fires.
168 * - Open-ended demuxing: Alternatively one can set a permanent (unless explicitly unregistered later)
169 * expectation for a given #Msg_which. Then the user-supplied handler will fire each time a matching
170 * in-message arrives.
171 * - (Queuing) Any messages that have no expect-this-`Msg_which` handler registered are queued inside `*this`.
172 * Registering a demux expectation later will immediately feed matching queued such in-messages, as if
173 * they'd arrived right then.
174 * - *Request/response demuxing*: Any message one sends can optionally be specified to be expecting response
175 * in-message(s). This is the alternative to #Msg_which-based demuxing. This establishes a basic
176 * request/response paradigm.
177 * - One-off response demuxing: One can specify that the given out-message expects up to exactly 1 response.
178 * - Open-ended response demuxing: One can instead specify that any number of responses may arrive
179 * (until one explicitly unregisters this expectation).
180 * - (Note) If a response in-message arrives, while `*this` has no response expectation registered, it is
181 * an error. However it is not necessarily a fatal error: There is a mechanism allowing the user
182 * to deal with this situation should it arise. However a properly designed protocol would eliminate this
183 * possibility in practice (informally speaking).
184 * - A single full-duplex pipe: transport::Channel, one notes, can be a bundle of up to 2 full-duplex pipes (though
185 * at least 1 is of course required): one for blobs only; one for blob/`Native_handle` combos.
186 * The 2 do not interact; therefore if one were to use both in parallel, various questions arise as to how
187 * (and why!) to use this functionality. struc::Channel elides these details completely and deals with
188 * them internally. Bottom line:
189 * - struc::Channel API models a single full-duplex pipe capable of transmitting messages, each of which
190 * is *either* just a structured message *or* a structured message plus #Native_handle. (The latter capability,
191 * naturally, is removed, if #Owned_channel lacks a handles pipe.) Internally the transport::Channel pipes are
192 * leveraged in such a way as to maximize performance.
193 * - Messages arrive in the same order as they are sent (as determined by the order of send() (et al) calls on the
194 * sender `*this`).
195 * - If a message arrives that has no event handler registered, then it is queued inside `*this`, as explained
196 * above. They are queued, and therefore popped if/when a handler is registered, in the same order as
197 * they arrived.
198 * - Suppose message with sequence # 5 has arrived but had to be queued due to lacking a registered handler.
199 * Suppose message with sequence # 6 has arrived and *does* have a registered handler. Message 6 will
200 * then be emitted to the user; in other words the act of being queued does not preclude the emission of
201 * subsequent messages.
202 * - Thirdly, there are some (mandatory) safety features. These are almost completely hidden from the user in
203 * regular operation. To the extent they are not, it has to do with the *log-in* feature. This is discussed
204 * separately below. Typically one need not worry about it.
205 *
206 * ### How to use it (deep-dive) (log-in excluded) ###
207 * Naturally there are two directions of transmission, sending out-messages and receiving in-messages. About the
208 * former first:
209 *
210 * To transmit an out-message:
211 * - Call create_msg(); this yields a cheaply-`move()`able out-message:
212 * #Msg_out. (See Msg_out doc header for formal public API.) You may also construct this
213 * directly -- even without a struc::Channel in existence at all; create_msg() merely helps by
214 * supplying the serialize/deserialize parameters (a/k/a builder/reader config) for you.
215 * - Fill out its contents: `M->body_root()` is a `Msg_body::Builder*`, where #Msg_body is determined by your
216 * choice of schema (`Message_body` template param); and `Msg_body::Builder` is the capnp-generated
217 * mutating API. For example: `M->body_root()->initCoolRequest().setCoolField(57);`
218 * - sets the message type to `Msg_body::Which::COOL_REQUEST` and initializes a blank `CoolRequest` capnp-struct;
219 * - mutates the field `coolField` -- presumably an integer of some kind in this example -- to equal 57.
220 * - Send it: `send(M)` or `async_request(M)`. Each is non-blocking, synchronous, and cannot would-block.
221 * - If `M` is a response to some in-message `I`, use `send/async_request(M, I, ...)`. Otherwise use
222 * `send/async_request(M, nullptr, ...)`.
223 * - If you expect *response* message(s) to `M`, use `async_request(M, ..., &id, H)`, where:
224 * - Set `&id` to `nullptr` if one expects at most 1 response to `M`; otherwise non-null.
225 * - In the latter case optionally use `undo_expect_responses(id)` to indicate no further responses
226 * to the out-message instance sent by that async_request() call are expected.
227 * - `H` is the handler for the reponse(s); `H(I)` shall be invoked, where `I` is the response in-message.
228 * - Alternatively to async_request(): `sync_request(M)`. This is similar, but instead of returning it
229 * awaits the response and returns that response once received.
230 * Note that by definition such a request is one-off (exactly 1 response is expected). sync_request() can
231 * optionally specify a timeout.
232 * - Note: See async_end_sending() doc header. TL;DR: You should call this when done using `*this`; then destroy
233 * `*this` once the async-handler you passed to async_end_sending() has fired.
234 *
235 * As usual, the incoming direction is somewhat more complex. It is also different from lower-level APIs
236 * such as boost.asio sockets and our various `Blob_sender`s and the like.
237 *
238 * Internally, all in-messages that do arrive on the "wire" are received immediately; each in-message at that point
239 * is then handled in exactly one of the following ways. (Here we do not discuss pipe-hosing errors, yet.)
240 * A given in-message `I` is handled thus:
241 * - If `I` is not a response to an earlier out-message `M`:
242 * - If `I` is being expected via an earlier expect_msg() or expect_msgs() call: *Emit* `I` to that user handler.
243 * - Else: queue it inside `*this` silently.
244 * - Else:
245 * - If `I` is being expected via a response-expectation, as registered in async_request() or
246 * sync_request() (see above):
247 * *Emit* `I` to that user handler, or return it via sync_request() synchronously (whichever applies).
248 * - Else: it is a non-pipe hosing error condition. While you should (informally speaking) avoid this, if
249 * it does occur, you can handle it as you see fit. See set_unexpected_response_handler().
250 * See also set_remote_unexpected_response_handler(). Spoiler alert: This condition is emitted on both
251 * sides: locally, indicating an unexpected response arrived; and remotely, indicating that side *issued*
252 * an unexpected response.
253 *
254 * Hence there are exactly 3 ways to register a handler for in-messages.
255 * - Call expect_msg() or expect_msgs() to expect a particular type of *unsolicited* (non-response) message.
256 * expect_msg() indicates up to 1 such message shall be expected; expect_msgs() indicates 0+ such messages,
257 * until undo_expect_msgs().
258 * - If expect_msg() or expect_msgs() is invoked, and matching message(s) are queued inside `*this` (per above)
259 * at that time, `*this` behaves as-if those messages had just arrived. I.e., they are immediately
260 * *emitted*.
261 * - Call async_request() to expect a response to a particular out-message.
262 * - Call sync_request() to send request and synchronously await one-off response to that particular out-message.
263 *
264 * Example:
265 * - I am a server, and I expect GET and POST messages. I invoke `expect_msgs(Msg_which_in::GET_REQ, handle_get_req)`
266 * and `expect_msgs(Msg_which::POST_REQ, handle_post_req)`. `handle_get_req(x)` shall be invoked for each
267 * incoming GET_REQ `x` and similarly for the POSTs.
268 * - I am a client, and I issue GET and POST messages. I invoke `async_request(x, ..., nullptr, handle_get_rsp)`
269 * for a GET and `async_request(y, ..., nullptr, handle_post_rsp)`, where earlier one did:
270 * - `x = ....create_msg()`; `x->root_body->initGetReq().set{Url|HostHeader}(...)`;
271 * - `y = ....create_msg()`; `y->root_body->initPostReq().set{Url|HostHeader|PostBody}(...)`.
272 *
273 * Last but not least: How does one read the contents of an incoming message? For both paths -- unsolicited message
274 * expectation via expect_msg()/expect_msgs() + response message expectation via async_request() alike -- the
275 * user-provided handler shall be of the form: `H(I&&)`, where `I` is a #Msg_in_ptr, a ref-counted handle to a new
276 * in-message. (See Msg_in doc header for the formal public API.) Note the `&&`: it is an rvalue reference
277 * to the `shared_ptr`, rather than a `const &`, in case you'd like to `move()` it somewhere for perf instead of
278 * ref-counting increment/decrement.
279 * - `I->body_root()` is a `const Msg_body::Reader&`, where #Msg_body is as explained earlier, and
280 * `Msg_body::Reader` is the capnp-generated accessor API.
281 * For example: `I->body_root().getCoolRequest().getCoolField()` shall equal 57 on the other end, following
282 * the earlier out-message example.
283 * - In the case of unsolicited messages: `I->body_root().which()` shall equal the #Msg_which_in passed
284 * to expect_msg() or expect_msgs(). Therefore one can perform `I->body_root().getCoolRequest()` without
285 * fear it's not actually a `COOL_REQUEST` (which would yield a capnp exception).
286 * - In the case of response messages: `I->body_root().which()` can be anything (i.e., `*this` will not enforce
287 * anything about it). It is up to you to design that protocol.
288 * - I (ygoldfel) considered enforcing a #Msg_which_in expectation for responses, to be supplied with
289 * async_request(). However it seemed too restrictive to mandate it.
290 * @todo Consider adding the *optional* expectation of a particular #Msg_which_in when registering expected
291 * responses in struc::Channel::async_request().

293 * ### Thread safety; handler invocation semantics ###
294 * In-message handlers, and all other handlers (including those from start(), set_unexpected_response_handler(),
295 * set_remote_unexpected_response_handler(), async_end_sending()), are invoked from an unspecified thread guaranteed not
296 * to be among the user invoking threads. No 2 such handlers shall be invoked concurrently to each other.
297 *
298 * Messages are emitted in the order received, except reordered as required by the queueing of not-yet-expected
299 * unsolicited messages (as explained earlier).
300 *
301 * It is not safe to invoke a non-`const` API concurrently with another API on the same `*this`.
302 *
303 * You *may* call `*this` APIs directly from any in-message handler or other handler.
304 * Exception: You may *not* invoke sync_request() (or any other `sync_*()` that may be added over time)
305 * from an in-message handler or other handler. Doing so can disrupt the timeliness of delivering
306 * other async results.
307 *
308 * Informal recommendation:
309 * You should off-load most or all handling of in-messages onto your own thread(s) (such as via boost.asio
310 * `post()` or the flow.async infrastructure). For example, suppose your program has some main worker thread U,
311 * implemented as a `flow::async::Single_thread_task_loop U`. Then, e.g.:
312 * - In thread U: `async_request(M, false, H)`:
313 * sends `M`; async-calls `H(I)` when response `I` to `M` arrives.
314 * - In (unspecified thread): `H(I)` fires. Your `H()` body:
315 * `U.post([I = std::move(I)]() { handle_i(std::move(I)); })`.
316 * - Actual handling of `I` occurs in your `void handle_i(...::Msg_in_ptr&&) { ... }`.
317 * `handle_i(){}` can be written with the assumption execution is back in thread U, same as the async_request()
318 * call that triggered all this (asynchronously).
319 *
320 * This is *not* a requirement. It is entirely conceivable one would want to piggy-back onto our internal
321 * unspecified thread (or threads -- formally speaking -- but the guarantee is handlers are called
322 * non-concurrently, so in most senses one can treat it as a single thread). However please be mindful:
323 * - If your handler code is "heavy," then you may slow down internal transport::Channel and struc::Channel
324 * processing on a number of layers (though, only relevant to `*this`).
325 * - Let's say we call your own thread, thread U, and the internal/unspecified `*this` thread where
326 * it invokes handlers, thread W. Then it is your responsibility to ensure you do not call
327 * a non-`const` method of `*this` concurrently to another `*this` method. If you choose a design
328 * where you make such calls both from thread U and thread W, you must protect `*this` access
329 * with a mutex or other mechanism.
330 *
331 * ### Pipe-hosing error semantics ###
332 * The underlying transport::Channel, which every struc::Channel takes over in the first place at construction,
333 * can fail -- encounter a pipe-hosing error. Briefly speaking, a transport::Channel out-pipe can -- depending
334 * on the actual low-level transport involved -- be hosed while the in-pipe(s) remain unaffected; and indeed when
335 * a transport::Channel internally consists of 2 bidirectional pipes, it is even possible for 1 out-pipe to be hosed,
336 * while the other is not. (The same holds of in-pipes.) However, these details strongly depend on the actual
337 * low-level transports involved. To simplify error handling, these details are hidden away and normalized to
338 * the following straightforward behavior.
339 *
340 * As noted before, struc::Channel models a single bidirectional full-duplex pipe of discrete messages.
341 * The out-pipe is controlled simply: call send(), async_request(), or sync_request() to send a message, synchronously,
342 * non-blockingly (if not sync_request()).
343 * The in-pipe takes in all messages as soon as they arrive on the transport::Channel "wire"; they may be queued inside
344 * or immediately (or later) emitted, but this is not relevant to error handling.
345 *
346 * The entire bidirectional pipe is either not-hosed, or it is hosed (forever). A pipe-hosing condition,
347 * indicated by a truthy #Error_code, occurs (if it occurs at all) exactly once. (Note: Receiving the
348 * `transport::Channel`-level graceful-close message -- transport::error::Code:: S_RECEIVES_FINISHED_CANNOT_RECEIVE --
349 * is not "special" -- it is also a pipe-hosing error.) When it occurs, it is immediately emitted to the
350 * `*this` user. There are 2 possible ways this can occur (again: only 1 of these can occur per `*this`):
351 * - Triggered by send(), async_request(), or sync_request(): Then that method shall emit the truthy #Error_code
352 * directly/synchronously via out-arg or exception.
353 * - Triggered spuriously in the background via incoming-direction processing: Then it shall be emitted via
354 * the on-error handler registered by the user. (If this occurs during sync_request(), then it shall be
355 * emitted via sync_request(), not via on-error handler.)
356 * - To receive any in-messages, you must call start() which takes the mandatory on-error handler as arg.
357 * Until then, any lower-level in-traffic shall be safely queued inside the #Owned_channel. Still, it is
358 * best to call start() early, to avoid RAM backup and such (informally speaking).
359 *
360 * After the pipe-hosing error is emitted, via exactly 1 of the above methods, further API calls on `*this` shall
361 * no-op and return `false` or a null value.
362 *
363 * Lastly, there is async_end_sending(). See its doc header. TL;DR: Although the pipe is hosed at the
364 * struc::Channel level once a pipe-hosing #Error_code is emitted, it is still a good idea to execute
365 * async_end_sending() and only destroy `*this` once the async-handler passed to it has fired. (Though, if
366 * send() (et al) emitted the truthy #Error_code, there is no point: async_end_sending() will no-op.)
367 *
368 * Informal recommendations:
369 * - Recall that we recommend any event handling, from within an in-message handler or any other handler,
370 * be immediately off-loaded onto a user worker thread. (In particular you may not call `*this` APIs from these
371 * handlers.)
372 * - When such a user-thread handler, or other user-thread function, detects the emitted truthy #Error_code
373 * (from on-error handler, or via send(), async_request(), or sync_request() emitting it synchronously), mark
374 * `*this`, somehow, as hosed (e.g., via a flag or saved truthy #Error_code).
375 * - When about to do work on `*this`, check that flag or mark, so as to be aware `*this` is hosed and shutting
376 * down -- and no-op instead of trying to call whatever API. Though, it may be viable to simply count on
377 * the fact `*this` APIs shall all (except async_end_sending()) no-op and return `false` or null in that case
378 * anyway.
379 * - Lastly, recommend then calling `async_end_sending(F)`; and in `F()` destroy `*this` finally.
380 * - If you have 1 thread (e.g., via `flow::async::Single_thread_task_loop`) dedicated to processing `*this`,
381 * then this is particularly straightforward; no need to synchronize on `*this` and/or the `*this`-is-hosed
382 * flag/mark.
383 *
384 * ### Lifecycle of an out-message ###
385 * #Msg_out represents the message payload. It is a data structure instance -- nothing more. Its outer shell sits
386 * wherever it is instantiated; typically either on the stack or in regular heap. It allocates space for the
387 * the serialization mutated by its `.body_root()` capnp `Builder` based on the `Struct_builder_config` type
388 * chosen at compile-time. (If you use create_msg(), this light-weight object -- essentially a few scalars -- is
389 * supplied to the `Msg_out` ctor for you. Otherwise you have to construct one directly and pass-in a
390 * `Builder_config` or `Builder` of the same type.) This builder *engine* determines where this internal
391 * bulk -- the serialization backing -- lives. For the rest of this discussion let us assume it is
392 * `shm::classic::Builder::Config` (or jemalloc equivalent); meaning the engine is zero-copy-enabling,
393 * for high perf and other benefits, and therefore allocates backing buffer(s) in SHM.
394 *
395 * #Msg_out, therefore, is similar to a container backed by a SHM-allocating allocator. It continues to take RAM
396 * resources, and the serialization continues to be accessible, on all ends until *both* of the following occur:
397 * - The #Msg_out is destroyed.
398 * - Every #Msg_in_ptr obtained in *any* opposing struc::Channel to have received it (via expect_msg(),
399 * expect_msgs(), and response-accepting handlers), after it had been send()ed (et al) there, is destroyed --
400 * along with all derived `Msg_in_ptr`s. (#Msg_in_ptr is `shared_ptr<Msg_in>`.)
401 *
402 * Suppose you create a #Msg_out and send() it. At this point there is "virtual" #Msg_in in existence:
403 * the underlying serialization will not be destroyed yet, even if #Msg_out is. Normally the opposing
404 * struc::Channel will be used to receive the message, in a real #Msg_in, via an aforementioned in-message
405 * handler. If this does not occur, the message leaks until the containing SHM arena is cleaned-up.
406 * (If you use ipc::session to manage SHM arena(s), which is the normal scenarios, then ipc::session will take
407 * care of this for you. See ipc::session documentation, starting with the namespace doc header.)
408 *
409 * Suppose in your application, at this point, you will never again write to the #Msg_out further (modify it).
410 * You may freely send() the same #Msg_out again. You may send it through a different struc::Channel or
411 * the same one. You may do so at will. You may receive it, obviously, at will as well. Note, however, that
412 * #Msg_in is not fully symmetrical to #Msg_out: It represents not the container but rather the in-message
413 * *instance*: that particular time you received a message, with a handle to the payload controlled by
414 * the original #Msg_out. So for one #Msg_out, there may be 2+ `Msg_in`s in existence, across various processes
415 * to have received it. If one process has received it 2x, then there are potentially 2 `Msg_in`s in existence
416 * in that process. The original #Msg_out and each #Msg_in each contributes 1 to the cross-process
417 * ref-count.
418 *
419 * Lastly suppose all the same is true, except that you would like to continue modifying the #Msg_out post-`send()`.
420 * This is allowed. However, this must be done with care. Of course there is the matter of
421 * synchronization as usual: Behavior is undefined if one reads a message while the #Msg_out is being mutated
422 * in the original process. While it is conceivable one could use an outside synchronization mechanism for this --
423 * an interprocess mutex for example -- informally we expect the best way is to use your struc::Channel
424 * messaging protocol. In particular it make sense to avoid any modification of a #Msg_out until one has received
425 * a response to any async_request() of it, with the guarantee that the module receiving that *particular* #Msg_in
426 * in the opposing process shall not further read its contents after issuing the async_request() for that response.
427 *
428 * Lastly suppose `Struct_builder_config` is *not* SHM-based; e.g. Heap_fixed_builder::Config.
429 * In that case, simply, #Msg_out is the original message container backed by an in-local-heap serialization;
430 * and every #Msg_in emitted by a receive-handler is a *copy*, also backed by an in-local-to-receiver-heap
431 * serialization. While #Msg_out can certainly be send()ed repeatedly, and modified in-between,
432 * this simply has no effect on any resulting `Msg_in`s, as they are copies.
433 *
434 * ### Log-in ###
435 * Internally, each out-message is supplied by `*this` with a *session token*. In regular operation, every
436 * out-message's session token is the same value through the lifetime of a `*this`. Every in-message must also
437 * contain that same session token value. If an in-message arrives with an unexpected session token value, it
438 * is a pipe-hosing error. (The rationale for this is beyond the scope of discussion here; it is a safety feature.)
439 * This all happens internally.
440 *
441 * In regular operation, at struc::Channel construction time, the constant session token value must be supplied
442 * as a ctor argument. This session token must not be nil. While, formally speaking, any non-nil value will do --
443 * provided the same one is given to the struc::Channel on the other end -- correct use (in terms of safety)
444 * is to provide a properly generated unique token -- on both sides. Where to get this value, though? Answer:
445 *
446 * Before constructing any regular-operation `struc::Channel`s, in a given session between two processes A and B,
447 * one shall first construct *one* special-purpose transport::Channel, called the *session master channel*, and wrap it
448 * in a special-purpose struc::Channel on each side. This is the 2nd ctor form, the one without a `session_token`
449 * arg; one side must pass the value `is_server = true`; the other conversely `is_server = false`.
450 *
451 * This special channel is not yet in logged-in phase; it is in logging-in-as-client phase on 1 side and
452 * logging-in-as-server phase on the other side. The two `struc::Channel`s must then undergo a rigid, but short,
453 * sequence of steps to get to the logged-in phase (at which point the session master channel enters its own
454 * regular-operation phase; the purpose of that channel from that point on is unspecified here -- but read on).
455 *
456 * Namely: the client peer must create a message as normal (create_msg() or #Msg_out construction),
457 * fill out the returned out-message as desired, and async_request() it.
458 * The server peer conversely must invoke expect_log_in_request(), exactly once, handle the incoming request
459 * by filling out a response out-message, and send() that (response-supplying form is mandatory).
460 * - The server peer `*this` enters logged-in (regular-operation) phase upon `send()`ing the log-in response.
461 * Did you invoke send() successfully? Congrats: you're in regular-operation.
462 * - The client peer `*this` enters logged-in (regular-operation) phase upon receiving that log-in response
463 * Did your handler for the response get fired? Congrats: you're in regular-operation.
464 *
465 * While this mechanism is fairly general, and hopefully reasonably easy to operate, in practice most users will
466 * not need to ever do so themselves. Instead: On the server side, establish a `Server_session` (see
467 * session::Session_server). On the client side, establish a `Client_session`. They will operate an adequate
468 * session master channel internally. `Server_session` and `Client_session` shall then each reach their PEER state,
469 * at which point they each satisfy the formal session::Session concept (see session::Session doc header). Now,
470 * session::Session::session_token() shall return the uniquely generated session token. This value shall be passed to
471 * all subsequent `struc::Channel`s in the session, into their ctors (possibly via internal delegation).
472 * These subsequent `struc::Channel`s do not need a log-in phase and are immediately in regular-operation.
473 *
474 * Therefore, the only time a user must worry about session tokens at all is when creating a regular-operation
475 * struc::Channel, and even then only if no tag-form struc::Channel ctor is applicable. In that case:
476 * - Access session::Session::session_token().
477 * - Pass the result into struc::Channel ctor on each side.
478 *
479 * Incidentally, `Session` (as implemented by `Server_session`, `Client_session`, and variants) can also easily open new
480 * `Channel`s. Internally they do so by leveraging various features of struc::Channel as applied to the *one*
481 * (per process A-B session) session master channel. Each `Channel` obtained this way can then be fed (upgraded) into
482 * a new struc::Channel, if indeed structured messaging is desired for that transport::Channel. The information in
483 * this paragraph is, formally speaking, outside the purview of struc::Channel API, but I mention it here
484 * for context. For more detail see session::Session (and its impls session::Server_session, session::Client_session
485 * and variants).
486 *
487 * @todo struc::Channel should probably be made move-constructible and move-assignable. No concept requires
488 * this, unlike with many other classes and class templates in ::ipc, so it is less essential; but both for
489 * consistency and usability it would be good. It would also make some APIs possible that currently would require
490 * the user to explicitly wrap this class in a `unique_ptr`. For example, imagine
491 * a `Session::Structured_channel Session::structured_channel_upgrade(Channel_obj&& channel, ...)` that
492 * constructs a suitably-typed struc::Channel, subsuming the raw `channel` just opened in that session::Session, and
493 * returns that guy. Currently it would need to return `unique_ptr<Session::Structured_channel>` or something.
494 *
495 * @internal
496 * ### Implementation overview (threads U, W) ###
497 * struc::Channel sits atop several layers of concepts and objects implementing them: from the raw
498 * transports (transport::asio_local_stream_socket namespace, Persistent_mq_handle concept and impls); to
499 * the component Blob_sender, Blob_receiver, Native_handle_sender, Native_handle_receiver impls; to the
500 * transport::Channel (#Owned_channel) bundling them together; and lastly -- and most immediately relevantly --
501 * to the struc::sync_io::Channel core. (The latter is aliased as struc::Channel::Sync_io_obj.)
502 *
503 * In implementing struc::Channel, then, we already rely on much logic that is abstracted-away; we only
504 * need to "decorate" their work by, essentially, some pre- and post-processing -- to provide, specifically,
505 * the features explicitly outlined above publicly ("What does struc::Channel add, broadly speaking, that
506 * transport::Channel does not already have?").
507 *
508 * Perf, in particular low latency, is essential for this work-horse class template. For performance -- or perhaps
509 * more accurately for most direct/deterministic flow control/thread structure --
510 * we use `sync_io` objects throughout; our #Sync_io_obj core (`m_sync_io`) and the #Owned_channel subsumed by it
511 * create no handler-invoking threads. We, however, are ourselves an async-I/O-pattern object, and therefore
512 * essentially by definition of how that pattern works we start a worker thread W in which to (1) perform all
513 * asynchronously-delayed (i.e., contingent on event async-waits) work of all the aforementioned layers below
514 * us (helpfully encapsulated in #Sync_io_obj `m_sync_io`); and (2) invoke user-supplied completion handlers.
515 *
516 * This thread W (`m_worker`) is the only async-worker thread. Everything else is (to reiterate) `sync_io`-pattern.
517 *
518 * Thread U is (as per usual) the term of the user's calling thread (as usual they may of course use 2+ threads if
519 * desired; however on a given `*this` all such calls must be non-concurrent... so as usual we call it thread U,
520 * as if it is one thread). We do allow transmission APIs to be invoked from within completion handlers of ours
521 * own other transmission APIs (e.g., `send()` or `expect_msg()` directly within `expect_msg()` handler).
522 * (sync_request(), being blocking, is the exception.)
523 *
524 * This is a familiar setup -- similar to, say, transport::Native_socket_stream. I will immediately note, however,
525 * that we do not make any allowances for concurrent execution of receive-op APIs and send-op APIs; in our case
526 * almost all outgoing-direction APIs touch data structures in common with incoming-direction APIs -- such
527 * as async_request() modifying expected-in-message `m_rcv_*` stuff.
528 *
529 * Since there are 2 threads, U and W, with async-processing in the latter potentially occurring concurrently with
530 * API calls in the former, there's a big fat mutex `m_mutex` locked around most code. That said thread contention
531 * is unlikely in most use cases. (Details omitted here.)
532 *
533 * ### Implementation ###
534 * Unlike in the past, when `*this` template was monolithic, now it is split into the core functionality
535 * in sync_io::Channel (#Sync_io_obj `m_sync_io`) and the (relatively limited) logic necessary on top
536 * of this to adapt that into the async-I/O pattern, wherein user handlers fire from thread W, and
537 * any necessary internal async ops occur automatically in thread W without user participation.
538 *
539 * Therefore the really complicated stuff is abstracted away in the #Sync_io_obj. The adapting logic is
540 * simple enough to be understandable by reading the code and inline comments, especially on the data members.
541 * @endinternal

543 * @tparam Channel_obj
544 * An instance of transport::Channel class template. See transport::Channel doc header. It must have
545 * `Channel_obj::S_IS_SYNC_IO_OBJ == true`. Reminder: the template params to
546 * transport::Channel template (along with certain documented requirements on how to use its API when
547 * initializing it before feeding it to struc::Channel ctor) determine the number and nature of underlying
548 * transport pipes of the channel. struc::Channel will work with anything you specify, as long as it
549 * results in a `sync_io`-core-bearing transport::Channel object in PEER state fed to `*this` ctor.
550 * @tparam Message_body
551 * The capnp-generated class for the capnp-`struct` describing the schema of messages you intend to transmit
552 * over this channel. (Therefore, in particular, `Message_body::Builder` and `Message_body::Reader` shall
553 * be capnp-generated classes as well.) The contents of this capnp-`struct` are arbitrary -- whatever suits
554 * one's purpose -- except for the following requirement: It must contain a top-level anon capnp-`union`.
555 * #Msg_which shall therefore be the enumeration of the possible active value of this capnp-`union`.
556 * This union conceptually corresponds to the different types of messages one might transmit over the channel.
557 * @tparam Struct_builder_config

562 * `Owned_channel::send_meta_blob_max_size()`, for the applicable 1-2 pipes in #Owned_channel.

564 * This determines how, internally, `*this` shall transform structured out-messages provided by the user
565 * into binary blobs sent via the #Owned_channel.
566 * This Struct_builder_config must match the opposing peer's choice of Struct_reader_config.
567 * E.g., if this is Heap_fixed_builder::Config, then the other guy's
568 * thingie is Heap_reader::Config.
569 * In practice it is usually best to use a struc::Channel alias (as of this writing
570 * Channel_via_heap, shm::classic::Channel, shm::arena_lend::jemalloc::Channel)
571 * and accordingly to use one of the Channel_base tags:
572 * Channel_base::S_SERIALIZE_VIA_HEAP, Channel_base::S_SERIALIZE_VIA_APP_SHM,
573 * Channel_base::S_SERIALIZE_VIA_SESSION_SHM.

578 * peer over the #Owned_channel into structured in-messages emitted to the user.
579 * This Struct_reader_config must match the opposing peer's choice of Struct_builder_config.
580 * See `Struct_builder_config` notes just above.
581 */

877 /**
878 * In the synchronous/non-blocking manner of Blob_sender::send_blob() or Native_handle_sender::send_native_handle()
879 * sends the given message to the opposing peer, as a *notification* (without expecting response).
880 * The message may contain a #Native_handle.
881 *
882 * send() shall no-op (and return `false`) if one of the following is true.
883 * - Log-in phase, as a client, has not yet been completed.
884 * Use the other form of send() instead, as a log-in response (to the log-in request `msg`) is required.
885 * - Log-in phase, as a server, has not yet been completed, and `originating_msg_or_null` is null.
886 * Use non-null instead.
887 *
888 * Further documentation of behavior shall assume this no-op condition is not the case.
889 *
890 * @see async_request() overload which sends a *request* (same thing as here but *with* expecting response(s)).
891 *
892 * @note You may use this in logged-in phase; or in logging-in phase, as a server.
893 *
894 * ### Send mechanics (applied to async_request() also) ###
895 * The purpose of this important method is hopefully clear. That it's, essentially, a wrapper for
896 * one of the aforementioned transport::Channel methods is also probably obvious. However its error-reporting
897 * semantics are somewhat different from transport::Channel and the `Blob_sender`/`Native_handle_sender` concept(s) it
898 * implements. Long story short, it reports errors similarly to the core Blob_sender::send_blob() or
899 * Native_handle_sender::send_native_handle() with the following differences:
900 * - If `msg` contains a #Native_handle, but #Owned_channel has no native-handles pipe: no-op, return `false`.
901 * - If called after async_end_sending(): no-op, return `false`.
902 * - If called after a background receive or preceding send() / async_request() / sync_request() emitted error:
903 * no-op, return `false`.
904 * - Otherwise it'll attempt the core `send_*()` and return `true`.
905 * - If that is successful, send() is successful.
906 * - If not -- if it emitted error E -- then send() emits error E.
907 * - And `*this` shall be considered hosed for any subsequent transmission.
908 *
909 * ### In more detail (applied to async_request() also) ###
910 * Here is what the method does, ignoring logged-in phase checks and potential change inside.
911 * -# If `msg` contains a #Native_handle, but Channel::S_HAS_BLOB_PIPE_ONLY is `true`, then returns `false`
912 * and otherwise no-ops. This is a misuse of the chosen #Owned_channel as configured: a handles pipe
913 * is required to send handles. Else:
914 * -# Checks whether the pipe is hosed by a preceding error (which would have been reported via the
915 * on-error handler registered by user on `*this`; or by the previous `send()`/etc. synchronously, if that's what
916 * discovered the pipe is hosed instead). If so returns `false` and otherwise no-ops. Else:
917 * (Note the previously discovered/emitted #Error_code is not re-emitted this time.) Else:
918 * -# If you've called async_end_sending() already: Returns `false` and otherwise no-ops. Else:
919 * -# Invokes either `Owned_channel::send_blob()` or `Owned_channel::send_native_handle()` 1+ times.
920 * (If Channel::S_HAS_BLOB_PIPE_ONLY: the former. If Channel::S_HAS_NATIVE_HANDLE_PIPE_ONLY: the latter,
921 * even if `msg` stores no #Native_handle (it's not required to send a handle). If Channel::S_HAS_2_PIPES:
922 * the latter or the former depending on whether `msg` stores a #Native_handle.)
923 * This determines the possible #Error_code emitted; see doc header for the chosen
924 * sender concept impls for #Owned_channel. However:
925 * - Note: transport::error::Code::S_SENDS_FINISHED_CANNOT_SEND shall not be emitted; instead send() will return
926 * `false` as noted above.
927 * - Note: Since we are in charge, internally, of invoking the low-level `send_*()` method, we shall ensure
928 * transport::error::Code::S_INVALID_ARGUMENT is not a possibility.
929 * - Serialization errors are also possible (can be thought of similarly to `send_*()` failing).
930 * -# If such a call emits an error, the present method shall forward (emit) that error (but return `true`).
931 * Moreover the pipe shall be considered hosed:
932 * - The triggering #Error_code will *not* be emitted to user via on-error handler registered on `*this`,
933 * because it will be emitted to the caller synchronously here by send().
934 * - No futher on-in-message user handlers (such as from expect_msg()) shall fire.
935 * - Any future transmission API calls will no-op indicating this in various ways:
936 * - expect_msg(), expect_msgs(), `set_*_handler()` shall return `false`;
937 * - `create_*()` shall return null;
938 * - send() shall return `false` (as noted above).
939 * -# Otherwise, cool. It shall return `true` and not emit a truthy #Error_code.
940 *
941 * @see async_end_sending() for properly flushing out-messages near the end of `*this` lifetime.
942 *
943 * @param msg
944 * The out-message, which may or may not store a #Native_handle.
945 * @param originating_msg_or_null
946 * Pointer to the in-message to which this is responding. If the in-message is from another
947 * struc::Channel, behavior is undefined. Set to null if this message is unsolicited (not a response).
948 * @param err_code
949 * See `flow::Error_code` docs for error reporting semantics. #Error_code generated:
950 * See above. Not `INVALID_ARGUMENT`, not `SENDS_FINISHED_CANNOT_SEND`.
951 * Also possible: serialization-failed errors (at least error::Code::S_INTERNAL_ERROR_SERIALIZE_LEAF_TOO_BIG
952 * unless `Struct_builder_config` is SHM-based).
953 * @return `false` if no-op-ing due to one of the following conditions:
954 * invoked in logging-in phase as client; invoked in logging-in phase as server but
955 * `originating_msg_or_null` is null; `msg` with-handle, but #Owned_channel has no handles pipe;
956 * an #Error_code was previously emitted via on-error handler or send() / async_request() / sync_request();
957 * async_end_sending() has already been called.
958 * `true` otherwise (but some #Error_code may be emitted due to send failure).
959 */
960 bool send(const Msg_out& msg, const Msg_in* originating_msg_or_null = 0, Error_code* err_code = 0);

1035 /**
1036 * A blocking (subject to timeout) operation that first acts as-if async_request() was invoked
1037 * in one-off-request mode; then awaits the one-off response and synchronously returns that reponse
1038 * (or pipe-hosing error, or timeout error).
1039 *
1040 * Unlike send() and async_request(), you *may not* invoke this method directly from an in-message handler or other
1041 * handler. Doing so can disrupt the timeliness of delivering async results and formally results in undefined
1042 * behavior.
1043 *
1044 * sync_request() shall no-op (and return null without a truthy `Error_code` emitted) if:
1045 * - log-in phase has not yet been completed.
1046 *
1047 * Further documentation of behavior shall assume this no-op condition is not the case.
1048 *
1049 * ### Send mechanics ###
1050 * As the first step this method invokes essentially async_request(). Hence the following notes apply as they do
1051 * for async_request():
1052 * Long story short, it reports errors similarly to the core Blob_sender::send_blob() or
1053 * Native_handle_sender::send_native_handle() with the following differences:
1054 * - If `msg` contains a #Native_handle, but #Owned_channel has no native-handles pipe: no-op, return null
1055 * with falsy `Error_code`.
1056 * - If called after async_end_sending(): no-op, return null with falsy `Error_code`.
1057 * - If called after a background receive or preceding send() / async_request() / sync_request()
1058 * emitted error: no-op, return null with falsy `Error_code`.
1059 * - Otherwise it'll attempt the core `send_*()`.
1060 * - If that yields a (pipe-hosing) error, then: return null with truthy `Error_code` emitted.
1061 * - And `*this` shall be considered hosed for any subsequent transmission.
1062 * - Otherwise the method moves on to the wait phase.
1063 *
1064 * The result of the wait phase shall be one of:
1065 * - non-null response returned, no truthy `Error_code` emitted (success);
1066 * - null response returned, truthy `Error_code` that is not transport::error::Code::S_TIMEOUT (pipe hosed during
1067 * wait);
1068 * - null response returned, `Error_code == S_TIMEOUT` (pipe is fine, but the wait timed out).
1069 *
1070 * ### Late-response semantics ###
1071 * If the response, or pipe-hosing error, occurs during this method's wait, then hopefully everything is clear.
1072 * Now suppose it does time out and hence emits `Error_code` transport::error::Code::S_TIMEOUT.
1073 * In the background `*this` continues to await the one-off response. (Moreover it is not possible to
1074 * "un-expect" it or expect a response some other way; as response expectation can only be specified
1075 * at async_request() or sync_request() time, and you've already done the latter for that out-message, and a one-off
1076 * request expectation by convention cannot be undone as of this writing.)
1077 *
1078 * If the response (after timeout) never arrives, there's nothing further to discuss.
1079 *
1080 * Now suppose it does arrive at some point (after the timeout).
1081 * It shall be handled as-if an unexpected response was received (albeit with slightly different args as noted
1082 * just below). Namely:
1083 * - If set_unexpected_response_handler() is in effect, it shall be invoked (with `expected_but_timed_out` set).
1084 * - The opposing peer is informed of this event, and if on that side set_remote_unexpected_response_handler()
1085 * is in effect, the opposing-side user shall be informed via that handler (again
1086 * with a flag indicating expected-but-timed-out).
1087 *
1088 * @param msg
1089 * See async_request().
1090 * @param originating_msg_or_null
1091 * See async_request().
1092 * @param timeout
1093 * If no appropriate response to `msg` occurs, nor does a pipe-hosing error, before this time elapses
1094 * sync_request() shall emit transport::error::Code::S_TIMEOUT. Note that the latter does *not* indicate
1095 * the pipe has been hosed. To disable timeout you *must* use the special value
1096 * `util::Fine_duration::max()`. (Rationale: Internally, a certain boost.thread API may use
1097 * `pthread_cond_timedwait()` internally; this function in at least some Linuxes barfs on very high values
1098 * of `Fine_duration`, as they translate into negative absolute `timespec` components. Therefore we
1099 * detect the specific desire to have unlimited timeout and in that case use un-timed wait.)
1100 * @param err_code
1101 * See `flow::Error_code` docs for error reporting semantics. #Error_code generated: See
1102 * async_request(). If the initial non-blocking send succeeds then other #Error_code possibilities
1103 * include: Any pipe-hosing error that might be otherwise emitted outside a `sync_*()` due to
1104 * in-traffic; and transport::error::Code::S_TIMEOUT. Note that the latter does *not* indicate the pipe has
1105 * been hosed.
1106 * @return Non-null response message on success within timeout; otherwise null.
1107 * If null, emitted `S_TIMEOUT` indicates non-fatal timeout; emitted other truthy `Error_code` indicates
1108 * either the non-blocking send emitted pipe-hosing error, or during the wait the pipe got hosed.
1109 * If null, but no `Error_code` is emitted, then the method was a non-blocking no-op for one
1110 * of the reasons listed above:
1111 * invoked in logging-in phase; `msg` with-handle, but #Owned_channel has no handles pipe;
1112 * an #Error_code was previously emitted via on-error handler or send() / async_request() / sync_request();
1113 * async_end_sending() has already been called.
1114 */
1115 Msg_in_ptr sync_request(const Msg_out& msg, const Msg_in* originating_msg_or_null,
1116 util::Fine_duration timeout, Error_code* err_code);

1197 /**
1198 * Operating at the unstructured layer, executes a graceful-close send by forwarding to the async
1199 * transport::Channel::async_end_sending() on the owned #Owned_channel. Reminder: Assuming not called
1200 * after async_end_sending() again: it queues a graceful-close after any other messages sent previously;
1201 * and executes `on_done_func(E)` once that graceful-close is actually sent into the low-level transport
1202 * (then `E` is falsy), or a transport error is encountered at any point, including in the past (then `E`
1203 * is truthy). Because the owned channel's lifetime is equal to that of `*this`, if `on_done_func()` has
1204 * not executed when `*this` dtor runs, it will be executed at that time with the usual
1205 * transport::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER.
1206 *
1207 * `on_done_func()` shall, as usual, execute from an unspecified thread that is not among the user calling thread(s);
1208 * and it shall not execute concurrently to any other handler invoked by `*this`.
1209 *
1210 * Formally -- that is all it does, period. Informally it is important to understand the relationship between
1211 * this call and the rest of the API which operates at the structured layer, particularly insofar as it relates
1212 * to the error semantics on that layer. (Please read the class doc header first.)
1213 *
1214 * ### Why and when to use this ###
1215 * The short answer as to when and how to use it: While optional, the best practice is: If you are done using
1216 * the channel (for any reason, whether due to error -- as reported via the error handler -- but especially
1217 * *without* any error), you should call `async_end_sending(F)`, asynchronously wait for `F()` to execute, and in
1218 * or after `F()` destroy `*this` to free resources.
1219 *
1220 * The long answer which hopefully explains the short answer is as follows. It's long, but the short answer is
1221 * sufficient arguably. So read if desired.
1222 *
1223 * Let us first consider the case where no `*this` error has occurred so far, meaning the `start()`-passed error
1224 * handler has not been invoked, and you no longer need the channel. The immediate thought might be, simply,
1225 * this means a graceful-close is in order as a matter of decent behavior, which is fair enough; although it
1226 * would not be wrong to implement your own application-layer message exchange for this (by adding these into
1227 * your #Msg_body capnp schema and sending it like any other message). However that is *not*, I (ygoldfel)
1228 * contend, the main reason `async_end_sending(F)` is a good idea in this context. That real reason: send() (et al),
1229 * externally, behaves as-if it is both synchronous and non-blocking yet cannot yield a would-block condition.
1230 * However, per Native_handle_sender and Blob_sender concept model docs, `send*()` actually may (though this should be
1231 * rare in practice) be internally asynchronous, in that it may encounter a would-block low-level transport
1232 * condition (e.g., MQ full, or local-stream-socket would-block), save a copy of the payload, and only succeed in
1233 * passing it to the out-wire later. `async_end_sending(F)` can and should be viewed as the async *out-flush*: it
1234 * ensures all the queued sends have truly succeeded -- or a transport error triggered -- and only *then* executes
1235 * `F()` asynchronously. Hence by waiting until `F()` before destroying `*this` you will have ensured that
1236 * all those care-free synchronous non-blocking sends have truly gone out; therefore `*this` can be destroyed
1237 * without accidentally cutting off the end of the conversation.
1238 *
1239 * Next let's talk about the error cases. There are roughly 2 as of this writing. One, a "soft" non-channel-hosing
1240 * situation can occur, namely information about a local or remote application-protocol error might trigger
1241 * a handler like those set by set_unexpected_response_handler() and set_remote_unexpected_response_handler().
1242 * While this may well indicate a serious problem to the user, `*this` does not consider it that way. It is up
1243 * to the user to decide what to do when such a "soft" handler executes. They could do nothing: conceivably they
1244 * coded in a loose way and consider this not an issue. Or they could consider this worth ending the conversation.
1245 * In that case, the situation is not *too* different from the non-error case discussed above. Want to end `*this`?
1246 * Flush any sends via `async_end_sending(F)`, then after `F()` destroy `*this`. At least this features less
1247 * entropy and is better, or at least not worse, than abruptly destroying the transport::Channel peer (and possibly
1248 * whole channel).
1249 *
1250 * Lastly: Two, an actual fatal error may occur. This is where there is some subtlety, due to the subtle difference
1251 * between the unstructured layer (#Owned_channel operation) and structured layer (the protocol in struc::Channel
1252 * on top of that). Suppose a channel-hosing error occurs: the `start()`-passed error handler *has* been invoked; by
1253 * definition this means `*this` can transmit no more *structured* messages in either direction; and in fact all
1254 * further APIs, with the exception of async_end_sending(), will return `false` (or similar) and no-op. However:
1255 * a channel-hosing error is (this is the subtlety!) one of 2 rough types.
1256 * -# It might be an unstructured-layer error: the #Owned_channel was trying to send or receive some blob or handle,
1257 * on behalf of `*this` internal code, and it encountered a pipe-hosing error E. E was then immediately
1258 * emitted by `*this` via the `start()`-passed error handler. Most public APIs from that point will no-op
1259 * indicating prior pipe-hosing error.
1260 * -# It might be a structured-layer error: that is the #Owned_channel is transmitting stuff fine still, but
1261 * the logical contents of what has been received is corrupted or unexpected -- maybe the other side misbehaved,
1262 * or there's a bug somewhere, maybe auth failed for some reason. Blobs and handles could continue to be sent,
1263 * but `*this` considers that bad, because the logical consistency of its world cannot be guaranteed.
1264 * In this case `*this` internally generates an appropriate E (for example
1265 * error::Code::S_STRUCT_CHANNEL_INTERNAL_PROTOCOL_BAD_AUTH) and emits it via the `start()`-passed error handler.
1266 * Most public APIs from that point will no-op indicating prior pipe-hosing error.
1267 *
1268 * Type 1 is more straightforward to think about. If a prior send emitted E due to #Owned_channel layer error E,
1269 * then async_end_sending() will simply emit E as well. If a prior receive emitted E similarly, then
1270 * async_end_sending() may re-emit E as well or not; for Native_socket_stream (wherein internally there's a single
1271 * local-socket connection for both directions despit being full-duplex -- an error in one direction hoses the
1272 * other direction) yes; for the MQs (wherein the 2 pipes are fully decoupled) no. That's not all that
1273 * straightforward already.
1274 *
1275 * Type 2 is even less straightforward. In this case #Owned_channel is fine: so async_end_sending() will yield a
1276 * successful E most likely -- but meanwhile `*this` is generally behaving as if it's hosed; and indeed it is -- just
1277 * at the higher (strucuted) layer!
1278 *
1279 * The point of all this: It is pointless to worry about how async_end_sending()'s emitted #Error_code relates to
1280 * what was emitted to the `start()`-passed error handler. It may be the same thing; it may be different; and it may
1281 * well even be a falsy (success) #Error_code! `async_end_sending(F)` -- again -- accomplishes a lower-level task:
1282 * it waits until any unsent messages are flushed out and then informs user via `F()`; and if this is at any stage
1283 * impossible (including immediately, due to prior error) then it indicates it right then via `F(E)` as well.
1284 * *They operate at different layers, even though their error may overlap depending on the situation.*
1285 * Hence, you should indeed invoke `async_end_sending(F)` from or after the error handler is invoked, then destroy
1286 * `*this` after `F()`.
1287 * - If #Owned_channel out-pipe is hosed anyway, this will simply get to `*this` destruction stage immediately,
1288 * which is fine: it's what one would do if they decide to not mess with async_end_sending() at all.
1289 * - If #Owned_channel out-pipe is fine (despite something else -- possibly in-pipe, possibly structured protocol --
1290 * being not-at-all-fine), this will in civilized fashion flush anything that might still be pending for whatever
1291 * reason -- and gracefully inform the other side that, at any rate, this peer does not want to talk anymore.
1292 * This reduces entropy; and at worst does not help anything but does not hurt.
1293 *
1294 * @todo Consider adding blocking `struc::Channel::sync_end_sending()`, possibly with timeout, since
1295 * `async_end_sending()` to completion is recommended (but not required) to execute at EOL of any struc::Channel.
1296 * The "blocking" part is good because it's convenient to not have to worry about handling completion with async
1297 * semantics boiler-plate. The "timeout" part is good because it's a best-effort operation, when in a bad/slow
1298 * environment, and blocking -- even during deinit -- would be important to avoid. For example,
1299 * the `Session` impls' dtors perform this op; we don't want those to block for any significant time if at all
1300 * avoidable. The reason it's a mere to-do currently is that a bug-free opposing struc::Channel should
1301 * not let would-block occur for any real length of time; so blocking is presumably unlikely. Nevertheless.
1302 *
1303 * @tparam Task_err
1304 * See transport::Channel::async_end_sending().
1305 * @param on_done_func
1306 * See transport::Channel::async_end_sending().
1307 * @return See transport::Channel::async_end_sending().
1308 */
1309 template<typename Task_err>
1310 bool async_end_sending(Task_err&& on_done_func);

1428 * Mutex protecting the in-message/out-message state in `*this`.
1429 *
1430 * The concurrency design is summarized in the impl section of our class doc header.
1431 *
1432 * In thread U, we generally forward to `m_sync_io.same_thing()`; and that `sync_io` core may request
1433 * an async-wait which we execute as a literaly `m_worker.async_wait(F)`, where `F()` -- from thread W now --

1435 * it is a callback `(*on_active_ev_func)()` which executes code within #m_sync_io.) Formally speaking
1436 * sync_io::Channel forbids its APIs (including that quasi-API) being involved concurrently to each
1437 * other. So we lock `m_mutex` both in thread U and within the thread-W `F()`. Informally -- `m_sync_io`
1438 * accesses many of the same data structured when within `m_sync_io.same_thing()` as within the async-wait
1439 * handler `on_active_ev_func` it passes to us to async-wait on its behalf. For example expect_msg() will
1440 * register a one-off expectation; whereas receiving an unstructured message that is decoded by
1441 * #m_sync_io as satisfying that expectation unregister that one-off expecation in `m_sync_io`'s internal map.
1442 *
1443 * ### Perf ###
1444 * It's similar to such discussions in Native_socket_stream for example. The additional items here are as follows.
1445 * - We operate potentially 2 pipes' logic in the same thread. Theoretically all else being equal handling
1446 * a blob in one could delay handling a blob+possible native handle in the other and thus add latency. However
1447 * that is not really how the #Sync_io_obj uses them: they're really doing stuff in parallel vast majority of
1448 * the time. It uses the native-handle pipe (if any!) only to transmit native handles, basically. I'll
1449 * skip the pontification about why this is unlikely to create much concurrent traffic -- leaving that as
1450 * an exercise to the reader.
1451 * - Native_socket_stream allows independent -- concurrent -- operation of the out-pipe versus the in-pipe.
1452 * E.g., `send_blob()` can run concurrently to receipt of an in-message due to `async_receive_blob()`.
1453 * In our case the #Sync_io_obj forbids any such thing, and indeed we use just one #m_mutex.
1454 * (Native_socket_stream has 2, one per direction: one in Async_adapter_sender, one in Async_adapter_receiver.)
1455 * We cannot really do otherwise. (Possibly send() -- not async_request() -- could be factored out. Maybe
1456 * look into it. Not an official to-do yet.) Does this create lock contention? It could in limited
1457 * circumstances: if the user protocol is not of ping-pong fashion but rather involving heavy downloads and
1458 * uploads concurrently. If this actually presents a problem in practice we could look into
1459 * allowing more concurrency in #Sync_io_obj and then make use of it in `*this` by adding more mutexes
1460 * and more complex logic.
1461 */

1464 /**
1465 * The current state of the blocking sync-op mechanism; a `variant` that broadly describes the state as:
1466 * no sync-op in progress, sync-op in progress sans result, sync-op in progress with result that has not yet
1467 * been returned to user (sync-op almost over). Please read on.
1468 *
1469 * ### Design of blocking sync-op mechanism ###
1470 * A `sync_*()` operation is one that synchronously awaits in-traffic of some kind and returns some result
1471 * based on the contents of this in-traffic, or an error if the pipe got hosed during the wait (in which
1472 * case that error is *not* reported to the regular on-error handler, in the same way that non-blocking send()
1473 * returns a pipe-hosing error). In addition it is subject to a (possibly infinite) user-specified timeout;
1474 * if neither a successful result or an error appears by that time, then that fact is returned by `sync_*()`.
1475 * After `sync_*()` returns, further in-pipe errors again go to the regular on-error handler until
1476 * a `sync_*()` op starts (from thread U/W).
1477 *
1478 * As of this writing there is only one such op: sync_request(). In this discussion we discuss that one
1479 * specifically (while mentally considering how it might be generalized in the future; the mechanism is designed
1480 * hopefully in such a way as to make it not-too-tough to extend to other `sync_*()`s; but we don't want to
1481 * over-engineer this and/or confuse the reader needlessly).
1482 *
1483 * Like all mutating APIs, sync_request() is never invoked concurrently with other APIs. (In addition it
1484 * is explicitly not to be invoked from thread W (i.e., directly from an on-receive handler), as that would
1485 * delay other handlers' execution; so it is to be invoked from thread U only.) This means that, broadly,
1486 * at any point in time for `*this`, there is either no sync_request() running, or there is exactly one
1487 * running -- and no other APIs are -- and moreover `*this` is safe from being destroyed.
1488 *
1489 * sync_request() first performs essentially a normal non-blocking async_request(), and if that succeeds the blocking
1490 * part begins. `m_sync_op_state->m_op_done_signal` is initialized to a promise and a future-wait (with timeout)
1491 * begins in thread U. Meanwhile in the background `*this` is operating normally,
1492 * handling all in-traffic. If it gets a successful result (namely a response
1493 * to the particular message the nb-send sent -- not some other in-traffic, which is
1494 * handled normally) while such a wait is in progress in thread U, it loads the result into #m_sync_op_state and
1495 * tickles thread U to stop waiting via `m_sync_op_done`. If it gets any pipe-hosing error, it loads that
1496 * different result and similarly tickles the `promise`. If the timeout happens first, the promise is destroyed,
1497 * and more importantly #m_sync_op_state is reset to steady state.
1498 *
1499 * (After this, an error is handled normally through on-error handler. More interestingly, suppose
1500 * a response to the formerly-expecting-response-but-now-timed-out out-message does arrive later.
1501 * That out-message ID is still registered in #m_sync_io as expecting a response due to
1502 * a sync-op; but our special internal handler for that response detects that there is no longer a sync-request in
1503 * progress, or if there is it is for a different out-message ID -- so it timed out. So it is just dropped.
1504 *
1505 * That's broadly how it works. Now specifically here are the states in the state machine. Note this machine is
1506 * protected by #m_mutex.
1507 * - Null (steady state, no sync-op in progress): #m_sync_op_state is empty/null. sync_request() is not executing,
1508 * or is running before or after the blocking part.
1509 * - In-progress, no result: #m_sync_op_state is non-null, and its Sync_op_state::m_result_if_any
1510 * holds `monostate`: sync_request() is waiting on the promise inside #m_sync_op_state, subject to a timeout,
1511 * and no result has yet been received. (Or it is just about to enter this wait having unlocked mutex,
1512 * or it had just exited it and is about to lock mutex.)
1513 * - sync_request() has permanently set Sync_op_state::m_id to equal a unique ID and captured this in the
1514 * internal response handler, so it can be checked against `m_id` later.
1515 * - In-progress, with result: #m_sync_op_state is non-null, and its Sync_op_state::m_result_if_any
1516 * holds *not* `monostate` (but rather either a truthy `Error_code` indicating error; or else the response):
1517 * sync_request() is as in the previous bullet. However, since the result is ready, the code
1518 * (in thread W) -- having detected the result condition -- will, just before unlocking the mutex
1519 * having set the result in Sync_op_state::m_result_if_any, tickle the promise, waking up the wait
1520 * in sync_request().
1521 * - Suppose sync_request() wins the race by timing out first. It will then set #m_sync_op_state to null
1522 * (and return timeout to caller, all in thread U).
1523 * - If an error occurs in thread W subsequently, we'll see there's no sync-op in progress and
1524 * report error normally via on-error handler.
1525 * - If response to the ID arrives, we'll see there's no sync-op
1526 * in progress, or a new one is in progress but with different `m_id`, so it is
1527 * a late-arrival. Then we drop it.
1528 */
1529 std::optional<Sync_op_state> m_sync_op_state;

1668{
1669 using flow::async::Single_thread_task_loop;
1670 using flow::util::ostream_op_string;
1671
1672 // We are in thread U.
1673
1674 FLOW_LOG_INFO("struc::Channel [" << *this << "]: Shutting down. "
1675 "struc::sync_io::Channel shall free various resources soon. First as needed: our "
1676 "internal async handlers (at most end-sending handler) will be canceled; "
1677 "and worker thread will be joined.");
1678
1679 /* This (1) stop()s the Task_engine thus possibly preventing any more handlers from running at all
1680 * (any handler possibly running now is the last one to run). To be clear that's one of these:
1681 * - The async_wait() handler from start_ops() (see ctor); if that's running it's deep inside a peer object
1682 * inside m_sync_io (which just forwards to its Owned_channel m_channel... black box to us but just saying).
1683 * So like for example it might updating its internal cached in-messages not yet emitted to user, because
1684 * they haven't expect_msg()ed it yet.
1685 * - We've post()ed (onto thread W -- this very Task_engine/Single_thread_task_loop) a user handler for
1686 * an expect_msg(), or the error handler given to start(), or response to a send()ed request... etc.
1687 * So that guy is running right now then.
1688 *
1689 * Anyway, this (1) will await for that to finish -- as-if the dtor was invoked just after. Moving on:
1690 * (2) at that point Task_engine::run() exits, hence thread W exits; (3) it joins thread W (waits for it to
1691 * exit); (4) returns. That's a lot, but it's non-blocking. */
1692 m_worker.stop();
1693 // Thread W is (synchronously!) no more.
1694
1695 /* As we promised in doc header(s), the destruction of *this shall cause any registered completion
1696 * handlers to be called from some unspecified thread(s) that aren't thread U, and once
1697 * they've all finished, the dtor returns. For us there's only possible such one-off handler:
1698 * from async_end_sending(). Plus... just...

1701 *
1702 * The 2 tasks, summarized are:

1713 *
1714 * We, however, don't use Mr. transport::Native_socket_stream or transport::Blob_stream_mq_sender/receiver

1720 *
1721 * My point: As the async-I/O-pattern N_s_s or B_s_mq_sender/receiver each run a thread W and thus each has to

1723 * The logic is therefore quite similar. */

1730 one_thread.start([&]()
1731 {

1734

1736 task_engine->restart();
1737 const auto count = task_engine->poll();

1739 {
1740 FLOW_LOG_INFO("Blob_stream_mq_sender [" << *this << "]: "
1741 "In transient finisher thread: Ran [" << count << "] internal handlers after all.");
1742 }
1743 task_engine->stop();
1744
1745 if (!m_snd_end_sending_on_done_func_or_empty.empty())
1746 {

1748 "Shall run pending graceful-sends-close completion handler.");
1749 m_snd_end_sending_on_done_func_or_empty(transport::error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER);
1750 } // if (!m_snd_end_sending_on_done_func_or_empty.empty())
1751
1752 FLOW_LOG_INFO("Transient finisher exiting.");
1753 });
1754 // Here thread exits/joins synchronously.

1758template<typename Task_err>
1759bool CLASS_STRUCTURED_CHANNEL::async_end_sending(Task_err&& on_done_func)
1760{
1761 /* We are in thread U, or thread W (executing from a handler but not concurrently with other APIs).
1762 * Nothing is locked regardless (see discussion in expect_msg() where we set up that handler). */
1763
1764 FLOW_LOG_TRACE("Structured channel [" << *this << "]: async_end_sending(F) invoked.");
1765
1766 /* We access m_sync_io, so gotta lock. async_end_sending() is "special," though -- so maybe it is not
1767 * required? Formally it is, but we have evil special knowledge of how Sync_io_obj works internally; indulge me.
1768 * Right at this moment an async_wait() (see inside start_ops() statement in ctor) handler may be firing
1769 * due to, say, a writable FD because of some queued-up-in-would-block payload waiting to be sent-out over
1770 * a Native_socket_stream. It may be deep inside sync_io::Native_socket_stream::Impl::snd_*() code dealing with
1771 * that. Naturally we've locked m_mutex to protect against concurrent access to that Impl's send side.
1772 * And we are about to do m_sync_io.async_end_sending(); which will call its m_channel.async_end_sending();
1773 * which will call Native_socket_stream::async_end_sending() which forward to that Impl's .async_end_sending().
1774 * That is against the contract of Native_socket_stream: you *may* call rcv-op and snd-op APIs concurrently
1775 * to each other; but *not* 2 snd-op APIs concurrently. And inside said Impl: async_end_sending() would try
1776 * to send-or-queue a graceful-close payload; but if the aforementioned async-wait-ready code inside
1777 * Impl::snd_*() is touching the relevent send-queue data structure -- kaput.
1778 *
1779 * That is to say, yes, we need to lock it. The above was for perspective of why the contract is the way it is.
1780 * I digress!!! */
1781
1782 Lock_guard lock(m_mutex);
1783
1784 if (!m_snd_end_sending_on_done_func_or_empty.empty())
1785 {
1786 FLOW_LOG_WARNING("Structured channel [" << *this << "]: async_end_sending(F) invoked; but we have a saved "
1787 "completion handler -- which has not yet fired -- for it already, so it must be a dupe-call. "
1788 "Ignoring.");
1789 return false;
1790 }
1791 /* else: It might also be .empty() and still be a dupe call (if it has fired already -- then they called us again).
1792 * m_sync_io.async_end_sending() will catch that fine. For now though: */
1793
1794 /* Save it (instead of capturing it in lambda) in case dtor runs before async-end-sending completes;
1795 * it would then invoke it with operation-aborted code. */
1796 m_snd_end_sending_on_done_func_or_empty = std::move(on_done_func);
1797
1798 Error_code sync_err_code;
1799 const bool ok = m_sync_io.async_end_sending(&sync_err_code,
1800 [this](const Error_code& err_code) mutable
1801 {
1802 /* We are in thread W (async_end_sending() specifically promises to *not* invoke on_done_func(), or
1803 * remember it in any way, if the async-end-sending is synchronously successful -- which it will be most of
1804 * the time (only not so if an out-peer object is currently in would-block)). In this case, though,
1805 * there was would-block, so we are being invoked async-style.
1806 *
1807 * m_mutex is locked!
1808 *
1809 * Should we post()? Yes, for the same reason as in expect_msg() and others (see discussion in that method). */
1810 m_worker.post([this, err_code]() mutable
1811 {
1812 // We are in thread W. Nothing is locked.
1813
1814 decltype(m_snd_end_sending_on_done_func_or_empty) on_done_func;
1815 {
1816 Lock_guard lock(m_mutex); // Accessing m_snd_end_sending_on_done_func_or_empty.
1817
1818 assert((!m_snd_end_sending_on_done_func_or_empty.empty()) && "Only we or dtor can clear this. Bug?");
1819
1820 on_done_func = std::move(m_snd_end_sending_on_done_func_or_empty);
1821 m_snd_end_sending_on_done_func_or_empty.clear(); // (Just in case move() didn't do it.)
1822
1823 // Now dtor won't call it with operation-aborted.
1824
1825 /* Unlock before invoking handler! They can call any number of mutex-locking stuff: expect_msg(),
1826 * even async_end_sending() again, if they're weird people who like to do odd things. */
1827 } // Lock_guard lock(m_mutex);
1828
1829 // This should be rare, so (another) INFO message seems okay.
1830 FLOW_LOG_INFO("struc::Channel [" << *this << "]: Now executing async-end-sending completion handler: "
1831 "async path.");
1832 on_done_func(err_code);
1833 FLOW_LOG_TRACE("Handler completed.");
1834 });
1835 }); // ok = m_sync_io.async_end_sending() (async handler path)
1836
1837 if (!ok)
1838 {
1839 // False start (dupe call). It logged enough.
1840 m_snd_end_sending_on_done_func_or_empty.clear();
1841 return false;
1842 }
1843 // else: It either finished synchronously; or it will finish asynchronously.
1844
1845 if (sync_err_code == transport::error::Code::S_SYNC_IO_WOULD_BLOCK) // It TRACE-logged enough about that.
1846 {
1847 // Async-wait needed before we can complete. Live to fight another day.
1848 return true;
1849 }
1850 // else
1851
1852 // We are in thread U, possibly (probably). Never fire handlers except from thread W.
1853 m_worker.post([this, sync_err_code]()
1854 {
1855 // We are in thread W. Nothing is locked. Similar to above; keeping comments light. @todo Bit of code reuse.
1856
1857 decltype(m_snd_end_sending_on_done_func_or_empty) on_done_func;
1858 {
1859 Lock_guard lock(m_mutex); // Accessing m_snd_end_sending_on_done_func_or_empty.
1860 assert((!m_snd_end_sending_on_done_func_or_empty.empty()) && "Only we or dtor can clear this. Bug?");
1861
1862 on_done_func = std::move(m_snd_end_sending_on_done_func_or_empty);
1863 m_snd_end_sending_on_done_func_or_empty.clear(); // (Just in case move() didn't do it.)
1864 }
1865 FLOW_LOG_TRACE("struc::Channel [" << *this << "]: Now executing async-end-sending completion handler: "
1866 "sync path.");
1867 on_done_func(sync_err_code);
1868 FLOW_LOG_TRACE("Handler completed.");
1869 });
1870
1871 return true;
1872} // Channel::async_end_sending()

1875template<typename Task_err>
1876bool CLASS_STRUCTURED_CHANNEL::start(Task_err&& on_err_func)
1877{
1878 using flow::util::Blob;
1879 using boost::movelib::make_unique;
1880 using std::holds_alternative;
1881 using std::get;
1882 using std::monostate;
1883
1884 // We are in thread U. Nothing is locked.
1885
1886 Lock_guard lock(m_mutex);
1887 /* We're accessing m_sync_io at least, so we "should" lock. It actually should not be
1888 * be necessary as of this writing: until start() there is nothing pending on thread W, and user may not
1889 * call public APIs concurrently. Better safe than sorry, though.
1890 * @todo I dunno. On one hand there's better safe than sorry, but on the other hand there's a certain purity
1891 * to only doing stuff we should be doing, by design, and no more. Revisit. */
1892
1893 return m_sync_io.start_and_poll([this, on_err_func = std::move(on_err_func)]
1894 (const Error_code& err_code) mutable
1895 {
1896 /* Though somewhat unlikely (but definitely possible -- there could be an error on the socket/whatever) --
1897 * start_and_poll() contract says any relevant handler -- including on_err_func() -- may fire *synchronously*.
1898 * This is the only such API of (our) m_sync_io. We however promise to never do that (must always use thread W).
1899 * So for that we have to post().
1900 *
1901 * Otherwise (if indeed we are already in thread W, executing from inside the async_wait() handler as seen inside
1902 * ctor's start_ops() statement, with m_mutex locked even) this adds a small, unnecessary delay -- no big deal
1903 * generally and definitely no big deal in this situation. (We could detect whether it fired synchronously and
1904 * avoid the post() if not; but it's not worth the effort.) */
1905 m_worker.post([this, err_code, on_err_func = std::move(on_err_func)]() mutable
1906 {
1907 /* We are in thread W. Nothing is locked!
1908 *
1909 * So, if not for a possible sync_request(), we'd literally just do: on_err_func(err_code). However suppose
1910 * a sync_request() is currently executing in thread U. (In the future we could add other sync_*(): E.g.,
1911 * sync_<wait for a particular message a-la expect_msg()>.) In that case we must emit the error to the sync_*()
1912 * waiting for a result instead of the on-error handler. That is our overall contract in *this: Report a
1913 * channel-hosing error exactly once (if ever): either through a send() or sync_request(); or through
1914 * this error reporting handler -- whichever one happens first. */
1915
1916 {
1917 Lock_guard lock(m_mutex);
1918
1919 /* Check that (obviously) there is a sync-op in progress; but also (carefully) ensure it does not have a result
1920 * stored already. To show this is technically possible imagine this scenario: sync_request() is waiting in
1921 * thread U; the expected response was received in time and thus placed into m_sync_op_state->m_result_if_any,
1922 * and promise m_sync_op_state->m_op_done_signal was tickled to wake up thread U's future.wait_for();
1923 * but thread U hasn't yet woken up and nullified m_sync_op_state. So now we let that sync_request() return
1924 * success; we then emit the error asynchronously. Basically it's almost as-if the error occurred just after
1925 * the successful sync_request(). */
1926 if (m_sync_op_state)
1927 {
1928 assert((!holds_alternative<Error_code>(m_sync_op_state->m_result_if_any))
1929 && "Only a previous error-handler would have set that to an Error_code; and m_sync_io shall not "
1930 "emit error more than once.");
1931 if (holds_alternative<monostate>(m_sync_op_state->m_result_if_any))
1932 {
1933 FLOW_LOG_TRACE("struc::Channel [" << *this << "]: Sync-IO core reported error (details above); "
1934 "this occurred while awaiting sync-op (ID [" << m_sync_op_state->m_id << "]) completion. "
1935 "Therefore we emit the error to that guy, loading that result and waking the waiting guy.");
1936
1937 m_sync_op_state->m_result_if_any = err_code;
1938 m_sync_op_state->m_op_done_signal.set_value();
1939 return;
1940 }
1941 // else { sync_*() already received a success result; so we let that happen. }
1942 } // if (m_sync_op_state)
1943 // else if (!m_sync_op_state) { No sync_*() in progress. }
1944 } // Lock_guard lock(m_mutex)
1945
1946 FLOW_LOG_TRACE("struc::Channel [" << *this << "]: Calling error handler from async-worker thread.");
1947 on_err_func(err_code);
1948 FLOW_LOG_TRACE("Handler completed.");
1949 }); // m_worker.post()
1950 }); // m_sync_io.post()
1951} // Channel::start()

2051TEMPLATE_STRUCTURED_CHANNEL
2052template<bool MSG_ELSE_LOG_IN, typename On_msg_handler>
2053bool CLASS_STRUCTURED_CHANNEL::expect_msg_impl(Msg_which_in which, On_msg_handler&& on_msg_func)
2054{
2055 using boost::make_shared;
2056
2057 /* We are in thread U; or thread W (if invoked from a user handler); always serially.
2058 * Nothing is locked either way due to how we actually invoke user-supplied handlers (which we are about
2059 * discuss). */
2060
2061 /* Wrap (a moved version of) their function object in a shared_ptr<Function<>>. (Perf-wise, ultimately
2062 * m_sync_io will do this anyway; we probably aren't adding much churn with this op alone.)
2063 * Why? Answer: Unfortunately we have to first pass on_msg_func to m_sync_io.expect_*(); and then possibly
2064 * also invoke it ourselves -- if message(s) immediately available synchronously. So we can't
2065 * std::move(on_msg_func), as it'll get blown away (its captures will at least) before we can use it
2066 * synchronously. We could capture a copy of on_msg_func, but that is slow: Capturing a copy of a shared_ptr<>
2067 * is muuuuuch faster. */
2068 auto on_msg_func_ptr = make_shared<On_msg_func>(std::move(on_msg_func));
2069
2070 // See below first regarding the purpose of this guy. Then come back here.
2071 auto real_on_msg_func = [this, on_msg_func_ptr](Msg_in_ptr&& msg_in) mutable
2072 {
2073 // We are in thread W. m_mutex is locked!
2074
2075 /* Now comes a key decision: We are in thread W. We must call user handler on_msg_func(). Thread W isn't
2076 * thread U: it isn't the user thread. So it would appear we could just straight-up synchronously
2077 * `on_msg_func(std::move(msg_in))`. Yet instead we post() onto thread W again -- and do it there. Why?
2078 * Answer:
2079 *
2080 * Firstly, just technically speaking, m_mutex is locked (we are right now down the call stack from
2081 * async_wait() handler inside start_ops() statement inside our ctor: it locks m_mutex before calling
2082 * (*on_active_ev_func)(). So, it being a non-recursive mutex, if they were to call expect_msg(), or
2083 * send(), or anything else like that -- which they are explicitly allowed to do -- inside such a call
2084 * we'd deadlock right inside the hypothetical synchronous on_msg_func().
2085 *
2086 * However if we post() that onto thread W, then everything is cool. So at least we know it's safe.
2087 * Is it performant? Before trying to answer that let's consider the alternative.
2088 *
2089 * Sure: we could make m_mutex recursive. Or we could not-lock it inside our various APIs (including the
2090 * expect_msg() you're reading now -- albeit the async-handler inside it), which would be sort-of a user-space
2091 * recursive mutex action. So why not? My (ygoldfel) answer is arguably hand-wavy, but it's 2-fold.
2092 * 1, in every instance of my career where I tried to or saw someone tried to solve a deadlocking algorithm
2093 * by making a recursive mutex, it was the wrong move that was reversed soon enough. Also generally designs
2094 * with recursive mutexes that at least appeared to work seemed to do so out of a desire to avoid a few lines
2095 * of boiler-plate (_impl() methods that had mutex-lock as a pre-condition), in code of quite limited size --

2103 * At the very least it is quite difficult to prove that it's *safe*, as it's so tough to reason-about due to

2106 * Okay, so then, back to post()ing it onto thread W. We know it's safe. Is it performant? The question really

2113 *
2114 * So is that delay okay? In absolute terms I would say... for now at least... yes. It'll add a few usec of
2115 * latency, as boost.asio winds up the stack into Task_engine::run(), then re-enters the post()ed handler of ours
2116 * and "finally" runs on_msg_func(). It's boost.asio flow control: it is common in boost.asio/async algorithms
