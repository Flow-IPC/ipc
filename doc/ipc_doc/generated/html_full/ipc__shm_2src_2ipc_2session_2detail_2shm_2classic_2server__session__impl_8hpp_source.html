31/**
32 * Core internally-used implementation of shm::classic::Server_session: it is to the latter what its `public`
33 * super-class Server_session_impl is to #Server_session.
34 *
35 * @see shm::classic::Server_session doc header which covers both its public behavior/API and a detailed sketch
36 * of the entire implementation.
37 *
38 * ### Impl notes ###
39 * The use of app_shm() and session_shm(), and various public APIs around them, is totally symmetrical with
40 * shm::classic::Client_session_impl; so that all is in super-class Session_impl. We do what's asymmetrical:
41 * assigning session_shm() and app_shm() values; and provide the server-specific APIs (notably
42 * async_accept_log_in()).
43 *
44 * To implement app_shm() assignment we share a member map (from Client_app to `Arena`)
45 * in the "parent" (emitting) shm::classic::Session_server -- after all it did have to create such a handle,
46 * in create-only mode, possibly even during `pre_rsp_setup_func()` (if the opposing Client_app is
47 * encountered the first time by that server). So we just access it through shm::classic::Session_server::app_shm(),
48 * passing-in the relevant Client_app; and save that pointer to be returned by app_shm().
49 *
50 * (Why jump through such hoops? Couldn't we simply `OPEN_OR_CREATE` it ourselves; after all the OS would
51 * then essentially implement the map for us in the kernel, and we needn't all the boiler-plate? Answer:
52 * That's fantastic and superior; except the shm::classic::Pool_arena thus created could not live past `*this`;
53 * which would break the contract of shm::classic::Session::app_shm() as listed in its doc header.)
54 *
55 * session_shm(), however, is simple. We just `CREATE_ONLY` it, as its lifetime by definition equals the
56 * session's (`*this`).
57 *
58 * @tparam S_MQ_TYPE_OR_NONE
59 * See shm::classic::Server_session counterpart.
60 * @tparam S_TRANSMIT_NATIVE_HANDLES
61 * See shm::classic::Server_session counterpart.
62 * @tparam Mdt_payload
63 * See shm::classic::Server_session counterpart.
64 */

86 /**
87 * See Session concept counterpart; but read on for notes on per-session SHM-pool cleanup.
88 *
89 * In particular this dtor shall perform (if `*this` is opened OK)
90 * shm::classic::Pool_arena::remove_persistent() on the pool returned by Session_impl::session_shm()
91 * (the per-session pool). Reminder: like all kernel-persistent `remove_persistent()` methods this will not
92 * necessarily free the RAM held on behalf of the pool; any currently opened `Pool_arena`s -- presumably
93 * one still (possibly) held by the counterpart shm::classic::Client_session_impl -- will continue to function
94 * fine.
95 *
96 * ### Rationale for aforementioned `remove_persistent()` op ###
97 * Why do it? Answer: Naturally this should be freed sometime; otherwise the RAM will leak, even though the
98 * session pertaining to the per-session pool is gone.
99 *
100 * Why not do it earlier? Answer: It would be nice to do it super-early... like, say, just after creating the
101 * pool during the async portion of async_accept_log_in(). The only reason *not* to do it is if the client
102 * handle (`Pool_arena`) to the pool has not yet had a chance to get opened by the name deleted via
103 * `remove_persistent()`. The opening occurs in the counterpart shm::classic::Client_session_impl during the
104 * log-in procedure, initiated on this side by async_accept_log_in(). However, as of this writing, the async
105 * portion of async_accept_log_in() (at least when there are no init-channels) completes just after
106 * creating the SHM-pool -- upon sending the log-in response message; it does not wait for any ack from
107 * the client. While in practice the client side will indeed open the pool quite soon after we send
108 * the log-in response, we don't know exactly when that is. Hence we postpone it until this dtor, at which
109 * point the session is definitely done-for. (We could do it earlier, perhaps by adding the ack from client,
110 * but this way is solid if less zealous.)
111 *
112 * Why not do it later? Answer: Session is done-for, if we are being invoked; the counterpart `Client_session`
113 * could not possibly hope to open the pool now, even if it somehow wanted to do so. So why wait?
114 *
115 * ### What if this dtor never runs? ###
116 * In any program that exits without aborting abruptly, it *will* run. Furthermore the proper behavior
117 * (see Session::~Session() doc header) is to either invoke the Session dtor proactively (graceful exit triggered
118 * locally) or in response to an error (triggered by client counterpart ending session and our detecting this).
119 * If the user follows this recommended behavior, then this dtor will run just as soon as it should.
120 * If not, then it will at least run when the program is exiting gracefully.
121 *
122 * It however may not run due to abrupt termination (a crash). In that case the `session_shm()` pool will indeed
123 * leak, in that this dtor won't execute, so the `remove_persistent()` will not occur in that fashion.
124 * The shm::classic::Session_server has a contingency for this, which we don't describe here, wherein it has
125 * *cleanup points* at which it cleans up anything thus leaked during previous invocations of the same
126 * Server_app.
127 *
128 * ### What about the per-app pool? ###
129 * The above talks only of the Session_impl::session_shm() pool; what about
130 * Session_impl::app_shm()? Shouldn't we clean it up here? Answer: No. By definition that pool
131 * is cross-session; more sessions may be opened within for the same Server_app and will expect the pool
132 * to persist. Hence any graceful cleanup is under the purview of Session_server dtor (in loosely
133 * similar fashion to the present dtor).
134 */
135 ~Server_session_impl();

201TEMPLATE_CLSC_SRV_SESSION_IMPL
202template<typename Session_server_impl_t,
203 typename Task_err, typename Cli_app_lookup_func, typename Cli_namespace_func, typename Pre_rsp_setup_func,
204 typename N_init_channels_by_srv_req_func, typename Mdt_load_func>
205void CLASS_CLSC_SRV_SESSION_IMPL::async_accept_log_in
206 (Session_server_impl_t* srv,
207 typename Base::Base::Channels* init_channels_by_srv_req,
208 typename Base::Base::Mdt_reader_ptr* mdt_from_cli_or_null,
209 typename Base::Base::Channels* init_channels_by_cli_req,
210 Cli_app_lookup_func&& cli_app_lookup_func,
211 Cli_namespace_func&& cli_namespace_func, Pre_rsp_setup_func&& pre_rsp_setup_func,
212 N_init_channels_by_srv_req_func&& n_init_channels_by_srv_req_func, Mdt_load_func&& mdt_load_func,
213 Task_err&& on_done_func)
214{
215 using boost::movelib::make_unique;
216
217 // The extra stuff to do on top of the base vanilla Server_session_impl.
218 auto real_pre_rsp_setup_func
219 = [this,
220 // Get the Session_server<> such that its core comprises the arg `srv`.
221 srv = srv->this_session_srv(),
222 pre_rsp_setup_func = std::move(pre_rsp_setup_func)]
223 () -> Error_code
224 {
225 // We are in thread W.
226
227 auto err_code = pre_rsp_setup_func();
228 if (err_code)
229 {
230 // Any Session_server-given setup failed => no point in doing our SHM-classic per-session setup.
231 return err_code;
232 }
233 // else
234
235 /* Do our extra stuff on top of the base vanilla Server_session_impl. Namely: app_shm() and session_shm().
236 *
237 * app_shm first: please see our class doc header; then come back here. To summarize:
238 * - pre_rsp_setup_func() had to have created what we want app_shm() to return.
239 * - If it already existed by then (Client_app seen already), even better.
240 * - If that creation failed, then pre_rsp_setup_func() just failed, so we are not here.
241 * - Parent session server's app_shm(<the Client_app>) = what we want. */
242 auto app_shm = srv->app_shm(*(Base::Base::Base::cli_app_ptr()));
243 assert(app_shm && "How can it be null, if pre_rsp_setup_func() returned success? Contract broken internally?");
244
245 /* Now session_shm(). This is different (see our doc header again): we actually must create it ourselves, as it's
246 * a per-session thing we're making available (both to us and the opposing Client_session once it receives
247 * word that log-in succeeded, from us, right after this).
248 *
249 * The name is similar -- but in the per-session scope, so we supply cli_namespace() instead of the default
250 * (SENTINEL). The same thing regarding 1ST_OR_ONLY applies here too. */
251 const auto& srv_app = Base::Base::Base::m_srv_app_ref;
252 auto shm_pool_name = build_conventional_shared_name(Shared_name::S_RESOURCE_TYPE_ID_SHM,
253 Shared_name::ct(srv_app.m_name),
254 Base::Base::Base::srv_namespace(),
255 Shared_name::ct(Base::Base::Base::cli_app_ptr()->m_name),
256 Base::Base::Base::cli_namespace())
257 / SHM_SUBTYPE_PREFIX
258 / Shared_name::S_1ST_OR_ONLY;
259
260 auto session_shm = make_unique<typename Base::Arena>(get_logger(), shm_pool_name, util::CREATE_ONLY,
261 size_t(1024 * 1024) * srv->pool_size_limit_mi(),
262 util::shared_resource_permissions
263 (srv_app.m_permissions_level_for_client_apps),
264 &err_code);
265 if (err_code)
266 {
267 /* It logged. GTFO.
268 * Note: If app_shm assignment above did in fact create the pool, perhaps we should delete it now
269 * via Pool_arena::remove_persistent(<name>). However:
270 * - How to even find out that we did, versus it having already existed?
271 * - Even if we could, is that even the right thing? The existence of a nearly-empty cross-session
272 * arena is not such a crime; and technically a given session failing to open is *not* considered
273 * the doom of the daddy Session_server or future session-open attempts. So if it did get created,
274 * at least formally speaking it's not necessarily futile and could be used. And if the Session_server
275 * does get totally hosed, we have sophisticated mechanisms for cleaning up old `Session_server`s' resources.
276 *
277 * So yeah, don't touch it. */
278 return err_code;
279 }
280 // else
281
282 Base::init_shm_arenas(std::move(session_shm), app_shm);
283
284 /* Cool! Let's return (and continue the log-in async op!). ...Not so fast. Almost.
285 * Please now see the doc header for our dtor, where we promise -- on dtor invocation -- to remove the
286 * session_shm() pool. It explains why we do that, and what actual effect it has. The below is *how*
287 * we accomplish it: We have just made session_shm() non-null; and therefore (and this is the only way it
288 * would become non-null and need cleaning up) we now ensure that *in* the dtor, the pool removal shall execute.
289 *
290 * It's tempting to Just Do It in the dtor, but there are caveats. Vaguely, all we want to do is something like,
291 * right in the dtor:
292 * if (session_shm()) { Arena::remove_persistent(N); } // Where N is same as computed into shm_pool_name above.
293 * I.e., do nothing if async_accept_log_in() never actually did set up session_shm(); otherwise do it.
294 * This would work 99.999% of the time just fine... except for the annoying possibility that the user
295 * (some Server_session-type-guy really, so internal code) destroys *this while async_accept_log_in() is
296 * outstanding still. (I don't bother to mentally track down what the actual user would have to do to make
297 * it happen... even if it's not actually possible -- at this stage I (ygoldfel) don't even remember. We need
298 * to code defensively: we have an async_() method, and the dtor may be called while it's still outstanding...
299 * period. Let us not write brittle code.)
300 *
301 * In my mind this leaves two possible approaches.
302 *
303 * One is to operate in thread W but from the dtor; we could (in the dtor) post basically the
304 * above code and wait until it executes concurrently; then return (and let the super-class dtor chain run).
305 * However this is entropy-laden and seems to involve a race against the potentially ongoing async_accept_log_in().
306 * Like, suppose we do that -- and session_shm() is not yet set-up -- so we no-op; then while the super-class
307 * dtors are running (including the highly significant Server_session_impl dtor) the log-in happens to advance
308 * a bit and sets up session_shm(), before that dtor has a chance to stop/join thread W. Now we've leaked
309 * the RAM. (Granted, the anti-crash contingency in Session_server should catch it when the next
310 * instance -- if any -- of this Server_app starts up. Relying on that outside a crash scenario is a misuse
311 * of that contingency logic; there is nothing un-graceful about our dtor being invoked before log-in completes.)
312 *
313 * The other approach is more orderly: We know the dtor chain is being invoked, so when ~Server_session_impl()
314 * does execute (very soon), it will indeed stop thread W -- in whatever state it happens to be at the time.
315 * At *that* stage there is no more race involved: there's just what that super-class dtor executes (in caller
316 * thread U) after stopping thread W. Now the state is straightforward: Either thread W did create session_shm()
317 * pool -- then it should be removed -- or it did not (then there is nothing to remove).
318 *
319 * So that is what we do. Server_session_impl supplies a `protected` API for any custom de-init code. We
320 * simply use that; and we only do that right here, the one place we make session_shm() pool actually exist. */
321 Base::Base::sub_class_set_deinit_func([this,
322 /* ^-- Careful: We only use this->get_logger() which is explicitly okay.
323 * *this is destroyed w/r/t Server_session_impl by the
324 * time this lambda is executed; so we do not use that. */
325 shm_pool_name = std::move(shm_pool_name)]()
326 {
327 // We are in thread U (in dtor chain).
328 Error_code sink; // We do not care about any error. This is our best effort; it'll WARN on (unlikely) error.
329 Base::Arena::remove_persistent(this->get_logger(), shm_pool_name, &sink);
330 });
331
332 return err_code; // == Error_code().
333 }; // auto real_pre_rsp_setup_func =
334
335 /* That's all we needed to do: ensure m_session_shm's underlying pool exists; m_session_shm points to it,
336 * so session_shm() will work locally; and Base has recorded app_shm, so app_shm() will work locally.
337 * If that failed, on_done_func(<failure>) will be called by contract -- satisying our contract.
338 * If that succeeded, and so did everything else, we need not do anything else.
339 * ...Damn, almost true. If the above succeeded, but they still failed for some reason (as of this writing
340 * it can only be the m_master_session_channel.send(<log-in response>)), then as promised in the
341 * m_session_shm doc header we should aggressively clean up our SHM-pool handles. So we just add a bit of
342 * post-processing in that case. */
343
344 Base::Base::async_accept_log_in(srv,
345 init_channels_by_srv_req, mdt_from_cli_or_null, init_channels_by_cli_req,
346 std::move(cli_app_lookup_func),
347 std::move(cli_namespace_func),
348 std::move(real_pre_rsp_setup_func),
349 std::move(n_init_channels_by_srv_req_func),
350 std::move(mdt_load_func),
351
352 [this, on_done_func = std::move(on_done_func)]
353 (const Error_code& err_code)
354 {
355 // We are in thread W (from Base).
356
357 // Be Aggressive in cleaning up early. (Just no-op if `*this` is being hosed though.)
358 if (err_code && (err_code != error::Code::S_OBJECT_SHUTDOWN_ABORTED_COMPLETION_HANDLER))
359 {
360 Base::reset_shm_arenas();
361 }
362
363 on_done_func(err_code);
364 }); // Base::Base::async_accept_log_in()
365} // Server_session_impl::async_accept_log_in()
